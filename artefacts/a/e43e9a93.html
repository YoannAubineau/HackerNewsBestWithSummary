Le workflow megamerge dans Jujutsu permet de travailler simultanément sur plusieurs branches

Titre original : Jujutsu megamerges for fun and profit

Publié sur HN le 2026-04-20 21:32 · Entré dans /best le 2026-04-21 17:21 · Résumé généré le 2026-04-21 17:22

Résumé de l'article

Le workflow megamerge est une technique avancée de Jujutsu qui consiste à créer un commit octopus (avec trois parents ou plus) combinant toutes les branches de travail actives, puis de développer au-dessus de ce commit fusionné. Cette approche offre une vue d'ensemble de tous les changements interagissant ensemble sans exposer le megamerge lui-même au serveur distant.

  • Les commits de fusion ne sont que des commits ordinaires avec plusieurs parents ; un commit octopus en combine trois ou plus et constitue la base du workflow megamerge
  • Le megamerge regroupe branches de correction, branches de fonctionnalités, branches en attente de PR et branches de tiers, permettant de vérifier que tout interagit correctement
  • Pour créer un megamerge : jj new x y z puis jj commit --message "megamerge" ; le travail s'effectue ensuite au-dessus sur les commits WIP
  • L'intégration des changements utilise jj absorb (autosquash magique), jj squash --interactive ou jj rebase selon que les modifications doivent rejoindre des commits existants ou en créer de nouveaux
  • Maintenir le megamerge à jour avec la branche trunk utilise l'alias restack : jj rebase --onto trunk() --source "roots(trunk()..) & mutable()" --simplify-parents

Discussion sur Hacker News

Confirmations :

  • Jujutsu offre une amélioration significative pour les workflows complexes impliquant plusieurs branches parallèles et dépendances ; plusieurs utilisateurs confirment l'adoption complète après quelques jours d'utilisation intensive et ne reviendraient pas à Git.
  • La commande absorb fonctionne comme de la "magie" et automatise efficacement la distribution des modifications dans les commits en aval, rendant les megamerges pratiques et fluides sans intervention manuelle constante.
  • L'absence de staging area et le modèle de working-copy-as-commit éliminent les frictions quotidiennes (stash, détachement) et permettent une navigation plus fluide entre les tâches sans perdre le contexte.
  • Jujutsu fonctionne comme backend Git, réduisant les risques d'expérimentation et permettant une transition progressive sans risque de perte de données.
  • Pour les cas d'usage spécifiques (10-30 changements en parallèle, migrations volumineuses, travail upstream fragmenté sur plusieurs années), l'outil offre une ergonomie supérieure à Git.

Réfutations :

  • Les megamerges deviennent inmanageables lors de collaborations en équipe lorsque des collègues n'utilisent pas Jujutsu, car les conflits multi-parents (>2 voies) explosent rapidement et aucun outil ne peut les résoudre automatiquement ; une approche en arbre (merge 2-à-2) est plus pratique.
  • La révolution annoncée masque potentiellement un problème organisationnel : avoir 10-30 changements en vol non coordonnés suggère des lacunes dans la culture d'examen/intégration plutôt qu'une justification technique légitime pour des workflows complexes.
  • Jujutsu reste pré-1.0 avec des changements d'interface cassants réguliers (-d→-o, dépréciations) et une incomplétude notable (gestion des bookmarks, jj git sync futur), rendant l'adoption risquée pour les débutants ou équipes stables.
  • La comparaison avec Git omet que ce dernier suffit amplement pour les workflows simples (feature branches + rebase) qui fonctionnent depuis des décennies ; compliquer les choses avec des megamerges introduit une complexité cognitive sans proportionner les bénéfices réels pour la majorité des cas.
