Extension WebUSB pour Firefox : installation et configuration du stub natif

Titre original : WebUSB Extension for Firefox

Publié sur HN le 2026-04-20 11:51 · Entré dans /best le 2026-04-21 11:50 · Résumé généré le 2026-04-21 11:55

Résumé de l'article

TL;DR : Cette extension ajoute la fonctionnalité WebUSB à Firefox via un programme natif séparé appelé "native stub" que l'utilisateur doit installer en parallèle de l'extension navigateur.

  • L'extension nécessite l'installation d'un composant externe (native stub) sur l'ordinateur en plus du téléchargement dans Firefox, via des binaires préconstruits ou en compilant depuis les sources
  • L'API WebUSB n'est disponible que sur la page principale et non dans les Web Workers, contrairement à Chrome
  • Les binaires préconstruits sont fournis pour macOS (x86_64, ARM64), Linux (x86_64, aarch64) et Windows (AMD64, ARM64) avec des installateurs automatisés
  • La configuration requiert de placer un fichier manifest JSON dans un répertoire spécifique selon le système d'exploitation (ou un registre sur Windows) pour que le navigateur localise le stub natif
  • Android n'est pas supporté en raison de l'absence de capacités de native messaging

Discussion sur Hacker News

Arguments en faveur :

  • WebUSB offre un meilleur bac à sable qu'installer des exécutables non vérifiés : l'accès est restreint à un appareil spécifique après consentement explicite de l'utilisateur, sans permissions système larges
  • Cas d'usage concrets et utiles : flashing GrapheneOS, configuration de claviers mécaniques, applications IoT, minidisques, radiofréquences sans logiciels supplémentaires
  • Alternative plus sécurisée aux anciens systèmes (drivers malveillants, applications Windows slopware) : les utilisateurs font déjà confiance aux sites pour installer des outils, WebUSB contrôle mieux cette confiance
  • Élimine la fragmentation multiplateforme : une application web unique remplace les binaires spécifiques par OS
  • Le modèle de permission est transparent et limité dans le temps comparé aux accès système complets des applications natives

Arguments contre :

  • Exposition USB d'appareils non conçus pour des interactions potentiellement malveillantes : clavier, carte réseau, équipements sensibles risquent des dégâts permanents
  • Fatigue des permissions : 26+ permissions différentes dans les paramètres Chrome créent une habituation du clic-partout, neutralisant le consentement significatif
  • Absence de standard véritable : seul Chromium l'implémente malgré le travail de Mozilla et Apple, c'est une API propriétaire non stabilisée
  • Risques de tracking via identifiants USB et contournement de modèles de sécurité existants (ex : FIDO2 authentificateurs compromis si JS accède directement aux USB)
  • Extension est la bonne approche : réserver cette API aux utilisateurs avertis plutôt que d'élargir la surface d'attaque par défaut pour tous les utilisateurs
