## Résumé de l'article

Cal.diy est une plateforme de planification de calendrier open-source et entièrement gratuite, basée sur Cal.com mais sans les fonctionnalités commerciales ni d'édition entreprise. Destinée à l'auto-hébergement, elle est licensée MIT et ne nécessite aucune clé de licence ni compte Cal.com pour fonctionner.

- Licence MIT 100% open-source sans code propriétaire, contrairement au modèle "Open Core" de Cal.com
- Installation requiert Node.js (≥18), PostgreSQL (≥13), Yarn et des compétences avancées en administration serveur
- Déploiement possible via Docker Compose, Git clone direct, ou plateformes comme Railway, Northflank ou Vercel Pro
- Intégrations disponibles avec Google Calendar, Microsoft Office 365, Zoom, Daily.co, Basecamp, HubSpot et Zoho CRM via configuration d'API
- Contributions bienvenues mais n'affectent que le projet Cal.diy, pas la plateforme Cal.com commerciale

## Discussion sur Hacker News (31 commentaires analysés)

**Confirmations** :
- Cal.com a effectivement fermé le code source en justifiant par la sécurité, ce qui est perçu comme un argument faible : l'IA peut scanner du code fermé tout autant qu'ouvert, et l'open-source bénéficie de l'audit communautaire gratuit, rendant cette justification suspecte.
- C'est clairement un bait-and-switch : il y a un an, Cal.com promouvait le déploiement on-premises open-source comme plus sûr ; aujourd'hui, ils déconseillent cette utilisation, révélant que le changement est motivé par des raisons commerciales et non de sécurité.
- La vraie raison est commerciale : maximiser le contrôle pour monétiser les features enterprise (Teams, Organizations, SSO/SAML, Workflows) et attirer les investisseurs, en utilisant "la sécurité" comme couverture marketing.
- L'écosystème réagit avec des alternatives (calrs, rustical) et une sentiment de trahison parmi les utilisateurs historiques, certains annulant leurs souscriptions par principe.

**Réfutations** :
- La restriction légale est justifiée : une licence OSS disclaime déjà toute responsabilité, mais Cal.com veut signaler que les standards de sécurité ne seront pas prioritaires pour la version libre, ce qui est une position défendable.
- Les abandons de projets open-source sont courants et souvent pires qu'une fermeture contrôlée ; cal.diy offre au moins une alternative maintenue à ceux qui refusent le propriétaire.
- Certains reconnaissent que les décisions commerciales sont légitimes : les projets doivent parfois fermer le code pour survivre et prospérer, même si cela déçoit l'early adopter community.
