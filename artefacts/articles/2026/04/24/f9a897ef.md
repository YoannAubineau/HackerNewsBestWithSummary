## Résumé de l'article

Spinel est un compilateur AOT (Ahead-Of-Time) qui transforme du code Ruby en exécutables natifs autonomes via inférence de types et génération de code C optimisé. Le compilateur est auto-hébergé : son backend écrit en Ruby se compile lui-même en binaire natif, offrant un gain de performance de ~11,6x par rapport à miniruby sur des charges intensives.

- **Pipeline de compilation** : Ruby (parser Prism) → AST sérialisé → inférence de types + génération C (backend self-hosted) → compilation C standard → binaire natif sans dépendances runtime
- **Optimisations clés** : promotion de types de valeur (structures C sur la pile), propagation de constantes, hoisting des boucles, inlining de méthodes courtes, aplatissement des concaténations de chaînes, auto-promotion bigint, et élimination du code mort
- **Fonctionnalités supportées** : classes, héritage, mixins, gestion d'exceptions, pattern matching, blocs/lambdas/Fiber, chaînes mutables, regexp intégrée, variables globales typées et I/O fichiers
- **Mémoire et performance** : garbage collector mark-and-sweep avec petites classes automatiquement stack-allouées, symboles interned à la compilation, réutilisation des buffers de split en boucle, zéro allocation pour certaines opérations sur strings
- **Bootstrap self-hosting** : le compilateur se compile lui-même en trois générations jusqu'à stabilisation, 74 tests passent, build sans dépendances runtime

## Discussion sur Hacker News (26 commentaires analysés)

**Avis positifs** :
- Matz et son expertise des sémantiques Ruby limitent les risques ; le design pragmatique (utilisation de Prism, génération de C) rend le projet réaliste et maîtrisé.
- Le sous-ensemble fonctionnel suffit pour beaucoup de code réel (40+ projets Ruby personnels utiliseraient send() plutôt qu'eval) ; la démonstration live à RubyKaigi 2026 confirme la viabilité.
- L'accélération de développement via Claude (1 mois) et les performances obtenues (86× plus rapides) valident l'utilité des LLMs et rendent les critiques antérieures obsolètes.
- Cette version simplifiée (sans eval, métaprogrammation, threads) correspond à l'évolution désirée par les développeurs Ruby expérimentés : maintainabilité accrue sans sacrifier l'esthétique du langage.

**Avis négatifs** :
- Les limitations majeures (pas d'eval, pas de métaprogrammation dynamique send/method_missing, pas de threads) excluent une large portion du code Ruby réel et de l'écosystème Rails ; ce n'est qu'un sous-ensemble comparable à mruby.
- La qualité du code généré (spinel_codegen.rb : 21k lignes, 15 niveaux d'imbrication) est difficile à maintenir sans aide IA ; le pattern par défaut if-return des LLM crée du code anti-idiomatique et loin d'être optimal.
- La documentation est confuse et l'écosystème Ruby reste fragmenté (MRI, mruby, TruffleRuby, JRuby, Spinel) ; l'expérience utilisateur pâtit du manque d'unification.
- Spinel n'offre aucun avantage décisif par rapport à Crystal (qui a une vraie base d'utilisateurs et est optimisé nativement pour l'AOT) ou à d'autres langages compilés existants ; c'est un outil expérimental mal positionné.
