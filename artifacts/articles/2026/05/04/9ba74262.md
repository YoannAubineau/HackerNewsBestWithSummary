## Résumé de l'article

GitHub a subi une panne affectant plusieurs services critiques, caractérisée par une latence accrue et des délais d'expiration. L'incident a touché Git Operations, Webhooks, Issues, Pull Requests, Actions, Packages, Pages et Codespaces.

- La dégradation des performances a commencé sur Issues et Webhooks, puis s'est étendue à d'autres services avec une latence globale
- GitHub a progressivement atténué les problèmes sur chaque service : Git Operations, Actions, Packages, Pull Requests, Pages et Codespaces ont tous été rétablis
- L'incident a été résolu et tous les services fonctionnent normalement ; GitHub continue de surveiller et publiera une analyse détaillée de la cause racine

## Discussion sur Hacker News (212 commentaires)

**Avis positifs** :
- GitHub connaît une surcharge légitime due à l'explosion du volume de commits (14 milliards prévus en 2025) et des minutes d'Actions (2,1 milliards par semaine), causée par l'adoption massive des agents d'IA et des outils de completion de code.
- L'infrastructure Ruby on Rails et la dépendance croissante à Azure compliquent les efforts de scaling horizontal ; des migrations technologiques vers des langages plus performants (Rust, Go) pourraient aider.
- Les SRE de GitHub font face à des défis d'architecture complexes (monolithe Rails, opérations coûteuses, choix d'infrastructure) et méritent d'être reconnus plutôt que blâmés individuellement.
- Les problèmes de stabilité pourraient être atténués en optimisant les opérations critiques (ex : recherche au lieu d'API directes pour les pull requests) plutôt que de migrer entièrement vers de nouveaux langages.
- Des alternatives comme Forgejo, Codeberg et sourcehut existent et fonctionnent mieux, mais l'effet réseau et l'inertie maintiennent les utilisateurs sur GitHub malgré les pannes répétées.

**Avis négatifs** :
- GitHub souffrait de problèmes de stabilité bien avant l'explosion de l'IA ; les véritables problèmes remontent à l'acquisition par Microsoft et la migration vers Azure, pas seulement aux volumes accrus.
- Une disponibilité de 84,92% en 90 jours (moins de deux 9s) est inacceptable pour une infrastructure critique ; les entreprises avec des SLA commencent à envisager des alternatives sérieusement.
- Microsoft pousse agressivement Copilot et l'IA tout en laissant l'infrastructure s'effondrer sous la charge ; c'est un choix commercial délibéré de prioriser Copilot sur la fiabilité générale.
- La majorité du code généré par l'IA est du « slop » (slop, bruit, code de mauvaise qualité) et spam sans valeur réelle ; GitHub a bon marché de blâmer l'IA plutôt que d'admettre des choix architecturaux et commerciaux désastreux.
- GitHub aurait pu atténuer les problèmes en facturant les utilisateurs libres ou les agents d'IA intensifs, en séparant l'infrastructure gratuite de celle payante, ou en restreignant les abus ; au lieu de cela, Microsoft accepte les pannes comme le coût de la dominance du marché.
