## Résumé de l'article

lib0xc est un ensemble d'APIs inspirées de la bibliothèque standard C conçues pour améliorer la sécurité des programmes C, notamment en prévenant les débordements de buffer et les erreurs de types. Bien que le C ne puisse pas être complètement sécurisé au niveau du langage, cette bibliothèque offre des utilitaires pour activer plus d'avertissements du compilateur et réduire les classes de bogues courantes.

- Repose sur des macros C et le préprocesseur pour exposer une API compatible avec le C11 GNU, sans nécessiter d'allocateur dynamique dans la plupart des cas
- Fournit des APIs de remplacement plus sûres pour des fonctionnalités standard (curseurs pour les buffers, fonctions de conversion de types avec détection d'overflow, gestion d'E/S simplifiée)
- Embraces les extensions clang bounds-safety pour indiquer les limites de mémoire des pointeurs de manière source-compatible
- Conçue pour être portée vers de nouveaux environnements d'exécution en implémentant quelques interfaces clés (gestion de la mémoire, gestion du panic, streams de logs)
- Supportée sur macOS et Linux (arm64, x86_64), avec compilation via GNUMake et suite de tests complète

## Discussion sur Hacker News (93 commentaires)

**Avis positifs** :
- Lib0xc codifie des pratiques de sécurité établies dans l'industrie depuis des décennies, offrant une approche pragmatique et incrémentale pour améliorer la sécurité des codebases C existants sans réécriture complète.
- La bibliothèque fournit un terrain de jeu utile pour les développeurs C qui ne sont pas des experts en compilateurs, permettant d'expérimenter des modèles de programmation plus sûrs et d'identifier ce qui pourrait être standardisé.
- C++ et les langages modernes ne remplacent pas C pour les systèmes embarqués et legacy : lib0xc offre des améliorations incrementales réalistes pour les vastes codebases existantes qui ne seront jamais migrées.
- Les API proposées (cast sûr, gestion des contextes, bounds-checking avec clang) résolvent des problèmes concrets sans réinventer la roue, en capitalisant sur les extensions GNU et C11.
- C reste crucial pour l'embedded et les systèmes critiques où la simplexité, la portabilité et la compilation rapide sont essentielles ; cette approche préserve ces avantages tout en améliorant la sécurité.

**Avis négatifs** :
- Les comités de standardisation (notamment C++) auraient dû intégrer ces API sûres depuis longtemps plutôt que de les laisser émerger en tant que librairies tierces; le refus historique d'admettre les problèmes de sécurité a retardé les progrès.
- Adopter lib0xc impose d'apprendre une nouvelle API et de réécrire du code existant fonctionnel et testé; cette friction limite adoption par rapport à choisir simplement un langage plus sûr dès le départ.
- L'absence de support MSVC et Windows pour un projet Microsoft soulève des questions sur le soutien long terme et la viabilité producteur, particulièrement après le départ du créateur.
- C++ avec std::span, std::string et exceptions offre déjà une meilleure sécurité mémoire; forcer les développeurs à rester en C et bricoler lib0xc est une résignation face à des choix techniques supérieurs.
- La limite fondamentale demeure : lib0xc ne fournit pas la sécurité de type complète, les garanties à la compilation ou le modèle de propriété que Rust/Zig/Odin offrent nativement, rendant ce compromis insuffisant pour les nouveaux projets.
