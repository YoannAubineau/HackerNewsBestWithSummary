## Résumé de l'article

Honker est une extension SQLite qui implémente des primitives de files d'attente, de flux d'événements et de pub/sub (NOTIFY/LISTEN) directement dans une base de données SQLite, sans nécessiter Redis, Celery ou un broker externe. Les opérations conservent les garanties ACID : enqueuer une tâche et modifier une table métier dans la même transaction signifie que l'annulation (rollback) supprime les deux atomiquement.

- L'extension fonctionne comme une simple extension SQLite loadable, offrant un format sur disque partagé pour Python, Node, Rust, Go, Ruby, Bun et Elixir
- La détection de changements utilise PRAGMA data_version (un compteur monotone incrémenté par SQLite à chaque commit) sondé toutes les millisecondes, avec latence cross-process autour de 0,7 ms en p50
- Les files, streams et pub/sub sont implémentés comme des INSERTs dans des tables gérées par l'extension, éliminant les problèmes de dual-write et de sauvegarde entre datastore primaire et queue
- Un seul thread de polling par base de données, indépendant du nombre d'abonnés, réduit les coûts d'inactivité à une requête SELECT légère par milliseconde sans contention de verrous
- Honker s'adresse aux projets SQLite existants (Bluesky PDS, Fly LiteFS, Turso) qui ont besoin d'une queue sans infrastructure supplémentaire, tout en reconnaissant que Postgres avec pg-boss ou Oban reste préférable pour les utilisateurs de Postgres

## Discussion sur Hacker News (55 commentaires)

**Avis positifs** :
- Approche intéressante pour les projets nouveaux combinant état métier, files d'attente, et offsets de flux dans une seule base transactionnelle
- SQLite peut satisfaire les requêtes PRAGMA data_version via la mémoire partagée en WAL mode sans pression significative sur le cache
- Cas d'usage légitime dans la zone intermédiaire : trop important pour les files en mémoire, trop petit pour justifier une infrastructure Kafka
- Kine (k3s) démontre que ce type de polling sur SQLite fonctionne en production sans CPU excessif malgré plusieurs watches actives
- Avantage vs Redis : évite dépendances supplémentaires (npm, Docker, Java) pour les petits projets

**Avis négatifs** :
- Le polling 1000 fois par seconde gaspille ~0.3% CPU par thread/processus même inactif, incompatible avec les appareils mobiles et états CPU bas
- SQLite ne supporte pas nativement NOTIFY/LISTEN (Postgres/MySQL) ; les alternatives comme inotify sur les fichiers -wal ou les file watchers seraient plus efficaces
- Contrainte du single-writer dans SQLite crée des problèmes concurrents ; les solutions in-process (anneau circulaire + futex) seraient plus appropriées
- Aucun benchmark publié, pas de métriques messages/seconde ; revendications de performance non substantiées
- Le polling au milliseconde n'est pas vraiment mieux qu'un file watcher kernel et la justification marketing semble exagérée (selon les auteurs eux-mêmes)
