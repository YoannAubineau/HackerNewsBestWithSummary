## Résumé de l'article

L'auteur a développé un système qui capture automatiquement les screenshots de la documentation d'une application web et les met à jour lors de chaque reconstruction. Cela élimine le problème courant des captures obsolètes après des modifications de l'interface.

- Les articles d'aide en Markdown contiennent des commentaires HTML spéciaux qui instruisent un système automatisé pour capturer des screenshots spécifiques via Capybara et Chrome sans interface graphique
- Le système offre plusieurs modes de capture : élément DOM par sélecteur CSS, page complète, ou viewport visible, avec options pour cliquer sur des éléments, attendre des animations, recadrer des zones ou masquer du contenu
- Une simple commande Rake (`rails manual:build`) exécute le pipeline complet : capture tous les screenshots et construit les pages d'aide à partir des fichiers Markdown
- L'approche réduit drastiquement la friction pour maintenir la documentation à jour, car les changements d'interface se reflètent automatiquement sans travail manuel de cropping ou de gestion

## Discussion sur Hacker News (31 commentaires)

**Avis positifs** :
- Intégrer les instructions de capture directement dans le markdown est une solution simple et durable, bien meilleure que les outils externes complexes
- Cette approche s'avère particulièrement utile pour les projets mobiles où générer des captures pour plusieurs tailles d'écran et localisations était auparavant fastidieux
- Permet d'automatiser les captures de tests e2e et de les versionnner avec le code, assurant que la documentation reste toujours à jour
- Offre des bénéfices supplémentaires comme générer automatiquement les captures en thèmes clair/sombre ou ajouter des annotations via des outils comme Satori
- C'est une pratique qui devrait devenir standard ; elle améliore significativement la qualité de la documentation, particulièrement pour les applications en évolution continue

**Avis négatifs** :
- Pour certains cas (selon le moteur ou l'architecture), configurer un chemin de capture headless et les outils CLI demande un effort initial significatif
- Une approche de rendu live direct dans le navigateur pourrait être plus optimal visuellement en respectant les paramètres d'accessibilité et les préférences utilisateur
- La solution décrite génère des rendus statiques alors qu'une prévisualisation en direct du produit dans un rectangle interactif offrirait une meilleure expérience
