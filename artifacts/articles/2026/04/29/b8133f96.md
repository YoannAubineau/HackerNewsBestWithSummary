## Résumé de l'article

Copy Fail (CVE-2026-31431) est une vulnérabilité du kernel Linux affectant les distributions construites entre 2017 et le correctif, permettant à un utilisateur non privilégié d'obtenir l'accès root via l'API cryptographique du kernel (AF_ALG). Le même exploit fonctionne sans modification sur toutes les distributions Linux affectées.

- La vulnérabilité ne nécessite qu'un compte utilisateur local sans accès réseau, AF_ALG étant activé par défaut dans les configurations standard des distributions mainstream
- Les systèmes particulièrement exposés incluent les machines partagées, les CI/CD (GitHub Actions runners, GitLab runners), les conteneurs multi-locataires et les serveurs de compilation
- Un correctif est disponible via le commit mainline a664bf3d603d qui désactive l'optimisation in-place algif_aead de 2017
- En attendant le correctif, désactiver le module algif_aead limite l'exposition; bloquer la création de sockets AF_ALG via seccomp est recommandé pour les charges de travail non fiables
- Le PoC est publié et ne nécessite que Python 3.10+ avec les modules stdlib standards

## Discussion sur Hacker News (171 commentaires)

**Avis positifs** :
- La vulnérabilité est extrêmement grave : escalade de privilèges locale fiable sur des noyaux Linux affectés depuis 2017, exploitable en moins de 732 octets de Python sans dépendances externes
- L'exploit fonctionne largement : confirmé sur Ubuntu 24.04, Debian, et de nombreuses distributions modernes ; affecte potentiellement des millions de conteneurs, serveurs partagés et environnements cloud
- Nommer les CVE (ex: 'copy.fail') facilite la mémorisation et la communication par rapport aux numéros génériques, comme pour Heartbleed ou Log4Shell
- Les chercheurs ont respecté un processus de divulgation responsable : 1 mois accordé aux distributions pour patcher avant la publication publique
- La vulnérabilité révèle l'importance de l'audit de sécurité par IA, trouvée rapidement après 9 ans d'existence

**Avis négatifs** :
- La page officielle contient des erreurs AI flagrantes (RHEL 14.3 inexistant, prétention d'avoir testé sur '4 distributions' en affirmant qu'il marche 'depuis 2017'), compromettant la crédibilité malgré le sérieux des chercheurs
- Le code est volontairement obfusqué avec 'os as g' et zlib compressé pour réduire les octets : rend l'exploit moins lisible et plus difficile à vérifier pour les utilisateurs non-techniques
- Les distributions majeures tardent à classifier comme 'haute priorité' et à patcher : Ubuntu 24.04 reste vulnérable même après mise à jour, Debian Stable et RHEL non patché des mois après la divulgation
- Les réclamations marketing sur l'échappement de conteneurs Kubernetes manquent de preuves techniques publiées ; l'exploit ne fonctionne pas en conteneur rootless sans modifications supplémentaires
- La solution requiert soit une mise à jour complète du noyau (6.18.22+, 6.19.12+, 7.0+), soit une mitigation manuelle (désactiver algif_aead) et ne fonctionne pas sur certaines plateformes comme Android ou WSL2
