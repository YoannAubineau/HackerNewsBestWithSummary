## Résumé de l'article

Le projet Zig maintient une politique stricte interdisant les contributions assistées par LLM (modèles de langage), contrairement à de nombreux projets open source. La Fondation Zig justifie cette position par une philosophie misant sur le développement humain des contributeurs plutôt que sur l'accumulation de code.

- Zig interdit les LLM pour les rapports de bugs, les demandes de fusion (pull requests) et les commentaires, encourageant l'anglais mais acceptant toute langue pour favoriser les traductions humaines
- Bun, le runtime JavaScript le plus connu écrit en Zig, a été acquis par Anthropic et utilise massivement l'IA, mais maintient sa propre branche du code sans contribuer les améliorations en amont en raison de cette politique
- Le projet considère que les contributeurs sont plus importants que les contributions individuelles : l'examen des PR vise à former de nouveaux contributeurs de confiance, pas simplement à ajouter du code
- L'assistance LLM casse cette dynamique car elle ne renforce pas la capacité du contributeur ; un mainteneur aurait logiquement intérêt à résoudre directement le problème avec une LLM plutôt que de revoir une PR générée par IA
- Cette approche, appelée « contributor poker », mise sur l'humain derrière la contribution plutôt que sur la qualité immédiate du code soumis

## Discussion sur Hacker News (157 commentaires)

**Avis positifs** :
- La politique favorise les contributeurs humains de confiance sur le long terme, car l'examen des PR LLM n'aide pas à construire une communauté engagée et à identifier les futurs mainteneurs fiables.
- Les contributions LLM créent du bruit avec des hallucinations, des erreurs de compilation et des PRs géantes sans valeur, qui consomment gratuitement le temps de révision des mainteneurs.
- Les mainteneurs ont déjà des alternatives : utiliser eux-mêmes les LLM pour leurs propres besoins au lieu de servir de point intermédiaire avec des contributeurs externes.
- La politique protège l'intégrité architecturale du projet en évitant des refactorisations mal conçues ; Zig dispose déjà de meilleures solutions en développement pour les performances.
- Les contributions manuelles stimulent la réflexion, l'itération et l'apprentissage, créant du code de meilleure qualité et une compréhension plus profonde du projet.

**Avis négatifs** :
- La politique est trop rigide et empêche les contributeurs responsables qui utilisent les LLM de façon éthique et réfléchie de participer, créant une barrière artificielle.
- La distinction entre code LLM et code humain est impossible à vérifier fiablement, rendant la politique difficilement applicable de manière équitable et transparente.
- Les projets sans LLM risquent de perdre des contributeurs à long terme si d'autres projets accueillent les contributions assistées ; cette inflexibilité pourrait marginaliser Zig.
- La vraie valeur ne vient pas de qui a écrit le code mais de sa qualité vérifiée ; rejeter catégoriquement tous les PRs LLM sacrifie des contributions valides au nom du principe.
- Les LLMs améliorent objectivement la productivité des bons programmeurs ; refuser ces outils c'est se handicaper volontairement et laisser d'autres projets avancer plus vite.
