## Résumé de l'article

GitHub a connu deux incidents majeurs en avril 2026 et restructure son infrastructure pour supporter une croissance exponentielle due aux workflows de développement assisté par IA. L'entreprise accélère son passage à l'architecture cloud et renforce l'isolation des services critiques pour améliorer la disponibilité et réduire les points de défaillance uniques.

- Le 23 avril, une régression dans les opérations de merge queue a affecté 658 dépôts et 2 092 pull requests en utilisant la méthode de fusion par squash, causant des inversions de commits sans perte de données Git
- Le 27 avril, une surcharge du cluster Elasticsearch (probablement due à une attaque botnet) a affecté les fonctionnalités de recherche à travers l'interface sans impacter les opérations Git
- GitHub passe d'un plan 10X à 30X de capacité en réponse à la croissance accélérée des workflows de développement depuis décembre 2025, notamment pour les grands monorepos
- Les priorités sont disponibilité d'abord, puis capacité, puis nouvelles fonctionnalités ; GitHub migre du code sensible en performance de Ruby vers Go et isole les services critiques
- L'entreprise améliore la transparence en publiant les numéros de disponibilité et en statusant les incidents de tous les niveaux

## Discussion sur Hacker News (193 commentaires)

**Avis positifs** :
- La croissance exponentielle due aux agents IA (10-30x) est un défi réel et sans précédent que peu de systèmes ont dû affronter
- GitHub a historiquement fait des travaux d'optimisation Git bénéfiques à tous, et les défis actuels reflètent une complexité d'infrastructure légitime à cette échelle
- La stratégie multi-cloud constitue une réaction sensée pour éviter la dépendance à un seul fournisseur, même si elle intervient tardivement
- Les initiatives techniques mentionnées (migration webhooks de MySQL, optimisation Ruby vers Go) adressent des goulots d'étranglement réels et justifiés

**Avis négatifs** :
- Le blog post manque de transparence : graphiques sans échelle sur l'axe Y, deux incidents seulement listés alors que les problèmes durent depuis un an, et priorités annoncées identiques à celles d'il y a 6 mois
- La migration forcée vers Azure a empiré la situation au lieu de l'améliorer, suggérant un échec majeur de planification chez Microsoft
- GitHub continue à ajouter des features chaque semaine malgré la déclaration que la disponibilité est la priorité, contredisant les engagements affichés
- Les données manquantes sur l'impact réel (dont chercher par texte est cassé depuis 6 jours, pull requests incomplètes affichées) montrent que les métriques publiées ne reflètent pas l'expérience utilisateur
- Le service critique que constitue GitHub pour les développeurs souffre d'une fiabilité inacceptable (97% uptime = ~40 min de downtime/jour), poussant les utilisateurs vers l'auto-hébergement
