## Résumé de l'article

L'article examine le problème du "sur-édition" où les modèles d'IA modifient le code bien au-delà de ce qui est nécessaire pour corriger un bogue, rendant la révision de code plus difficile. Les auteurs mesurent ce phénomène à travers des métriques comme la distance de Levenshtein au niveau des tokens et la complexité cognitive ajoutée, puis explorent les solutions.

- Les modèles frontières, notamment GPT-5.4, sur-éditent régulièrement : ils réécrivent des fonctions entières pour corriger un simple bogue, alors que Claude Opus 4.6 produit des modifications bien plus minimalistes
- Le sur-édition n'est pas une limitation fondamentale mais un comportement par défaut : l'ajout d'une instruction explicite pour préserver le code original réduit significativement les modifications dans tous les modèles, particulièrement les modèles de raisonnement
- L'entraînement par renforcement (RL) génère des éditeurs plus fidèles sans dégrader les capacités de codage générales, contrairement au fine-tuning supervisé qui provoque une amnésie catastrophique
- LoRA à faible rang suffit pour améliorer le comportement de sur-édition, suggérant que les changements de style de code ne nécessitent pas le fine-tuning complet
- Les gains de l'RL se généralisent à des modèles de tailles différentes (4B et 14B) et améliorent tous les critères : exactitude, minimalisme des modifications et complexité du code

## Discussion sur Hacker News (46 commentaires analysés)

**Confirmations** :
- Le sur-éditing des agents est un problème réel observé en pratique : les modèles modifient du code au-delà du nécessaire, ajoutent des commentaires superflus, et réécrivent des sections sans raison valide, brûlant inutilement des tokens.
- La tension existe entre minimaliser les changements (pour réduire la surface de test) et améliorer la codebase : les agents peinent à calibrer cette balance et biaisent généralement vers des diffs minimaux même quand cela crée de la duplication.
- L'autonomie complète des agents pose des risques sérieux : manque de transparence sur les actions effectuées, erreurs architecturales graves difficiles à corriger, et exposition à des comportements dangereux (manipulation de credentials, commandes destructrices).
- Le contrôle humain granulaire sur les agents (steerable semi-autonomous agents) est plus viable que l'automatisation totale : cela préserve la compréhension du code et permet de corriger les mauvaises hypothèses avant qu'elles ne se cristallisent.

**Réfutations** :
- L'over-editing n'est pas universel : plusieurs utilisateurs rapportent que Claude Code et d'autres modèles récents ne manifestent plus ce problème significativement, suggérant une amélioration rapide des capacités.
- L'argument du "non-déterminisme" des LLMs est exagéré : en réduisant la portée et en utilisant des prompts stables, on obtient des résultats quasi-reproductibles, comparables à l'efficacité des compilateurs que l'industrie a acceptés sans réserve.
- Critiquer l'over-editing ignore les gains réels de productivité : avec un bon pilotage (project skills, feedback loops), les agents génèrent 5-6x plus de commits/semaine tout en maintenant la qualité, remettant en question le besoin de révision manuelle exhaustive.
