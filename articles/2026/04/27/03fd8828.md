## Résumé de l'article

GitHub a connu une panne affectant plusieurs services le 27 janvier 2026, causée par une surcharge des clusters ElasticSearch. Les utilisateurs ont rencontré des défaillances lors de l'accès aux issues, pull requests, projects et Actions, ainsi que des problèmes de recherche.

- Les premiers rapports de dégradation de performance ont émergé sur GitHub Actions vers 16h31 UTC, suivi de problèmes en cascade sur Issues, Packages et Pull Requests
- Une surcharge non identifiée sur les clusters ElasticSearch a causé des problèmes de connectivité affectant les services en aval
- GitHub a identifié et désactivé la source de la charge supplémentaire, observant des signes de récupération du système
- Les défaillances étaient intermittentes, incluant l'impossibilité de charger les projets, les recherches expirées et les défaillances des workflows Actions

## Discussion sur Hacker News (92 commentaires)

**Avis positifs** :
- Les pannes de GitHub sont devenues fréquentes et récurrentes (plusieurs fois par semaine), affectant gravement les workflows de développement avec des problèmes d'affichage et d'incohérence des données.
- La migration vers Azure semble être la cause principale des problèmes de stabilité actuels, particulièrement depuis l'acquisition par Microsoft il y a plusieurs années.
- Les alternatives auto-hébergées (Gitea, Forgejo, GitLab CE) offrent une bien meilleure fiabilité, performance et coûts réduits, validant la stratégie de décentralisation.
- GitHub a concentré les efforts sur les features IA (Copilot) au détriment de la fiabilité et de la scalabilité, ce qui aggrave les problèmes d'infrastructure.
- Le problème révèle une dépendance dangereuse : les outils critiques pour le développement logiciel (git, email, internet) se sont centralisés auprès de Microsoft, créant un point de défaillance unique.

**Avis négatifs** :
- Le graphique d'uptime post-acquisition pourrait simplement refléter une meilleure documentation/suivi des pannes plutôt qu'une dégradation réelle; les historiques pré-2018 ne sont pas complètement documentés.
- La charge accrue provient en partie de l'augmentation massive du trafic IA général, pas uniquement des décisions de Microsoft; le problème de scalabilité est universel.
- Les alternatives auto-hébergées nécessitent du travail de maintenance continu et n'offrent pas les mêmes fonctionnalités sociales, d'intégration et de découverte que GitHub.
- Migrer du code n'est que l'aspect trivial; tous les tickets fermés, PR, permissions, règles de branche et références externes rendent une migration complète extrêmement complexe et risquée.
- Même lorsque GitHub fonctionne, les alternatives comme GitLab CE souffrent de problèmes similaires (lenteur des diffs, performance des SPA JavaScript).
