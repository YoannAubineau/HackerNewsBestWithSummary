## Résumé de l'article

Mitchell Hashimoto, fondateur du projet Ghostty, annonce que le projet migrera hors de GitHub en raison de pannes quotidiennes qui entravent la productivité et la livraison de logiciels. Malgré son attachement émotionnel profond à la plateforme, l'auteur a documenter des interruptions de service presque quotidiennes au cours du dernier mois.

- L'auteur, utilisateur GitHub depuis 2008, a passé plus de dix-huit ans à travailler quotidiennement sur la plateforme et considérait GitHub comme son rêve professionnel
- Les pannes de GitHub Actions et autres infrastructures (problèmes, pull requests) bloquent le travail plusieurs heures par jour, rendant la plateforme inadéquate pour des projets sérieux
- Ghostty sera migré vers une alternative (commercial ou open source) avec un plan d'implémentation progressif; un miroir en lecture seule restera disponible à l'URL actuelle
- Les projets personnels de l'auteur resteront sur GitHub pour l'instant; seul Ghostty est prioritaire pour cette migration
- Le départ est motivé par des considérations de fiabilité opérationnelle plutôt que par l'idéologie, l'auteur restant ouvert à un retour futur si GitHub s'améliore

## Discussion sur Hacker News (94 commentaires)

**Avis positifs** :
- GitHub a connu un déclin réel et documenté de fiabilité depuis son acquisition par Microsoft, avec des centaines d'incidents en 90 jours et une disponibilité d'environ 87% (3h de pannes par jour)
- La migration vers Azure, l'essor du code généré par IA, et l'adoption de pratiques « vibe coding » ont collectivement dégradé la qualité du service et la confiance envers GitHub
- La période avant l'acquisition (2014-2017) était plus stable ; les problèmes se sont aggravés après le rachat Microsoft et l'intégration de Copilot, indiquant un lien causal clair
- GitHub occupant une position critique dans l'infrastructure mondiale du développement logiciel, les standards de fiabilité requis ont augmenté alors que la qualité a diminué
- Des alternatives comme Codeberg, Forgejo, Sourcehut et des approches décentralisées émergent comme réponses légitimes au déclin de GitHub

**Avis négatifs** :
- Les graphiques de croissance présentés par GitHub manquent d'axes Y étiquetés, rendant impossible de vérifier si l'augmentation d'utilisation justifie les problèmes rapportés
- GitHub a toujours eu des problèmes de fiabilité historiques (notamment en 2008-2009 à l'époque de Rails) ; le contexte actuel n'est pas nécessairement pire, juste plus visible
- GitLab et d'autres alternatives présentent leurs propres défauts graves (bugs UI, manque de clarté, manque de fonctionnalités équivalentes), suggérant que le problème n'est pas unique à GitHub/Microsoft
- La croissance exponentielle due aux agents autonomes et au code généré est un défi d'infrastructure légitime, pas seulement une mauvaise gestion ou une « culture Microsoft »
- L'absence de vraie concurrence viable (peu d'alternatives réelles au-delà de Codeberg) signifie que même reconnaître les problèmes n'offre pas de solutions pratiques pour la plupart des projets
