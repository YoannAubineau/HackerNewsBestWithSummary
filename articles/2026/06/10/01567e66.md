## Résumé de l'article

npm v12, prévu pour juillet 2026, apporte des changements de sécurité par défaut au comportement d'installation. Ces changements transforment des comportements actuellement automatiques en actions explicites à autoriser, avec des avertissements disponibles dès npm 11.16.0.

- **allowScripts désactivé par défaut** : npm n'exécutera plus les scripts preinstall, install et postinstall des dépendances à moins qu'ils ne soient explicitement autorisés via `npm approve-scripts`
- **--allow-git défaut à none** : les dépendances Git (directes ou transitives) ne seront résolues que si explicitement autorisées, fermant un vecteur d'exécution de code via `.npmrc`
- **--allow-remote défaut à none** : les dépendances depuis des URL distantes (tarballs HTTPS) nécessiteront une autorisation explicite
- **Préparation dès maintenant** : mettre à jour npm 11.16.0+, exécuter l'installation habituelle, utiliser `npm approve-scripts --allow-scripts-pending` pour identifier et approuver les packages de confiance
- **Sauvegarde dans package.json** : la liste d'autorisation est enregistrée dans package.json et doit être versionnée

## Discussion sur Hacker News (134 commentaires)

**Avis positifs** :
- Cette modification était nécessaire : npm a attendu 10 ans avant de corriger une vulnérabilité critique de 2014, et seule la pression d'attaques activement exploitées (notamment contre Microsoft) a forcé le changement.
- La solution améliore la sécurité pour une majorité d'utilisateurs qui n'ont pas besoin de scripts d'installation, tout en gardant la possibilité de les activer explicitement pour ceux qui en ont besoin.
- npm suit enfin l'exemple de pnpm qui avait déjà implémenté des protections similaires il y a 18 mois, avec un système d'allowlist plus transparent et auditable.
- Cette approche rend les décisions de sécurité explicites et traçables : l'utilisateur doit activement choisir d'exécuter des scripts potentiellement dangereux plutôt que de les accepter par défaut sans le savoir.
- Le changement devrait forcer les développeurs à abandonner les dépendances mal conçues qui s'appuient sur des scripts d'installation, encourageant une meilleure ingénierie des packages.

**Avis négatifs** :
- Cette approche ne règle que partiellement le problème : le code malveillant peut toujours s'exécuter lors de l'import des modules ou pendant les tests, pas seulement lors de l'installation.
- Le système crée l'illusion d'une sécurité tout en décalant le problème : les utilisateurs vont probablement ajouter automatiquement tout à l'allowlist plutôt que d'auditer chaque paquet, reproduisant l'insécurité d'avant.
- Construire sur des sandboxes aurait été plus efficace qu'un simple changement de défaut, mais npm refuse d'implémenter une vraie isolation malgré les capacités existantes de Node.js et les leçons de Deno.
- Le véritable enjeu est la conception cassée de l'écosystème npm lui-même : des milliers de dépendances implicites sans audit, un modèle commercial insoutenable, et l'absence de mécanisme de validation avant publication (contrairement aux distributions Linux).
- Microsoft a laissé l'équipe npm se désagréger (licenciements majeurs pour se concentrer sur l'IA) et n'a agi que sous pression d'incidents publics, suggérant que cette correction relève du performatif plutôt que d'une vision de sécurité cohérente.
