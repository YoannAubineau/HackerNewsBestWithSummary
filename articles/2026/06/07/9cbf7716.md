## Résumé de l'article

Mark Dominus expose comment il a changé sa pratique en conservant et en versionnant les documents de synthèse générés par Claude (un assistant IA) à la fin de chaque projet, plutôt que de les supprimer. Cette approche répond à une frustration courante : les programmeurs acceptent de documenter pour une IA mais pas pour leurs collègues.

- Claude génère des documents de continuité utiles entre les sessions pour résumer les objectifs, le travail accompli et les étapes suivantes, que l'auteur peut conserver dans le dépôt
- À la fin de chaque projet, l'auteur demande à Claude de rédiger une synthèse structurée et détaillée du problème résolu et des modifications apportées, indépendante des notes de travail
- L'auteur relit systématiquement ces synthèses avant leur versionnement, assumant la responsabilité du contenu comme s'il s'agissait d'un contributeur humain sous sa supervision
- Les synthèses de Claude nécessitent peu de corrections et gagnent un temps considérable par rapport à une documentation manuelle (dix secondes d'écriture contre une heure)
- Cette documentation devient accessible aux développeurs futurs via git grep, même des années après, pour comprendre les décisions et les changements passés

## Discussion sur Hacker News (156 commentaires)

**Avis positifs** :
- Claude lit réellement la documentation contrairement aux humains qui l'ignorent systématiquement, ce qui justifie l'effort d'écriture
- Le ROI de la documentation augmente considérablement : Claude offre une garantie de lecture et d'application, contrairement aux collègues qui posent des questions sans lire
- La documentation pour Claude libère du temps : moins de préoccupations formatage/style/perfection, simplement transmettre l'intent, et Claude génère rapidement des docs utiles
- Cela crée enfin une incitation économique positive à documenter : l'effort individuel (meilleure IA) produit un bénéfice personnel immédiat, contrairement aux docs collectives ignorées
- La documentation structurée améliore réellement les résultats des agents ; cela valide enfin l'importance d'une bonne documentation qu'on prédisait depuis des années

**Avis négatifs** :
- Les docs générées par IA sont souvent incomplètes (70%), indirectes (10%) ou fausses (20%), créant un faux sentiment de couverture documentaire qui se désynchronise rapidement
- Le problème n'est pas la documentation elle-même mais la culture d'entreprise : les gens ne lisent rien par manque d'incitation à collaborer, pas par manque de docs
- Favoriser les docs pour Claude au détriment des docs humaines aggrave l'isolement social et la mésestime entre collègues au lieu de résoudre les problèmes culturels
- La prolifération de documentation générée par IA sans révision crée du bruit; les vrais problèmes (documentation en désynchronisation, hallucinations) restent non résolus
- Documenter pour une machine plutôt que d'investir dans une culture de collaboration efficace érode le tissu social des équipes techniques
