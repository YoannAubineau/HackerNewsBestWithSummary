## Résumé de l'article

Linear est un outil de gestion de projet connu pour sa réactivité exceptionnelle : mettre à jour une issue prend quelques millisecondes au lieu des 300 ms typiques des applications web. Cette performance ne provient pas d'une technique isolée, mais d'une architecture globale pensée dès le départ autour de la vitesse et de l'optimisation de chaque détail.

- **Base de données locale dans le navigateur** : Linear stocke les données dans IndexedDB au lieu de dépendre exclusivement du serveur ; les mutations s'appliquent localement d'abord, puis se synchronisent en arrière-plan via WebSocket, éliminant le besoin d'attendre les réponses réseau pour afficher les changements.

- **Chargement initial ultra-optimisé** : le code JavaScript est réduit de moitié (de 21 MB minifiés seulement en envoyé initialement), divisé en centaines de chunks chargés à la demande, et préchargés intelligemment en parallèle lors de la connexion ; une service worker met en cache l'application complète en arrière-plan pour des navigations ultérieures instantanées.

- **Synchronisation granulaire avec MobX** : chaque propriété d'objet est un observable MobX indépendant ; une mise à jour d'un champ d'une issue ne re-rend que les composants qui le lisent, pas toute la liste, permettant à l'app de rester fluide même avec de nombreux utilisateurs simultanés.

- **Conception centrée sur le clavier** : chaque action courante a un raccourci, et la palette de commandes (⌘K) permet d'accéder à presque toute fonction en recherchant dans les données déjà en mémoire, réduisant le chemin d'interaction nécessaire.

- **Animations performantes et épurées** : Linear n'anime que des propriétés GPU (transform, opacity) et certaines propriétés de peinture (background-color), évite complètement les animations de layout (width, height, margin), et maintient les durées courtes (< 250 ms) pour préserver la sensation de rapidité.

## Discussion sur Hacker News (105 commentaires)

**Avis positifs** :
- Le pattern d'optimistic UI avec synchronisation en arrière-plan est efficace et bien exécuté, apportant une expérience utilisateur considérablement meilleure que les applications CRUD traditionnelles lentes
- Réactiver des concepts éprouvés (Meteor, minimongo, client-side prediction) pour le web moderne est pertinent ; ces techniques ont montré leur valeur dans les jeux multijoueurs depuis longtemps
- Linear démontre une excellente qualité d'ingénierie et de design global en tant que produit, particulièrement en comparaison avec JIRA qui est devenu laborieux
- L'approche local-first avec gestion des synchronisations peut être fiable si implémentée correctement, comme le montre l'absence de pertes de données massives rapportées par la majorité des utilisateurs satisfaits

**Avis négatifs** :
- Le problème fondamental des 300ms n'est pas intrinsèque au paradigme serveur ; avec une bonne architecture backend (edge computing, latence RTT faible), une application CRUD classique peut atteindre 30ms sans les complications d'une couche client-side rendering
- La gestion des cas d'erreur (conflits, échecs de synchronisation, pertes de données lors de fermeture d'onglet, rollbacks silencieux) crée une complexité immense souvent minimisée ; zéro de Rocicorp a d'ailleurs renoncé aux écritures offline pour ces raisons
- Plusieurs utilisateurs rapportent une réversion involontaire de données lors de modifications rapides, des ongles figés en synchronisation bloquant l'application, et une perte mystérieuse de contenu, suggérant que la fiabilité n'est pas garantie malgré les affirmations
- Le modèle risque une adoption irréfléchie par d'autres produits sans plan solide pour la résolution d'erreurs ; difficulté d'identifier qu'une opération a échoué après navigation vers d'autres écrans
- Pour les cas d'usage critiques (transactions commerciales, contraintes d'intégrité référentielle complexe, changements de schéma), cette approche crée des risques inacceptables de perte ou d'incohérence de données
