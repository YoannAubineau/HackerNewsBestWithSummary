## Résumé de l'article

Le Steering Council de Python a formellement demandé l'arrêt du développement du compilateur just-in-time (JIT) expérimental intégré dans CPython jusqu'à l'acceptation d'une PEP (Python Enhancement Proposal) formelle. Le JIT est un composant logiciel conçu pour compiler le code Python à l'exécution afin d'améliorer les performances.

- Le JIT a été fusionné dans la branche principale de CPython en tant que projet expérimental, mais n'a jamais reçu de validation formelle via le processus PEP Standards Track, bien que plusieurs points critiques restent non résolus.
- Une nouvelle PEP Standards Track doit être rédigée et acceptée dans un délai de six mois, adressant la maintenance à long terme, la compatibilité avec les outils existants (profilers, debuggers, free-threading), les métriques de succès mesurables et les relations avec d'autres implémentations JIT.
- Jusqu'à l'acceptation de cette PEP, seules les corrections de bugs et de sécurité sont autorisées; toute nouvelle fonctionnalité, optimisation ou travail de performance du JIT est gelée.
- Si aucune PEP n'est acceptée après six mois, le code JIT sera retiré de la branche principale et le développement devra continuer en dehors du dépôt Python officiel.
- Le Steering Council invite également à discuter de propositions alternatives et d'une infrastructure JIT générique capable de supporter plusieurs stratégies d'implémentation plutôt qu'une seule approche.

## Discussion sur Hacker News (94 commentaires)

**Avis positifs** :
- La pause est justifiée par les précédents problèmes (ex. retrait d'urgence du GC incrémental), montrant qu'un processus formel via PEP est nécessaire pour des changements majeurs dans CPython.
- Le JIT a enfin atteint une performance comparable à l'interpréteur de base après avoir amélioré ce dernier de 40-50%, validant que le projet mérite formalisation.
- Exiger une infrastructure JIT pluggable et flexible plutôt qu'un seul implémentation permet d'expérimenter plusieurs approches et facilite la maintenabilité à long terme.
- Une pause sur la branche principale n'arrête pas le développement, elle le réorganise en demandant clarification du design, des objectifs et de la maintenance avant intégration officielle.
- Python doit rester simple et maintenable : charger la base de code d'une complexité JIT sans bénéfices clairs pour la majorité des utilisateurs pose un risque de régression.

**Avis négatifs** :
- Arrêter le développement tue les projets : une fois le momentum perdu et l'équipe dispersée (notamment après les licenciements chez Microsoft), redémarrer devient quasi-impossible.
- Le langage de l'annonce officielle est manipulateur ('pas de konkurence' suivi immédiatement d'un appel à propositions rivales) et suggère que les porteurs du projet sont méprisés.
- Un gain de 15% de performance ne justifie pas la complexité ajoutée, surtout que PyPy atteint déjà 2x. Les exigences additionnelles (PEP, infrastructure multi-JIT, approbation en 6 mois) semblent délibérément conçues pour étouffer le projet.
- La bureaucratie excessive étouffe l'innovation : d'autres langages (Ruby, C#, JVM) ont des JITs robustes sans ces obstacles, montrant que CPython est devenu rigide.
- Python devrait privilégier la flexibilité et la dynamicité plutôt que la performance brute ; si le JIT complique le langage au détriment de ces valeurs fondamentales, ce n'est pas le bon compromis.
