## Résumé de l'article

Un développeur expérimenté explique pourquoi il rejette régulièrement le code produit par des agents IA, malgré son fonctionnement, et préfère recommencer lui-même. La compréhension personnelle du problème et la qualité architecturale restent plus importantes que la rapidité d'exécution.

- Rejette le code IA quand il ne peut pas expliquer l'approche avec ses propres mots ou quand le diff dépasse la portée du problème
- Refuse les abstractions prématurées et le code qui complique la compréhension du système même s'il fonctionne localement
- Considère que l'acceptation trop rapide de changements générés par IA pose problème ; le code doit être adéquat, évolutif et extensible, pas juste fonctionnel
- Recommande une revue humaine obligatoire en parallèle des revues automatisées pour éviter une dépendance excessive aux modèles de langage
- Constate que même après consolidation de la compréhension du problème, les agents IA restent des outils guidés par l'ingénieur et ne peuvent pas résoudre les tâches complexes de manière autonome et durable

## Discussion sur Hacker News (148 commentaires)

**Avis positifs** :
- Le rejet de code AI qui fonctionne est justifié : un code qui marche n'est que le minimum requis. L'ingénierie logicielle exige des solutions adéquates, maintenables et extensibles, pas juste du code fonctionnel.
- Les développeurs seniors doivent pouvoir expliquer et défendre tout changement qu'ils intègrent, notamment en support client ou en debugging. Accepter du code qu'on ne comprend pas compromise cette responsabilité fondamentale.
- L'IA produit régulièrement du code sur-complexifié avec abstractions inutiles, patterns d'entreprise mal appliqués et duplication de code. Une révision critique reste nécessaire même quand les tests passent.
- Comprendre profondément le code écrit est un investissement productif à long terme : cela facilite la maintenance, réduit la dette technique et permet de déboguer efficacement, contrairement à la relecture passive du code AI.
- L'accumulation de code AI non compris crée des risques systémiques : dans les systèmes critiques (paiements, kernels, contrôle d'appareils), cette incompréhension peut avoir des conséquences graves.

**Avis négatifs** :
- Cette exigence de compréhension totale s'applique aussi au code humain : rejeter du code qu'on ne comprend pas immédiatement vaut pour tout contributeur, pas spécifiquement l'IA. L'article confond le symptôme avec la cause.
- L'utilisation d'abstractions et bibliothèques externes depuis toujours pose le même problème : on ne comprend pas tout ce qu'on utilise, et c'est acceptable avec des tests et des abstractions bien définies. L'IA ne change pas ce principe.
- Avec une bonne architecture, des linters custom et des tests écrits par l'IA mais validés, les développeurs peuvent déléguer efficacement la code generation tout en gardant le contrôle. Le cadre d'exécution, pas la compréhension ligne à ligne, assure la qualité.
- Pour du code peu critique (outils internes, prototypes, CRUD), le perfectionnisme total est une perte de temps. Le risque de vibe coding doit être proportionné au contexte et à l'impact réel du défaut.
- L'IA s'améliore avec du feedback et des contraintes : plutôt que rejeter du code AI, l'instruire via des linters, hooks et tests itératifs converge plus vite vers de la qualité acceptable que de le réécrire manuellement.
