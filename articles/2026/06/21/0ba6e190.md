## Résumé de l'article

Cet article, initialement publié en 2014, explique pourquoi il est souvent plus judicieux de tolérer la duplication de code plutôt que de forcer une abstraction inadéquate. Sandi Metz, auteure reconnue en conception orientée objet, démontre comment une abstraction bien intentionnée peut se dégrader progressivement lorsque de nouvelles exigences s'ajoutent, transformant le code en une suite de conditions intenables.

- Une abstraction initialement pertinente se détériore avec le temps quand chaque nouvelle exigence nécessite l'ajout de paramètres et de logique conditionnelle, rendant le code incompréhensible.
- Le biais du coût irrécupérable (sunk cost fallacy) pousse les développeurs à conserver et modifier du code complexe plutôt que de le réécrire, car ils sentent qu'abandonner l'effort investi serait du gaspillage.
- Quand une abstraction devient mauvaise, la meilleure stratégie consiste à réintroduire la duplication : dérouler le code abstrait dans chaque appelant, supprimer les branches inutiles, et permettre l'émergence de nouvelles abstractions pertinentes.
- Reconnaître que « ce code avait du sens à un moment, mais nous avons peut-être tout appris de lui » libère les développeurs pour repenser leurs abstractions selon les besoins actuels.
- Accepter la duplication temporaire offre une meilleure visibilité sur les différences réelles entre les cas d'usage et facilite l'extraction d'abstractions véritablement efficaces.

## Discussion sur Hacker News (192 commentaires)

**Avis positifs** :
- Les abstractions prématurées et mal pensées créent plus de complexité qu'elles ne résolvent, rendant le code difficile à maintenir et à étendre avec des configurations booléennes explosives
- Le duplication contrôlée permet de découvrir empiriquement les bonnes abstractions au lieu de les imposer avant de bien comprendre le domaine métier
- Les approches fonctionnelles et les petites fonctions simples réduisent naturellement la duplication sans générer les problèmes d'over-engineering des abstractions forcées
- Les mauvaises abstractions deviennent rapidement des passifs de couplage qui figent le système et résistent aux changements futurs, bien pires que des duplications gérables
- Le duplication isolée reste facile à maintenir et à refactoriser plus tard, tandis que les abstractions figées demandent des réévaluations complexes et risquées

**Avis négatifs** :
- À l'échelle réelle (5-100+ clients), maintenir des duplications cumulées génère une dette de maintenance insoutenable et des bugs en cascade lors de changements critiques
- La duplication rarement se gère correctement : elle s'accumule progressivement sans jamais être réabstraite, créant des divergences invisibles et des points d'incohérence
- Les duplications bien intentionnées dérivent rapidement en variantes subtiles incompatibles, rendant impossible de les refactoriser efficacement sans introducing des régressions
- Les bonnes abstractions (une fois identifiées) sont nettement moins coûteuses que la gestion décentralisée de dizaines de variations du même code réparti dans la base
- Les LLM amplifient le problème des duplications en les générant sans contrôle et en commettant des modifications inconsistentes, nécessitant une abstraction plus stricte, non moins
