## Résumé de l'article

Sem est un outil de compréhension sémantique du code qui s'appuie sur Git pour suivre les entités (fonctions, classes, etc.) plutôt que les lignes individuelles. Il offre une vue plus intelligible des changements de code en montrant ce qui s'est réellement passé du point de vue fonctionnel.

- Affiche les modifications au niveau des entités (fonctions, non lignes) : ajout, modification, suppression
- Améliore la lisibilité des diffs, blame, impact et logs en se concentrant sur les composants sémantiques
- Complète ou remplace les approches traditionnelles comme les serveurs de langage (LSP)
- Exemple concret : dans un fichier d'authentification, montre 3 changements d'entités (une fonction ajoutée, une modifiée, une supprimée) plutôt que les lignes affectées

## Discussion sur Hacker News (56 commentaires)

**Avis positifs** :
- Sem offre une granularité supérieure aux diffs ligne par ligne en analysant les entités (fonctions, classes, méthodes) et leur graphe de dépendances, ce qui aide les agents IA à comprendre l'impact réel des changements de code
- L'outil résout un vrai problème dans les larges monorepos TypeScript : identifier rapidement quels services, composants et tests dépendent d'une fonction modifiée, sans avoir à grepper manuellement
- Construire par-dessus Git plutôt que de créer un nouveau système de contrôle de version est une approche pragmatique et évite de réinventer la roue
- L'analyse structurelle du code ouvre des opportunités intéressantes comme l'analyse de flux de données, la détection de code mort, et la génération de tests comportementaux plutôt que des mocks
- Les performances sur de très gros répertoires ont été optimisées (6,5s sur 71K fichiers grâce à un cache SQLite), rendant l'outil pratique en production

**Avis négatifs** :
- Le benchmark affirmant que les agents IA sont 2,3x plus précis avec sem est non convaincant et trompeur : il ne prouve pas que l'outil améliore réellement le codage ou la correction de bugs en pratique
- Le processus de setup/unsetup qui modifie le comportement par défaut de git diff est confus et perçu comme hostile : les utilisateurs voudraient un simple flag CLI plutôt qu'une installation invasive
- L'approche actuelle reste structurelle et basée sur l'analyse statique, loin de l'analyse complète de flux de données (taint analysis) nécessaire pour vraiment tracer la propagation des changements à travers copies et transformations
- L'ajout d'un nouveau langage (comme Julia) n'est pas trivial et les capacités d'analyse de type sont limitées comparées aux LSP, même si tree-sitter offre une couverture multilangage plus rapide
- Les cas d'usage réels manquent dans la documentation : les exemples restent génériques et ne montrent pas clairement quand sem serait vraiment indispensable pour humains ou agents
