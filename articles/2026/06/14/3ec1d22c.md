## Résumé de l'article

Apple a remplacé l'implémentation en C de l'interpréteur de hinting TrueType — le composant qui optimise le rendu des polices vectorielles sur écrans basse résolution — par une version en Swift pour les versions d'automne 2025. Cette réécriture améliore la sécurité en éliminant les vulnérabilités mémoire tout en gagnant 13% de performance moyenne.

- L'interpréteur TrueType est une surface d'attaque critique puisqu'il traite des données de polices provenant de sources non fiables (web, PDF, etc.)
- Apple a atteint la compatibilité pixel-parfaite avec l'implémentation C en utilisant des tests unitaires (couverture 99,7%) et un corpus de 4 200 documents PDF minimisés contenant 27 millions de glyphes
- Les optimisations de performance reposent sur l'adoption de types noncopyable, l'élimination des allocations éphémères, l'utilisation de projection types pour accéder aux structures C en toute sécurité, et l'inlining
- Le code Swift conserve une petite surface d'interopérabilité C à la frontière des modules avec des déclarations unsafe minutieusement vérifiées
- Le code source de l'interpréteur Swift est publié sur GitHub comme référence de implémentation ; Apple a aussi généré des instructions pour aider les assistants de codage LLM à convertir du C/C++ en Swift

## Discussion sur Hacker News (124 commentaires)

**Avis positifs** :
- Apple étend systématiquement la récriture en Swift (RIS) à tous les niveaux du système d'exploitation, confirmant une stratégie cohérente de migration vers les langages à mémoire sûre
- Le projet démontre que Swift peut atteindre des performances élevées en systèmes critiques grâce à une conception minutieuse et des tests exhaustifs (4x plus de code de test que de code interpréteur)
- Swift offre une meilleure ergonomie que Rust pour les cas d'usage d'Apple : interopérabilité fluide avec Objective-C et C++, ABI stable, comptage de références, et pas besoin de .clone() explicite partout
- Cette récriture en Swift améliore la sécurité du composant TrueType en éliminant les vulnérabilités de corruption de mémoire présentes dans l'ancienne implémentation C++
- L'approche d'Apple avec des lifetimes et la propriété unique coexistant avec le comptage de références offre plus de flexibilité que Rust pour certains patterns de code

**Avis négatifs** :
- Le compilateur Swift et ses fonctionnalités de lifetime (lifetime features) présentent encore des bugs importants causant des crashes avec des programmes simples, limitant leur utilisabilité
- Swift reste une langue plus complexe que nécessaire pour certains cas d'usage et les vérificateurs de type sont historiquement lents, obligeant les développeurs à fragmenter le code
- Apple utilise massivement Claude Code en interne au lieu de tester et d'itérer sur ses propres agents Xcode, ce qui inquiète concernant la qualité et la dépendance aux modèles externes
- La décision d'Apple de ne pas soutenir les écrans basse résolution (1080p) pose problème comparé à Windows et Linux qui gèrent correctement le HiDPI, limitant l'accès aux utilisateurs avec du matériel plus ancien
- Contrairement à Microsoft qui a obtenu des gains de performance substantiels en Rust pour DirectWriteCore (5-15%), l'article ne quantifie pas clairement les bénéfices en performance de cette réécriture en Swift
