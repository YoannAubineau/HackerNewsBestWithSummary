## Résumé de l'article

Lore est un système centralisé de contrôle de version développé par Epic Games, optimisé pour gérer des contenus binaires volumineux. Il représente l'état du dépôt sous forme d'arbres de Merkle et de chaînes de révisions immuables.

- Architecture basée sur l'adressage par contenu (content-addressed) pour une meilleure traçabilité
- Stockage optimisé en priorité pour les fichiers binaires, avec déduplication intégrée
- Hydratation clairsemée et à la demande (sparse/on-demand) pour réduire la consommation de ressources à grande échelle
- Les dépôts Lore sont accessibles sur le GitHub d'Epic Games

## Discussion sur Hacker News (199 commentaires)

**Avis positifs** :
- Lore répond à un vrai besoin dans le développement de jeux : gérer les fichiers binaires volumineux (textures, modèles 3D, audio) que Git et Git-LFS gèrent mal, avec chunking natif et verrouillage de fichiers
- L'interface CLI simplifiée et la documentation utilisateur sont plus accessibles que Git pour les artistes et développeurs non-spécialistes, contrairement aux outils actuels comme Perforce
- Perforce étant un système vieillissant coûteux en maintenance, Lore offre une alternative open-source sérieuse pour les studios, notamment avec le support natif dans Unreal Engine
- L'architecture centralisée permet des contrôles de permissions et d'accès plus granulaires adaptés aux besoins des studios (NDAs, données propriétaires), ce que Git ne peut pas faire facilement
- Avoir enfin un concurrent à Perforce dans l'écosystème gamedev, où la standardisation avait créé une dépendance technologique problématique

**Avis négatifs** :
- La documentation semble générée par IA avec du verbatim inutile et peu de relecture humaine, ce qui inspire peu de confiance pour un outil critique comme la gestion de version
- Le desktop client n'est pas encore open-source malgré l'affirmation de 'fully open source', les binaires dépendent de composants propriétaires d'Epic
- Epic Games a un historique préoccupant : suppression du support Linux pour Rocket League après l'acquisition de Psyonix, pratiques commerciales agressives, ce qui suscite de la méfiance
- L'approche par chunking de 64 KiB pourrait être inefficace pour les petits changements (1 KiB) et augmenter significativement l'espace disque utilisé et les transferts réseau
- Git+LFS reste une solution viable pour beaucoup de cas d'usage, et il est déjà largement adopté ; le risque de fragmentation du marché des VCS sans amélioration suffisante reste réel
