## Résumé de l'article

GitHub, a code repository platform owned by Microsoft, suffers from severe frontend performance and reliability issues driven by aggressive AI feature integration and architectural bloat. The article compares GitHub's frontend code and performance against competitors GitLab and Codeberg through empirical analysis.

- GitHub downloads 540 KiB of code just to display an empty repository—more than the entire original DOOM game—spread across hundreds of CSS and JavaScript files, causing 22-second load times on fast connections.
- GitHub's official changelog in 30 days mentions "Copilot" 59 times but "performance" and "reliability" zero times, contradicting stated priorities and revealing that AI feature development is prioritized over infrastructure stability.
- Heap memory usage is bloated (GitHub and GitLab both exceed 100 MiB for simple pages) and compression support is inadequate, wasting bandwidth and slowing load times across all network types.
- Codeberg offers comparable functionality with far smaller code footprint and faster load times (3 seconds vs 22+ for GitHub), demonstrating that the bloat is not inherent to the feature set.
- The root cause appears to be LLM-generated code, webpack over-splitting, aggressive AI button insertion, and disconnected technical leadership that prioritizes corporate AI initiatives over user experience and cost efficiency.

## Discussion sur Hacker News (112 commentaires)

**Avis positifs** :
- GitHub a effectivement des problèmes de fiabilité documentés : plus d'une panne par jour en moyenne, avec des priorités affichées (disponibilité, capacité, nouvelles fonctionnalités) démenties par les changelogs (59 mentions de Copilot contre 0 pour performance/fiabilité en 30 jours)
- L'article identifie un problème réel : les données critiques (issues, PRs, wikis, discussions, CI/CD) sont captives de GitHub, rendant la migration vers d'autres plateformes très difficile malgré la portabilité théorique du code Git lui-même
- Les alternatives viables existent et gagnent en adoption : Codeberg, Forgejo, GitLab auto-hébergé, Gitea offrent des expériences plus légères et stables, avec des utilisateurs rapportant une meilleure performance et une interface plus intuitive
- La charge accrue d'agents IA autonomes (Claude, etc.) surcharge l'infrastructure GitHub, ce que Microsoft n'a pas dimensionnée correctement malgré des ressources massives, reflétant un problème de priorités d'ingénierie
- L'auto-hébergement (Forgejo sur NAS Synology, Mac Mini avec Tailscale, VPS) devient praticable pour les petits projets et offre une autonomie retrouvée par rapport à une dépendance croissante envers une entreprise

**Avis négatifs** :
- GitHub reste le choix par défaut incontournable en raison d'effets réseau: l'inertie des millions de repos existants, la métrique de prestige des stars, et le fait que 'le monde entier est sur GitHub' rendent impraticable une migration généralisée malgré les problèmes
- Les solutions décentralisées ou alternatives fragmentent l'écosystème plutôt que de le consolider : gérer issues, PRs, wikis sur plusieurs outils (Youtrack, Gerrit, Confluence, Zulip) crée plus de points de défaillance et complique la recherche d'information par rapport à une plateforme unifiée
- Les alternatives comme GitLab auto-hébergé demandent beaucoup de maintenance (image Docker de 15GB, ressources importantes, sécurité complexe, mises à jour régulières) et ne résolvent pas vraiment le problème de dépendance vis-à-vis d'une plateforme propriétaire
- La majorité des développeurs sont satisfaits de GitHub : actions CI/CD fiables, Copilot utile, performances généralement acceptables pour la plupart des usages malgré quelques pannes mineures, ce qui contredit le catastrophisme de l'article
- L'article manque de nuance sur les causes réelles : l'explosion d'activité due aux agents IA autonomes en quelques mois est un défi d'infrastructure sans précédent que peu d'équipes auraient pu anticiper et résoudre rapidement
