## Résumé de l'article

Un développeur a identifié une campagne massive de distribution de malware sur GitHub : dix mille dépôts différents, créés par des contributeurs distincts et non des forks, contiennent tous un schéma commun — ils clonent des projets existants, suppriment et recréent régulièrement les commits pour ajouter des liens vers des archives ZIP contenant des chevaux de Troie.

- Le schéma détecté : les dépôts modifient uniquement le fichier README toutes les quelques heures en supprimant le commit précédent, ajoutent un lien vers une archive ZIP malveillante, et copient l'historique complet d'autres projets pour simuler de la légitimité
- L'auteur a développé un script exploitant GitHub Archive pour identifier les dépôts mis à jour fréquemment, puis filtré les résultats selon ce pattern spécifique, trouvant 10 000 correspondances exactes (25 % des 40 000 candidats)
- Les archives malveillantes contiennent des exécutables (.exe) qui passent inaperçus dans VirusTotal en tant qu'archive, mais sont détectés une fois l'archive extraite
- Les hypothèses sur les objectifs : contourner les algorithmes de sécurité de GitHub en modifiant régulièrement les commits, et monétiser le malware en rangeant les dépôts clonés dans des tags populaires pour améliorer leur indexation dans les moteurs de recherche
- GitHub a commencé à supprimer les dépôts identifiés après publication de l'article ; le support GitHub n'avait pas réagi pendant deux mois à la première demande de suppression

## Discussion sur Hacker News (102 commentaires)

**Avis positifs** :
- GitHub ne répond pas efficacement aux signalements de malware : certains repos restent en ligne pendant des mois malgré les rapports, tandis que d'autres sont supprimés en 24 heures sans cohérence apparente.
- Le problème révèle l'inadéquation du modèle de confiance open source : beaucoup supposent à tort que le code accessible signifie qu'il est sûr, sans réaliser que personne n'a le temps d'auditer réellement les sources ou de vérifier que les binaires correspondent au code.
- Les attaquants exploitent les faiblesses de la découverte logicielle : SEO optimisé, mises à jour fréquentes pour remonter dans les classements, et noms similaires aux projets légitimes trompent facilement les utilisateurs et les agents automatisés.
- Les systèmes de détection existants sont inadéquats : VirusTotal ne détecte pas le malware quand il est compressé dans une archive, et GitHub n'effectue pas de scannes systématiques des releases avec des outils antivirus tiers.
- L'usurpation d'identité des développeurs s'ajoute au problème : des projets frauduleux reprennent les noms des développeurs légitimes, amplifiant la confusion et la portée des attaques.

**Avis négatifs** :
- Le problème décrit n'est pas vraiment lié à l'open source : ce sont essentiellement des binaires malveillants hébergés sur GitHub avec des liens dans les README, comparable à télécharger des exécutables d'un inconnu sur un forum.
- Le principe que l'open source est intrinsèquement plus sûr reste valide : il offre la transparence nécessaire pour une validation tierce, même si cela exige de la vigilance; c'est un avantage relatif par rapport au code propriétaire.
- Les utilisateurs et les développeurs ne font pas preuve de diligence raisonnable : conclure qu'un dépôt est sûr parce qu'il a 10k étoiles ou a été audité par 'quelqu'un d'autre' est une responsabilité individuelle, pas un défaut du modèle open source.
- Les faux positifs rendaient les solutions de détection automatique impraticables : Microsoft et GitHub ont probablement essayé de scanner les malware mais l'ont abandonné en raison des taux d'erreur nuisant à l'expérience utilisateur.
- Des outils de détection existent déjà : Socket, Aikido, Step-Security et Wiz proposent des analyses continues de malware dans les dépendances open source.
