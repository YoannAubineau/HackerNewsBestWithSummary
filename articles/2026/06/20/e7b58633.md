## Résumé de l'article

F-15 Strike Eagle II est un jeu de simulation aérienne de 1989 dont un projet de rétro-ingénierie reconstitue le code source en C à partir des binaires originaux. Le projet vient de franchir une étape majeure : tout le code C a été reconstitué pour tous les exécutables, et les développeurs recherchent maintenant des testeurs pour identifier les bugs avant un prochain portage.

- Le projet a atteint v0.9.1 avec tous les exécutables reconstitués en C et tous les bugs optiques et de structure d'données à chasser
- Les testeurs doivent remplacer les exécutables originaux (version 451.03 avec l'extension Desert Storm) et signaler crashes, glitches graphiques et touches non-fonctionnelles
- Il s'agit d'une reconstruction "bug-for-bug" : les comportements anormaux du jeu original doivent être conservés pour l'instant
- Les testeurs sont invités à disposer d'une copie du jeu original pour vérifier qu'un problème n'existe pas déjà dans la version originale avant de faire un rapport

**Avis positifs** :
- Le projet de reverse engineering est techniquement impressionnant : conversion complète en assembleur puis en code C compilé, avec une méthodologie rigoureuse permettant des améliorations futures.
- L'accès au code source ouvre des possibilités de modification et portage bien supérieures à l'émulation simple : corrections de bugs, nouvelles fonctionnalités, portage sur Windows/Linux, amélioration graphique, etc.
- L'IA s'avère très utile pour analyser le code décompilé et comprendre la structure même sans symboles ni source maps originaux.
- Le projet respecte l'intégrité légale en requérant les assets originaux du jeu et documentant explicitement ses objectifs de reconstruction.
- La communauté retro-gaming bénéficie d'une excellente documentation (dev blog reconnu comme l'une des meilleures sur le reverse engineering rétro) et retrouve accès à des jeux emblématiques dégradés ou perdus.

**Avis négatifs** :
- Le reverse engineering soulève des préoccupations légales complexes autour des droits d'auteur et des marques, même si l'approche du projet est transparente.
- DOSBox et les émulateurs modernes suffisent amplement pour rejouer à ces anciens jeux sans effort technique considérable.
- Le processus de reverse engineering introduit des bugs subtils difficiles à tracer dans le code converti, et la phase de débogage peut s'avérer laborieuse.
