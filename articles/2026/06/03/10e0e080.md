## Résumé de l'article

VSCode webview, un système de virtualisation pour afficher du contenu de manière isolée, contient une faille de sécurité qui permet aux attaquants de voler des tokens GitHub d'accès complet. En cliquant sur un lien, un attaquant peut injecter du JavaScript malveillant dans un Jupyter notebook hébergé sur github.dev, qui installe ensuite une extension VSCode personnalisée donnant un accès complet aux données GitHub de la victime, y compris les dépôts privés.

- VSCode utilise des webviews (iframes) pour isoler les contenus non fiables, mais permet la transmission d'événements clavier via postMessage pour maintenir l'ergonomie utilisateur
- La faille permet au code malveillant d'émettre des événements clavier synthétiques (Ctrl+Shift+A, Ctrl+F1) pour déclencher des commandes VSCode sans intervention réelle de l'utilisateur
- L'attaquant combine un notebook Jupyter contenant du code malveillant avec une extension locale recommandée dans .vscode/extensions.json pour contourner les contrôles de sécurité du publisher trust
- Une fois l'extension installée, elle peut accéder au token GitHub de l'API et lister tous les dépôts privés auxquels l'utilisateur a accès
- Le chercheur en sécurité a opté pour la divulgation complète plutôt que de le signaler à Microsoft, invoquant ses mauvaises expériences antérieures avec leur processus de rapport de bugs VSCode

## Discussion sur Hacker News (40 commentaires)

**Avis positifs** :
- Le chercheur en sécurité a raison de publier publiquement : Microsoft/MSRC ignore systématiquement les rapports de vulnérabilité, corrige silencieusement les bugs sans créditer les chercheurs, ce qui est un manquement grave aux bonnes pratiques de divulgation responsable.
- L'architecture de github.dev est défectueuse : stocker des tokens GitHub avec permissions complètes dans un IDE web accessible est une violation du principe de défense en profondeur, alors que les Codespaces offrent déjà une portée par-repository.
- Le problème systémique de Microsoft s'étend au-delà de VSCode : la mauvaise gestion des rapports de sécurité (clôture sans vérification, manque de communication interne) révèle une culture d'entreprise qui ne valorise pas les chercheurs en sécurité.
- Les utilisateurs doivent adopter des mesures drastiques : révocation régulière des tokens, utilisation d'outils alternatifs plus légers (Neovim, Zed, VsCodium), opération sous l'hypothèse que tout credential peut être compromis.
- Le writeup technique est exemplaire : il donne les détails nécessaires pour que la communauté comprenne la menace sans servir de simple tutoriel d'exploitation.

**Avis négatifs** :
- La divulgation publique d'une vulnérabilité non corrigée peut mettre en danger les utilisateurs avant qu'un patch soit disponible, même si cela crée une pression sur le vendeur.
- Le chercheur pourrait être sanctionné par Microsoft pour cette divulgation agressive, ce qui dissuadera d'autres de signaler des bugs à l'avenir.
- Il existe des explications bénignes possibles à la gestion de MSRC : la vulnérabilité peut avoir été corrigée avant que l'équipe de sécurité ne valide le rapport, ce qui paraît invalide de leur côté sans visibilité sur l'historique interne.
- Les alternatives proposées (Zed, Neovim) présentent aussi leurs propres problèmes de sécurité et de vie privée (Zed télécharge des binaires sans demander permission, configuration complexe), donc la solution ne se réduit pas à changer d'outil.
- Une partie du problème vient des utilisateurs qui acceptent trop facilement les risques liés à des extensions et des intégrations sans comprendre la surface d'attaque qu'elles créent.
