## Résumé de l'article

Nub est un toolkit Node.js augmentant l'écosystème existant sans le remplacer, écrit en Rust pour offrir une expérience développeur proche de Bun. L'outil fonctionne comme surcouche du Node.js standard et inclut support TypeScript natif, gestionnaire de versions, exécuteur de scripts, et gestionnaire de paquets.

- Lance les fichiers TypeScript/JSX sans étape de build et supporte la résolution extensionless, les chemins tsconfig.json, les décorateurs, et le chargement automatique des fichiers .env
- Exécute les scripts npm/pnpm environ 24 fois plus rapidement via une implémentation Rust sans démarrage JavaScript
- Lance les binaires locaux (npx/pnpm dlx) environ 19 fois plus vite en éliminant le double appel Node.js
- Gère automatiquement les versions de Node.js en fonction des fichiers .node-version, .nvmrc ou package.json, avec installation auto-cachée
- Offre un mode watch natif, un gestionnaire de paquets compatible pnpm avec vérifications de sécurité (osv.dev, provenance), et des shims globaux de style Corepack pour pnpm/npm/yarn

## Discussion sur Hacker News (73 commentaires)

**Avis positifs** :
- Approche pragmatique : Nub améliore l'expérience Node.js sans réécrire l'écosystème, contrairement à Bun, en s'appuyant sur les technologies existantes plutôt que de les remplacer
- Performance remarquable : Migration d'un monorepo entier en une heure, overhead négligeable en temps et mémoire grâce aux hooks synchrones récents de Node.js
- Adoptabilité progressive : Compatible avec pnpm/npm/yarn/bun, fonctionne avec les lockfiles existants sans churn, permet adoption par étapes
- Zéro fragmentation : Pas d'API Nub-spécifiques (pas de global Nub, pas de config Nub), aucune dépendance propriétaire, maximise la portabilité du code
- Résolution TypeScript complète : Supporte tsconfig.json#paths, imports sans extension, decorators et JSX contrairement au type-stripping basique de Node

**Avis négatifs** :
- Overhead à la transpilation : Reste encore une couche d'abstraction par rapport à Node pur, nécessite des hooks et préload avec impact potentiel en production
- Support ESM limité : Utilise --require (CommonJS) par défaut pour des raisons de performance, pose des questions sur la robustesse du support des cas limites ESM comme top-level await
- Compilation manquante : Feature critique attendue (--compile comme Bun) n'existe pas encore, limite l'adoption en production pour SCP d'un binaire unique
- Réécriture fondamentale impossible : Certains problèmes architecturaux de Node (JSON sans commentaires dans package.json) ne peuvent pas être fixés sans fork majeur, limitant les améliorations DX possibles
- Jeunesse et risques : Projet très récent avec coauteurs impliqués dans la décision, potentiel d'instabilité ou de pivots stratégiques comparé à Node établi
