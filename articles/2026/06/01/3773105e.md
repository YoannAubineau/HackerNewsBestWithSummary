## Résumé de l'article

Des versions malveillantes ont été découvertes dans 30 paquets NPM du registre `@redhat-cloud-services` hébergé sur NPM. Ces paquets fournissent des composants et clients utilisés dans l'écosystème Red Hat Cloud Services.

- Chaque paquet affecté compte entre 2 et 3 versions compromises, généralement espacées d'une à deux versions mineures
- Les paquets compromis incluent des composants frontend critiques (`frontend-components`, `chrome`), des clients API (`compliance-client`, `rbac-client`, `insights-client`) et des utilitaires partagés
- La compromission suit un schéma régulier : versions X.X.1/2/4, X.X.3/4/6 ou X.X.11/12/14, suggérant une injection systématique
- Les développeurs utilisant ces paquets doivent vérifier leurs dépendances et mettre à jour vers des versions saines en dehors des plages compromises
- L'incident a été signalé par StepSecurity et documenté dans les flux de sécurité OSS

## Discussion sur Hacker News (309 commentaires)

**Avis positifs** :
- Les attaques supply-chain récentes (TrapDoor, Shai-Hulud) montrent que le problème s'étend à PyPI, Cargo et Composer : ce n'est pas uniquement un problème NPM mais un enjeu systémique d'écosystèmes de packages.
- Des solutions pratiques existent et se déploient : cooldown minimum de 1-7 jours avant installation, outils comme pnpm (par défaut depuis v11), Yarn 4 et npm v11 qui implementent ces garde-fous, permettant aux scanners de sécurité de détecter les attaques.
- NPM a commencé à agir avec des avertissements sur les scripts d'installation et un RFC pour les désactiver par défaut à terme, bien que tardivement.
- La sévérité relative dépend du contexte : les distributions Linux (apt/dnf) et Maven présentent moins d'incidents car ils s'appuient sur des mainteneurs de confiance et des processus de vérification, contrairement à npm où n'importe qui peut publier.
- L'écosystème JavaScript souffre d'une culture de micro-packages et d'absence de stdlib, multipliant les points d'entrée vulnérables, ce qui explique partiellement l'écart de fréquence avec d'autres langages.

**Avis négatifs** :
- NPM permet l'exécution arbitraire de code post-installation par défaut, sans isolation ni sanboxing, exposant l'intégralité de la machine utilisateur—bien plus grave que simplement installer du code non approuvé.
- Comparaisons défensives ('tous les package managers font ça') nient la réalité : npm ne nécessite même pas d'exécuter le package pour déclencher le malware, ouvrir un projet dans un IDE suffit, ce que d'autres écosystèmes évitent.
- Malgré seize ans d'existence, npm n'a pris aucune action structurelle sérieuse jusqu'en 2025 ; les solutions proposées (cooldowns, opt-in explicite) sont des rustines, pas des changements de design fondamental.
- La culture NPM favorise l'installation des dernières versions sans réfléchir et l'absence de révision des dépendances par des mainteneurs responsables, créant une 'wormabilité' autoreplicante que pnpm ou les alternatives offrent des contrôles plus fins mais ne peuvent pas imposer.
- Les pincements de version et les cooldowns ne sont efficaces que si largement adoptés par défaut, ce qui n'est pas le cas ; même avec des délais, les attaquants s'adapteront avec des bombes à retardement, rendant ces mesures insuffisantes seules.
