## Résumé de l'article

Un développeur a reçu une proposition d'emploi par LinkedIn d'une startup crypto, lui demandant de réviser un dépôt GitHub. En analysant le code avec prudence sur une machine isolée, il a découvert une porte dérobée camouflée dans un fichier de test qui s'exécute automatiquement après npm install, prête à recevoir et exécuter du code malveillant depuis un serveur distant.

- La porte dérobée était cachée dans app/test/index.js, déguisée en suite de tests, et exécutait des requêtes vers un serveur C2 pour recevoir une deuxième charge malveillante
- Le fichier s'exécutait automatiquement via le script "prepare" de package.json qui lance npm automatiquement après npm install
- L'instruction de vérifier les "dépendances Node obsolètes" était un prétexte pour amener la cible à installer les dépendances du projet
- La recruiter utilisait le profil usurpé d'une journaliste culturelle réelle, tandis que le dépôt était signé par les commits d'un vrai développeur dont l'identité avait été usurpée
- L'auteur a utilisé un agent en lecture seule pour analyser le code, ce qui a permis de détecter rapidement la porte dérobée; le dépôt restait toujours en ligne au moment de la rédaction malgré les signalements

## Discussion sur Hacker News (79 commentaires)

**Avis positifs** :
- Les tactiques de phishing ciblant les développeurs se multiplient sur LinkedIn avec des repos malveillants et des scripts déguisés, confirmant la croissance de ces attaques sophistiquées
- Le timing et l'urgence du contexte d'entretien d'embauche rendent les développeurs vulnérables, car ils cherchent à paraître compétents et ne posent pas de questions de sécurité basiques
- npm et les scripts curl | bash présentent des risques réels d'exécution de code arbitraire à l'installation, révélant une faille structurelle dans la distribution de paquets JavaScript
- La capacité limitée de LinkedIn à vérifier l'identité des recruteurs et à modérer les faux profils agrave le problème malgré les rapports d'utilisateurs

**Avis négatifs** :
- Les gouvernements et les entreprises technologiques manquent de volonté politique pour agir : les arnaqueurs opèrent depuis pays non-coopérants, et les solutions existantes (IC3, DMCA) restent inefficaces
- Certains commentaires suggèrent que les inquiétudes sur les failles de sécurité (comme les appels de phishing généralisés) pourraient être résolues techniquement si les entreprises le souhaitaient réellement, mais elles ne le font pas
- La responsabilité incombe aux utilisateurs d'être « hyper-vigilants » plutôt qu'aux plateformes de sécuriser leurs services, ce qui est un échec systémique
- Les solutions proposées (KYC sur les numéros de téléphone, régulation stricte) comportent des risques de surveillance et d'exclusion plus larges qu'elles ne résoudraient de problèmes
