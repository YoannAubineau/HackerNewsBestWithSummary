## Résumé de l'article

zeroserve est un serveur HTTPS haute performance qui exécute des scripts eBPF en espace utilisateur. Il supporte désormais un mode de compatibilité Caddy qui compile les fichiers Caddyfile en bytecode eBPF, puis en code machine natif x86_64/ARM64, exécuté dans une boucle io_uring, offrant 3x le débit et 70% de latence réduite par rapport à Caddy classique.

- Compilation JIT des Caddyfiles en eBPF puis en code machine natif pour exécution optimisée
- Support complet des directives Caddy inversées proxy sur une machine de test 2-threads AMD Ryzen 7 3700X
- Capacité à exécuter du code eBPF turing-complet personnalisé, par exemple middleware AWS SigV4 pour proxy S3
- Syntaxe d'appel `zeroserve_call` intégrée aux Caddyfiles pour invoquer des fonctions C compilées en eBPF
- Distribution via binaires pré-compilés pour différentes architectures (x86_64, ARM64)

## Discussion sur Hacker News (59 commentaires)

**Avis positifs** :
- Les améliorations de performance (3x throughput, 70% latence réduite) sont techniquement impressionnantes, particulièrement pour les cas d'usage haute charge comme les connexions courtes sans session resumption ou les très hauts RPS de petites réponses
- zeroserve offre une architecture plus épurée que nginx, potentiellement plus facile à auditer et maintenir, avec une codebase plus propre
- eBPF en userspace permet une flexibilité et des optimisations sans les contraintes du vérificateur du kernel, ouvrant des possibilités de performance supérieures
- Le projet adresse un besoin réel pour les CDN et autres cas haute performance où les proxies deviennent effectivement goulot d'étranglement

**Avis négatifs** :
- Absence de support ACME (gestion automatique des certificats), une fonctionnalité critique manquante par rapport à Caddy
- Manque de track record, d'écosystème de plugins et de support comparé à nginx/Caddy établis ; risque d'abandon ("vibe project") et aucun soutien d'une grande organisation contrairement à Envoy (Lyft) ou nginx (Rambler)
- Surface d'attaque massive due à la JIT compilation en userspace et à l'exposition d'io_uring via des requêtes HTTP contrôlées à distance, avec vulnérabilités de sécurité récentes d'io_uring
- Benchmark très synthétique (requêtes HTTP/1.1 ultra-courtes) qui ne reflète pas les cas d'usage réels ; la plupart des applications voient le backend comme vrai goulot d'étranglement, pas le proxy
- Interface mTLS problématique dans le navigateur (popup de sélection de certificat confuse) signalée comme suspecte par les utilisateurs, sans bénéfice clair pour l'accès public
