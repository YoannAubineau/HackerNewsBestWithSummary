## Résumé de l'article

Un auteur a découvert que Kobo utilise RMSDK, le moteur de rendu propriétaire d'Adobe datant de 2010, qui rejette des fichiers ePub pourtant valides selon epubcheck (la norme d'or du secteur). Son livre passait tous les tests epubcheck et fonctionnait sur Kindle, Apple Books et autres lecteurs, mais Kobo le déclarait « corrompu ».

- RMSDK, utilisé par Kobo pour tous ses appareils, n'a jamais été modernisé : son analyseur CSS est figé vers 2013, sans support pour flexbox, grid, ou les fonctions mathématiques CSS level 4
- Le problème venait d'une ligne CSS valide `max-width: min(150px, 30vw)` qui n'était pas reconnue par RMSDK, ce que epubcheck ne peut pas détecter puisqu'il teste la conformité générale, pas la compatibilité avec des rendus défaillants
- Kobo n'affiche aucun message d'erreur : le livre ne se charge simplement pas, sans feedback utilisateur
- L'auteur a dû décomposer manuellement son fichier pour identifier le problème, puis revenir à une syntaxe CSS plus ancienne
- Cette situation illustre le fossé entre les standards ouverts (ePub) et des implémentations fermées motivées par le DRM plutôt que par l'expérience de lecture

## Discussion sur Hacker News (93 commentaires)

**Avis positifs** :
- Adobe Digital Editions ne respecte pas les standards CSS et devrait ignorer les propriétés invalides plutôt que de rejeter les documents entiers, ce qui viole même CSS 1.0 de 1996 sur la compatibilité avant
- L'ePub est fondamentalement un format ouvert et raisonnable (HTML+ZIP) contrairement au PDF qui est un cauchemar architectural ; les navigateurs web peuvent afficher les contenus ePub sans problème
- Les générateurs ePub comme Calibre offrent une excellente expérience d'édition en HTML pur, validant l'approche du standard
- Des alternatives existent pour contourner les problèmes : KOReader, Kepubify, et les convertisseurs en ligne améliorent la compatibilité sans dépendre des lecteurs défaillants
- Adobe a démontré un historique chronique d'abandon de standards (Flash, support médiocre) et vient de vendre sa technologie ePub à Wipro, confirmant le manque d'investissement

**Avis négatifs** :
- Le passage du W3C à des standards vivants sans versioning (WHATWG HTML) a rendu les ePubs existantes non-conformes rétroactivement ; les outils recommandent toujours ePub 3.1 ou 2.0
- L'ePub impose HTML/CSS alors qu'un sous-ensemble simple et performant serait préférable pour les lecteurs ; cibler le dénominateur commun le plus bas reste nécessaire
- Il n'existe pas d'infrastructure de test standardisée comparable à CanIUse pour les lecteurs ePub ; seul Amazon fournit généreusement des plateformes de test
- Les droits d'auteur numériques fragmentent l'écosystème : Apple, Amazon et Adobe maintiennent des stacks propriétaires fermées, bloquant la concurrence et l'interopérabilité
