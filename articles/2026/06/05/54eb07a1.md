## Résumé de l'article

Ladybird, un navigateur web en développement open source, change sa politique de contribution en cessant d'accepter les pull requests publiques. Seuls les mainteneurs du projet pourront dorénavant introduire des modifications de code, une évolution justifiée par les risques de sécurité accrus et l'impact des outils d'IA sur la qualité et la traçabilité des contributions.

- Le projet ferme tous les pull requests publics actuels et n'acceptera plus que les modifications proposées par les mainteneurs eux-mêmes
- La justification principale est la vulnérabilité des navigateurs aux attaques : un navigateur exécute du contenu non fiable sur la machine de l'utilisateur, et une vulnérabilité bien déguisée suffit pour compromettre la sécurité
- Les outils d'IA ont diminué le coût et le temps nécessaires pour produire des contributions apparemment fiables, rendant plus difficile l'évaluation de la bonne foi du contributeur uniquement par la qualité du code
- Ladybird reste open source avec du code publiquement accessible ; les contributions externes sont toujours bienvenues sous forme de rapports de bugs, tests, retours techniques et discussions sur les standards, mais pas de code
- Le projet justifie ce changement par son passage à la phase de production : en se préparant à livrer un navigateur aux utilisateurs réels, le processus de développement doit correspondre à cette responsabilité

## Discussion sur Hacker News (289 commentaires)

**Avis positifs** :
- La décision est justifiée : les PR générées par IA ont massivemenr augmenté le volume de contributions de faible qualité, créant une charge insoutenable pour les mainteneurs et rendant impossible de distinguer les contributions honnêtes du « slop »
- Ce modèle n'est pas nouveau ni problématique : SQLite, Lua et d'autres projets reconnus fonctionnent ainsi depuis longtemps avec succès, prouvant qu'une développement fermé n'est pas incompatible avec du vrai open source
- C'est une réponse pragmatique à un problème réel : les mainteneurs doivent protéger leur temps et la qualité du projet, et l'examen de code reste coûteux même quand la génération de code devient gratuite
- Le projet reste open source et modifiable : la licence BSD-2 demeure inchangée, le code est public et forkable, seule la contribution directe est fermée, ce qui ne supprime pas les libertés fondamentales
- Cela pourrait revitaliser les communautés de confiance : un retour aux modèles pré-GitHub basés sur des relations humaines réelles et la confiance, plutôt que sur l'accumulation de contributions anonymes sans vérification

**Avis négatifs** :
- C'est la mort du modèle communautaire d'open source : fermer les contributions externes tue la raison d'être historique d'OSS, transforme « open source » en simple licence, et détruit les voies traditionnelles pour que les novices deviennent mainteneurs
- Comment recruter de nouveaux mainteneurs sans contributions externes ? Sans la possibilité de contribuer et de construire une réputation, comment identifier et intégrer les futurs responsables du projet
- Ladybird perd l'immense valeur non-monétaire des contributeurs : retours d'utilisateurs, détection de bugs, idées, scrutin du code et effort gratuit qui était le fondement de la viabilité économique d'OSS
- C'est une limitation artificielle qui semble exagérée : on pourrait imposer des garde-fous (petites PR seulement, vérification d'identité, système de réputation) sans tout fermer, comme d'autres projets le démontrent
- Le risque caché d'une concentration dangereuse du pouvoir : concentrer tout le code dans les mains de quelques personnes facilite les « rug pulls » et les changements de licence, tandis qu'une contribution distribuée offrait une protection
