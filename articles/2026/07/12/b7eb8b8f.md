## Résumé de l'article

Cet article compare l'anxiété actuelle des programmeurs face aux modèles de langage (LLM) à la disruption qu'a causée la CGI en 1993 à Phil Tippett, un maître du stop-motion. L'auteur argue que l'adaptation technologique est cyclique et que les développeurs doivent évoluer pour rester pertinents.

- Les LLM sont un nouvel outil comme les autres révolutions technologiques passées (web, mobile) : les programmeurs qui apprennent à les utiliser efficacement prospèreront, tandis que ceux qui les ignorent prendront du retard.
- La valeur réside dans la résolution de problèmes, non dans l'acte de coder lui-même ; les développeurs doivent maintenir des standards de qualité élevés en révisant le code généré et en fixant des directives de style pour les agents IA.
- Les LLM transforment les pratiques de développement : les tests deviennent plus accessibles, les révisions de code plus rigoureuses, les PR doivent être plus petites et mieux documentées, et les dépendances externes deviennent moins nécessaires.
- Des ressources comme les vidéos d'Andrej Karpathy et le livre de Sebastian Raschka aident à comprendre le fonctionnement des LLM, tandis que les outils matériels émergents (Tenstorrent, Cerebras) offrent des voies d'apprentissage intéressantes.
- La décision d'évoluer ou non appartient à chaque développeur et dépend de la conviction personnelle et de l'inspiration trouvée dans les réussites d'autres.

## Discussion sur Hacker News (84 commentaires)

**Avis positifs** :
- Les LLM permettent de créer des outils de support (tests, analyzeurs de logs, visualiseurs) qui auraient été trop coûteux en temps auparavant, améliorant significativement la qualité et la robustesse globale
- Les LLM excellent pour déboguer, explorer des bases de code complexes et triager les problèmes, libérant du temps pour se concentrer sur la logique métier et l'architecture
- Refuser d'adopter les LLM place objectivement les développeurs dans un déficit cognitif face à ceux qui les utilisent, notamment pour mémoriser les détails des tâches de programmation
- Les LLM accélèrent le processus de test et de documentation, rendant les changements architecturaux et les refactorisations largement plus gérables
- C'est un outil comme les autres (calculatrice vs règle à calcul, feuille de calcul vs crayon) : refuser son adoption est historiquement comparable à rejeter des avancées technologiques évidentes

**Avis négatifs** :
- La métrique de productivité basée sur le volume de code est trompeuse : les vrais goulots sont les réunions, la bureaucratie et la compréhension des problèmes, pas la vitesse d'écriture de code
- Comprendre et valider chaque ligne de code généré par IA introduit un goulot d'étranglement qui peut annuler ou réduire les gains de productivité supposés
- Les LLM produisent du code 'moyen' et reproduisent les défauts de leur ensemble d'entraînement ; à mesure que les développeurs perdent leurs compétences, la qualité du corpus s'effondre
- Le code généré par IA non relu/testé en profondeur peut contenir des bugs subtils ; les dépendances traditionnelles ont au moins bénéficié d'une révision antérieure
- Comme le transport motorisé, ce qui commence comme optionnel devient progressivement obligatoire pour rester employable, réduisant la liberté professionnelle et créant une dépendance envers les entités propriétaires qui contrôlent les meilleurs modèles
