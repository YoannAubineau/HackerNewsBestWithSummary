## Résumé de l'article

Ant est un runtime JavaScript construit de zéro, livré sous forme d'un binaire portable de 8,6 MB. Il exécute les packages npm réels, démarre instantanément sans étape de build, et propose un moteur maison appelé Ant Silver (pas un wrapper autour de V8, JSC ou SpiderMonkey).

- Démarrage ultra-rapide : 5,4 ms pour importer Hono et enregistrer deux routes, contre 31,1 ms pour Node
- Installation de packages jusqu'à 40 fois plus rapide que npm, sans configuration supplémentaire
- Support natif de TypeScript sans étape de build ni tsconfig
- Sandbox hardware-isolée pour exécuter du code non fiable : chaque sandbox est une VM KVM/Hypervisor.framework avec systèmes de fichiers en lecture seule et réseau dénégué par défaut
- Registry npm-compatible (ants.land) permettant de publier et consommer des packages, compatibles aussi avec npm, yarn, pnpm et bun

## Discussion sur Hacker News (95 commentaires)

**Avis positifs** :
- Le projet offre des avantages techniques réels : petite taille (~8mb avec runtime complet), démarrage rapide, sandboxing par hyperviseur et intégration desktop en développement (ant-desktop)
- L'initiative montre qu'un développeur individuel peut créer un runtime JavaScript viable, symbolisant l'évolution des outils de développement et l'accessibilité accrue à la création de projets ambitieux
- Le sandboxing par défaut répond à un besoin réel de sécurité face aux attaques de chaîne d'approvisionnement, domaine où Node et npm accusent un retard
- Les cas d'usage pour les applications desktop légères, l'embedding et les systèmes FaaS constituent une niche légitime distincte des runtimes existants

**Avis négatifs** :
- L'origine du code soulève des questions de confiance : le projet s'est initialement inspiré d'une base AGPL (Elk), avec un réécriture partiellement réécrite depuis février 2026, rendant l'affirmation d'un moteur « hand-built » techniquement trompeuse
- Le blog annoncé sur la création en un mois ressemble à du contenu généré par IA sans substantif clair du travail réel, alimentant le scepticisme sur la véracité des revendications
- Problèmes de gouvernance du projet : hébergement du code sur GitHub personnel malgré l'existence d'une entreprise (sf.tools), manque de feuille de route claire et bus factor critique avec un seul développeur actif
- Choix de nommage problématique : Apache Ant existe depuis longtemps, Ant Design domine le web JavaScript, créant une confusion et suggérant un manque de sérieux dans la planification du projet
- Les performances revendiquées (« near-V8 speeds ») contredisent les benchmarks publics zoo.js, bien que l'auteur prétende des améliorations récentes sans preuves indépendantes
