## Résumé de l'article

Grok Build, l'interface en ligne de commande de xAI pour la génération de code, transmet l'intégralité du contenu des dépôts — y compris les fichiers .env contenant des secrets — vers un bucket Google Cloud Storage nommé grok-code-session-traces, par deux canaux distincts : via les requêtes model-turn POST /v1/responses et via des archives session_state uploadées en POST /v1/storage.

- Les fichiers lus par l'agent (y compris les fichiers de secrets) sont transmis non redactés aux serveurs de xAI dans le corps des requêtes POST /v1/responses; une preuve directe capturée montre des clés d'API et mots de passe canary « CANARY7F3A9-SECRET-should-not-leave » et « CANARY7F3A9-DBPASS » dans les archives décompressées.
- L'intégralité du dépôt, indépendamment de ce que l'agent lit, est uploadée par un processus distinct; sur un dépôt de 12 Go jamais consulté par l'agent, 5,10 Go ont transité via /v1/storage (rapport ~27 800× par rapport aux 192 Ko du canal model-turn), prouvant qu'il s'agit d'une snapshot complète du workspace, pas seulement des fichiers ouverts.
- Les uploads sont effectués sous forme de bundles git vers un bucket Google Cloud Storage (grok-code-session-traces), nom capturé dans le binaire et les métadonnées; un fichier marqué explicitement pour ne pas être ouvert (src/_probe/never_read_canary.txt) a pu être récupéré verbatim d'un bundle uploadé capturé sur le réseau après un statut HTTP 200.
- Le mécanisme est actif par défaut sur les comptes consumer standard; le désactiver via le paramètre "Improve the model" n'arrête pas les uploads — l'endpoint /v1/settings continue de retourner trace_upload_enabled: true et upload_enabled: true même après désactivation.
- La destination de stockage (grok-code-session-traces) et le mécanisme d'upload ne figurent pas dans les matériaux d'installation et quickstart examinés; l'auteur précise que ce n'est pas un audit exhaustif de toute la documentation xAI, mais que la découverte n'est pas surfacée dans les guides de démarrage du CLI lui-même.

## Discussion sur Hacker News (122 commentaires)

**Avis positifs** :
- Uploader l'intégralité du codebase permet au modèle d'inspecter le contexte pendant le raisonnement sans appels client répétés, améliorant potentiellement les performances
- Tout outil cloud utilisé pour générer du code transmet nécessairement le code au serveur ; les utilisateurs devraient accepter ce compromis ou utiliser des modèles locaux
- Les fichiers .env ne devraient jamais contenir des secrets réels en production ; c'est une responsabilité du développeur de sécuriser son environnement local
- Les entreprises comme Google Cloud et Amazon ont établi des antécédents de confiance avec le cloud computing, suggérant que les préoccupations de vol de code peuvent être surestimées

**Avis négatifs** :
- Grok télécharge l'intégralité du dépôt, y compris les fichiers non utilisés pour la requête actuelle, ce qui est inédit et constitue une exfiltration de données potentiellement délibérée
- Le téléchargement est activé par défaut et reste actif même quand l'utilisateur désactive l'option « Améliorer le modèle », sans transparence dans les conditions de service
- Les fichiers sensibles (.env, clés SSH, secrets d'API) sont envoyés non filtrés et non masqués vers des serveurs GCP, et sont persistés dans des buckets nommés plutôt que traités temporairement
- La confiance envers Elon Musk et xAI est faible ; le modèle incitatif (vol de données d'entraînement sans divulgation) est aligné sur les comportements observés chez les sociétés d'IA concurrentes
- Même avec des configurations de sandbox, les outils propriétaires restent dangereux car les modifications futures ou les failles zéro-day pourraient contourner les protections locales
- Les modèles chinois (DeepSeek, Qwen) démontrent une approche opposée, post-traitant les données localement et ne transmettant que les chunks nécessaires, établissant un contraste frappant
