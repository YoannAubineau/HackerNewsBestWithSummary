## Résumé de l'article

SQLite est un moteur de base de données populaire, notamment utilisé pour le stockage local embarqué, mais ses paramètres par défaut posent des problèmes de sécurité et de performance. L'auteur propose d'introduire un système d'éditions inspiré de Rust pour permettre l'évolution des bonnes pratiques sans casser la rétrocompatibilité.

- Les contraintes de clé étrangère sont ignorées par défaut, ce qui peut créer des références invalides et des incohérences de données, aggravées par la réutilisation d'identifiants (ROWID) par SQLite
- Les tables non-strictes acceptent n'importe quel type de données dans n'importe quelle colonne, sans validation, ce qui peut introduire des bugs subtils (exemple : chaînes '1' et '0' stockées à la place de booléens)
- SQLite génère immédiatement une erreur SQLITE_BUSY lors de conflits d'écriture concurrente au lieu d'attendre jusqu'à un délai d'expiration, causant des crashes imprévisibles
- Le Write-Ahead Log (WAL) est désactivé par défaut, dégradant significativement les performances en écriture par rapport à sa configuration optimale
- La solution proposée : un pragma unique `PRAGMA edition = 2026` qui activerait automatiquement tous les bons paramètres (clés étrangères, timeout d'écriture, WAL, mode strict), tout en préservant la compatibilité rétroactive pour les applications existantes

## Discussion sur Hacker News (82 commentaires)

**Avis positifs** :
- L'idée capture les defaults universellement utilisés par les développeurs sérieux avec SQLite (foreign keys, strict tables, busy_timeout) et mérite d'être la nouvelle norme plutôt que maintenir des décisions de compatibilité vieilles de 26 ans.
- Le mécanisme d'éditions permettrait une évolution gracieuse des defaults sans casser la compatibilité rétroactive : les projets existants ignorent la nouvelle édition tandis que les nouveaux peuvent opter pour les meilleures pratiques modernes.
- Plusieurs bases de données et outils (Postfix, CMake, Perl) utilisent déjà avec succès des mécanismes similaires pour faire évoluer les defaults sans bifurquer l'écosystème.
- Les pragmas problématiques (comme SQLITE_BUSY) posent déjà des défis de compatibilité cross-platform et multi-version ; les éditions structureraient simplement ce qui existe déjà de manière ad hoc.
- Cela encourage les projets à adopter les meilleures pratiques modernes sans demander aux développeurs de lire la documentation complète et de configurer manuellement chaque pragma.

**Avis négatifs** :
- SQLite est un conteneur de données, pas un runtime : ajouter des éditions compliquerait l'interopérabilité puisqu'une base créée avec une nouvelle édition ne pourrait pas être lue par les anciennes versions, fragmentant l'écosystème et le stockage cross-plateforme.
- Les éditions aggravent le problème de compatibilité des pragmas au lieu de le résoudre : une édition 2030 impliquant PRAGMA foo introduit en 2027 bloquerait inutilement trois ans de versions intermédiaires.
- SQLite excelle dans sa niche (concurrent de fopen, pas de PostgreSQL) précisément parce qu'il est simple et prévisible ; demander à changer les defaults revient à vouloir en faire quelque chose qu'il n'est pas, ce qui risque de le dénaturer.
- Le type system permissif et les comportements par défaut de SQLite ne sont pas des bugs mais des choix de design intentionnels ; si les defaults ne conviennent pas, utiliser une base de données plus appropriée est la bonne réponse.
- Les fichiers de configuration (comme sqlite.conf au même niveau que la base) offriraient une flexibilité similaire sans fragmenter l'écosystème et en restant plus transparent pour les utilisateurs.
