## Résumé de l'article

Git 2.54 et 2.55 ont introduit la commande expérimentale `git history`, qui offre trois sous-commandes pour manipuler l'historique de commits sans les complications habituelles des rebase interactifs. Cette commande adresse plusieurs problèmes que les utilisateurs cherchent à résoudre avec des outils alternatifs comme jj, tout en restant dans l'écosystème Git standard.

- **fixup** : plie les modifications en attente dans un ancien commit et met automatiquement à jour toutes les branches qui en dépendaient, contrairement à `rebase --update-refs` qui ne redéplace que les références dans la plage en cours de rebase
- **reword** : modifie le message d'un ancien commit et reconstruit la pile sans toucher à l'index ou à l'arborescence de travail, permettant la réécriture d'un commit sur une branche non active
- **split** : divise un commit en deux en sélectionnant les hunks interactivement, sans avoir besoin de gymnastics avec rebase
- **Atomicité garantie** : aucune des trois commandes ne laisse l'arborescence dans un état partiellement cassé ; elles refusent les opérations susceptibles de créer des conflits
- **Limitations actuelles** : contrairement à jj, `git history` ne gère pas les états conflictuels comme premier ordre de citoyenneté, mais la documentation laisse la porte ouverte à des améliorations futures

## Discussion sur Hacker News (168 commentaires)

**Avis positifs** :
- Les nouvelles commandes `git history` simplifient des cas d'usage fréquents de `git rebase -i` (fixup, reword, split) en réduisant la friction et la complexité pour les utilisateurs
- Ces commandes sont particulièrement utiles pour aider les juniors à découper de grandes PR en commits plus petits et cohérents, améliorant la qualité des révisions de code
- Un historique bien curé est invaluable pour le bisect, le blame, la compréhension des changements et le déboggage : chaque commit atomique réduit drastiquement le temps de diagnostic des régressions
- L'interface de `git history` est plus ergonomique et moins intimidante que l'édition manuelle d'un fichier interactif, particulièrement pour les utilisateurs moins expérimentés
- Automatiser la mise à jour de toutes les branches descendantes lors d'un fixup (contrairement à `--update-refs`) élimine une source majeure d'erreurs et de frustration

**Avis négatifs** :
- Ces commandes ne fonctionnent que sans conflits ; le workflow interactif classique reste nécessaire dans les cas complexes, limitant leur utilité pratique
- Les commandes `git history` ne préservent pas les signatures GPG des commits modifiés, ce qui pose problème pour les workflows d'intégrité et de conformité
- Beaucoup de développeurs ne relisent jamais les commits individuels ; squasher tout avant fusion est une pratique valide qui rend ces commandes inutiles pour les équipes utilisant cette stratégie
- Automatiser la rebase de toutes les branches descendantes peut être dangereux : modifier des branches qu'on n'a pas explicitement sélectionnées risque de briser le travail en cours ou de créer des états difficiles à comprendre
- Le coût cognitif et la friction de Git proviennent davantage d'une mauvaise compréhension conceptuelle des modèles de données que de l'interface CLI ; des commandes plus simples ne résolvent pas ce problème fondamental
