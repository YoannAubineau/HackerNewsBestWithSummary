## Résumé de l'article

Cet article explique comment l'utilisation d'IA génératives pour écrire du code peut conduire à des mauvaises pratiques de programmation. Lorsque les développeurs laissent des raccourcis ou du code non optimisé dans leur base de code, l'IA les utilise comme modèles et les reproduit systématiquement, ce qui crée un cycle où les anti-patterns deviennent progressivement la « norme » du projet.

- Les LLM apprennent à partir du code existant dans votre référentiel et reproduisent les patterns qu'ils y trouvent, y compris les mauvaises pratiques
- Laisser passer des duplications de code (comme des conditionnelles copiées-collées) signale à l'IA que c'est le style du projet et elle perpétuera ce comportement
- Les code smells s'accumulent couche par couche, rendant plus difficile la correction ultérieure sans intervention manuelle
- Ce qui semblait être une externalisation de la maintenance devient en réalité un entraînement de l'IA à développer des habitudes de plus en plus mauvaises
- Écrire du code maintenable dès le départ est essentiel car les LLM absorbent et répètent tout ce qu'on leur montre

## Discussion sur Hacker News (180 commentaires)

**Avis positifs** :
- Le code clair et bien nommé réduit le besoin de commentaires et crée de meilleures abstractions, bénéficiant aussi bien aux humains qu'aux LLM qui héritent du projet
- Les bonnes pratiques de code (SOLID, séparation des préoccupations, DRY) ne sont pas obsolètes avec les LLM ; elles deviennent encore plus critiques car elles définissent le contexte qu'utilisera le modèle pour les futures modifications
- Les organisations ayant établi des standards de code, des normes de nommage métier et des dictionnaires de données réussissent mieux avec les LLM que celles laissant les agents générer du code sans contraintes
- Des outils comme CLAUDE.md, des linters stricts, des hooks pre-commit et des workflows de révision multi-passes permettent d'obtenir du code LLM de qualité maintainable comparable à du code humain
- Les commentaires inutiles générés par les LLM polluent le code bien plus que les commentaires bien ciblés expliquant le 'pourquoi' ; supprimer le bruit généralisé améliore la qualité globale

**Avis négatifs** :
- À court terme (quelques mois), les LLM produisent du code fonctionnel rapidement, mais la qualité se dégrade exponentiellement avec la taille du codebase, forçant à refactoriser régulièrement pour que les modèles restent efficaces
- Les LLM dupliquent systématiquement le code sans créer d'abstractions, même quand des patterns clairs existent ; obtenir des abstractions réutilisables nécessite une guidance intensive préalable souvent ignorée dans la pratique
- Demander aux LLM de respecter des directives (même longues listes de 200+ points) est moins fiable que des outils automatisés ; les modèles ignorent régulièrement les instructions textuelles pour suivre des patterns dans le code existant
- La plupart des développeurs n'ayant pas l'expertise pour bien utiliser les LLM manquent de discipline : le code dérive rapidement en spaghetti, accumule de la dette technique à vitesse 10x, et devient impossible à maintenir même pour les LLM après 6-12 mois
- Si tout le monde utilise les mêmes LLM publics avec les mêmes pratiques, personne n'a d'avantage compétitif ; la vraie valeur réside dans la conception et l'architecture, pas dans la génération de code
