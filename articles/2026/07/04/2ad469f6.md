## Résumé de l'article

Ce guide détaille comment construire et configurer un système informatique local pour exécuter des modèles de langage de pointe (LLM) sans dépendre de services cloud. L'auteur décrit deux configurations : une avec 2x RTX 3090 (48 GB VRAM, ~$2k) capable de faire tourner Qwen3.6-27B et Whisper pour la reconnaissance vocale, et une plus puissante avec 4x RTX 6000 Pro (384 GB VRAM, ~$40k) offrant des performances proches de Claude Opus.

- Les configurations reposent sur des GPU Nvidia recyclés (eBay), un système EPYC dernière génération, et des commutateurs PCIe4 permettant la communication directe entre cartes GPU pour réduire la latence sans coûter le prix des systèmes PCIe5/DDR5
- Les modèles et poids sont stockés localement sur un système de fichiers ZFS repliqué, puis exécutés dans des conteneurs Docker individuels accessibles via une API HTTP
- L'outillage des modèles inclut des capacités de navigation web (searXNG), communication (Telegram), gestion de code (Gitea privée) et peut fonctionner interactivement ou automatiser des tâches via des pull requests
- La configuration système critique implique de désactiver l'IOMMU et l'ACS PCIe au démarrage, d'ajuster les gains du commutateur selon la longueur des câbles SAS, et de vérifier que les vitesses PCIe ne sont pas dégradées
- La consommation électrique est maîtrisée via des limites de puissance (350W par GPU) pour fonctionner sur un circuit 110V standard, avec des vitesses de P2P mesurées à 50,4 GB/s en bidirectionnel et latence ultra-faible (0,37–0,45 µs)

## Discussion sur Hacker News (125 commentaires)

**Avis positifs** :
- Les modèles locaux offrent des avantages réels : confidentialité des données, pas de risque d'indisponibilité du service, pas de crainte de dépassement de tokens, et utilité pratique reconnue pour de nombreuses tâches courantes.
- Pour les budgets modérés (2-5k$), des solutions accessibles existent : single RTX 3090, Macs M-series récents, ou machines OEM comme les Spark offrent un bon rapport qualité-prix pour exécuter Qwen ou Gemma décemment.
- L'écosystème s'améliore : nouveaux samplers (DRY, XTC, top_n_sigma) réduisent significativement les problèmes de boucles infinies; des alternatives à Whisper (Parakeet) sont plus petites et performantes; MLX sur Apple est très prometteur.
- Les modèles petits et moyens (Qwen 3.6-27B, Gemma 4-31B) suffisent pour beaucoup d'usages : résumés, recherche web via MCP, scripts Python simples, vision, categorization - pas besoin de SOTA pour tous les cas.
- À long terme, l'infrastructure local pourrait être rentable pour les besoins stricts de confidentialité ou comme protection contre les augmentations de prix des fournisseurs.

**Avis négatifs** :
- Les coûts réels dépassent largement les estimations initiales : un setup $40k annoncé coûte plutôt 50-55k$, et les cartes PRO 6000 ne coûtent plus $8.5k mais $13k, rendant l'économie très désavantageuse face aux API cloud.
- Les modèles quantizés et modifiés (REAP, 4-bit) dégradent notablement la qualité : les benchmarks ne reflètent pas la performance réelle sur les tâches long-contexte ou complexes; l'écart se creuse avec les erreurs qui s'accumulent.
- Comparé aux API : même à $200/mo, Claude Opus via API revient à ~$4000/mo en pricing enterprise, soit équivalent à 10 mois seulement du setup local; les tokens cloud coûtent moins cher si on les utilise vraiment, sans ROI prévisible.
- L'infrastructure locale est encore aux balbutiements : pas de solution turnkey, maintenance complexe requise, refroidissement problématique sur laptops, et configuration logicielle délicate (drivers, containers, etc).
- Les alternatives cloud (renting GPUs à $32/hr, providers multiples via OpenRouter) offrent plus de flexibilité, moins de dépréciation matérielle, et évitent les maux de tête d'exploitation pour la plupart des cas d'usage.
