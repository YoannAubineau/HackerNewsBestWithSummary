## Résumé de l'article

Les tables strictes sont une fonctionnalité de SQLite (disponible depuis la version 3.37.0) qui applique un typage rigide en interdisant l'insertion de données du mauvais type dans une colonne, contrairement au comportement par défaut de SQLite qui accepte les conversions implicites.

- **Syntaxe simple** : ajouter le mot-clé STRICT à la fin de la définition de table (ex. `CREATE TABLE people (name TEXT) STRICT;`)
- **Prévient les erreurs de type** : empêche d'insérer du texte dans une colonne INTEGER ou d'autres combinaisons invalides, tout en acceptant les conversions sans perte de données (ex. la chaîne '123' convertie en entier)
- **Valide les types de colonnes** : rejette les types de colonnes invalides ou mal orthographiés (DATETIME, UUID, JSON, BLOBB, etc.), n'acceptant que INT, INTEGER, REAL, TEXT, BLOB et ANY
- **Flexibilité conservée** : le type ANY permet d'accepter n'importe quel type de données si nécessaire
- **Limitations** : impossible de modifier une table existante pour la rendre stricte (nécessite une migration manuelle), incompatibilité avec les versions SQLite antérieures à 3.37.0, impact potentiel mineur sur les performances bien que non démontré en pratique

## Discussion sur Hacker News (57 commentaires)

**Avis positifs** :
- Les tables strictes devraient être le comportement par défaut pour éviter les bugs involontaires (ex: stockage accidentel de chaînes dans des colonnes numériques affectant des milliers d'appareils)
- Avec STRICT, la schema communique clairement l'intention du développeur, ce qui est crucial pour la maintenabilité et l'héritage de projets
- Le type ANY permet toujours la flexibilité quand nécessaire, tout en signalant explicitement les intentions versus laisser le doute avec les paramètres actuels
- Les optimisations de performance pour la sérialisation/désérialisation deviendraient possibles avec le typage strict par défaut
- Pour les bases de données partagées entre applications, la fiabilité des types déclarés est essentielle pour éviter que une application casse les autres

**Avis négatifs** :
- Le manque de typage strict est une décision historique justifiée : SQLite était un ersatz de fopen et a évolué d'une architecture dbm où tout était stocké en chaînes
- Changer le défaut casserait la rétrocompatibilité massive nécessaire pour des millions d'applications existantes utilisant SQLite sans révision majeure
- SQLite concurrence fopen, pas Oracle/Postgres : pour une base de données intégrée mono-application, vérifier les types à runtime est une surcharge inutile si la validation statique du code suffit
- STRICT présente des limitations : absence de types DATE/BOOL, pas de NUMERIC en strict mode, nécessitant des contournements complexes avec CHECK constraints
- La flexibilité du typage dynamique aide des scripts aléatoires à fonctionner avec SQLite sans adaptation, et permet l'évolution progressive du schéma sans migration
