## Résumé de l'article

Scarf, entreprise qui exploite un service de distribution de paquets open source, a progressivement migré son backend d'Haskell vers Python après sept ans de production. Bien que Haskell ait tenu ses promesses en fiabilité et sécurité des types, l'émergence des modèles de langage a modifié l'équation économique du développement : les longs temps de compilation deviennent un goulot d'étranglement quand les agents IA génèrent du code en minutes.

- Les temps de compilation froids d'Haskell et la complexité des environnements de développement deviennent prohibitifs pour les workflows parallèles assistés par IA, nécessitant des contextes d'exécution bon marché et jetables
- Scarf a déployé un serveur API Python aux côtés de son infrastructure Haskell existante, migrant progressivement les nouvelles routes API sans risque de basculement brutal
- Les gains de productivité en vitesse d'itération, couverture de test automatisée et déploiement de correctifs compensent largement les pertes de sécurité de types, selon l'expérience de l'équipe
- L'auteur avertit que l'écosystème Haskell est en danger s'il n'optimise pas pour les workflows d'IA : temps de construction rapides, documentation riche d'exemples, amorçage facile et messages d'erreur agents-friendly sont désormais critiques
- La communauté Haskell doit réorienter ses priorités vers les préoccupations industrielles (compilation, intégration IA) plutôt que vers les recherches théoriques comme les types dépendants

## Discussion sur Hacker News (229 commentaires)

**Avis positifs** :
- Les temps de compilation longs de Haskell créent un goulot d'étranglement réel pour les cycles de feedback itératifs avec les LLM, particulièrement en mode multi-agents avec worktrees séparés sans cache partagé.
- Python offre une vitesse d'itération brute supérieure (pas de compilation) qui peut compenser partiellement l'absence de vérification de type si la couverture de tests est robuste.
- Les LLM génèrent du code suffisamment bon que les garanties de type statique apportent des gains marginaux comparés au coût infrastructurel, surtout pour les CRUD backends simples.
- D'autres langages avec compilation rapide (OCaml, Go, TypeScript) combinent des feedback loops rapides avec une sécurité de type supérieure à Python.
- La documentation et l'écosystème Haskell pour les exemples copy-pastables et adaptés aux agents représentent un vrai déficit comparé à des langages plus mainstream.

**Avis négatifs** :
- Le choix de Python (au lieu de Go, Rust, TypeScript ou OCaml) sacrifie la sécurité de type sans justification suffisante—Python a les mêmes problèmes de dépendances et de version que l'article reprochait à Haskell.
- Les LLM génèrent des code quality variables et peuvent introduire des bugs subtils (logique métier, mutations distantes) qu'un typage fort aurait détectés ; la confiance envers les LLMs est surévaluée.
- Les systèmes de type expressifs restent supérieurs pour contraindre les générations d'agents et accélérer la convergence, comme en témoignent les expériences réussies avec Lean, OCaml et Rust.
- Les problèmes de compilation Haskell sont gérables via caching multi-worktrees, incremental compilation et ghci—c'est un choix d'infra, pas une limitation intrinsèque du langage.
- Le passage à Python masque un capitulation idéologique vers la 'vibe coding' (génération sans relecture) plutôt qu'une analyse technique rigoureuse ; l'article reconnaît lui-même que les gains de type safety ne se sont 'pas encore remarqués'.
