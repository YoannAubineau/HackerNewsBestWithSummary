## Résumé de l'article

Une étude de recherche examine si la qualité structurelle et stylistique du code (« propreté ») influence la capacité des agents IA autonomes à le naviguer et le modifier. Les chercheurs ont créé des paires de dépôts identiques en architecture et dépendances mais différents en violations de règles statiques et complexité cognitive, puis testé Claude Code sur 660 essais à travers 33 tâches.

- Le taux de réussite des tâches reste inchangé quel que soit le niveau de propreté du code
- Les agents travaillant sur du code propre consomment 7 à 8% moins de tokens (unités de traitement)
- Les agents révisitent 34% moins de fichiers sur du code de meilleure qualité
- Les résultats confirment que les principes traditionnels de maintenabilité du code conservent leur pertinence dans le développement piloté par l'IA
- La propreté du code s'ajoute à d'autres facteurs (choix du modèle, configuration, prompts) qui influencent le comportement des agents IA

## Discussion sur Hacker News (86 commentaires)

**Avis positifs** :
- Le code propre réduit significativement les coûts opérationnels des agents : moins de tokens consommés (7-8%), moins de revisites de fichiers (34%), même si le taux de succès reste identique.
- Un code bien organisé améliore la performance des agents en réduisant le travail de traversal et de recherche, permettant aux modèles de trouver plus rapidement le code pertinent sans multiplier les tentatives.
- Les principes de maintenabilité traditionnels restent hautement pertinents pour les agents, tout comme pour les humains : bonne structure, nommage clair, abstraction appropriée et tests facilitent le travail des agents.
- En pratique, les équipes observent que traiter les agents comme des développeurs humains (demander du code propre, bien structuré, avec bonne documentation) produit des résultats significativement meilleurs.
- Le code propre aide les agents à imiter les bonnes pratiques plutôt que les mauvaises habitudes, créant un effet de feedback positif où la qualité initiale influence positivement les générations ultérieures.

**Avis négatifs** :
- L'étude repose sur une méthodologie discutable : utiliser des agents pour synthétiquement créer des codebases 'dégradés' ou 'nettoyés' n'est pas représentatif des vrais scénarios où le code se dégrade organiquement par des PRs successives et imparfaites.
- Les résultats sont peu fiables car l'étude ne contrôle pas si les agents cassent les tests existants non liés à la tâche, rendant les conclusions sur la consommation de tokens pratiquement insignifiantes sans garantie de qualité équivalente.
- Les outils statiques (linters, analyseurs) résolvent déjà les problèmes triviauxmentionnés (code mort, duplication). Les vrais défis restent les cas complexes : code hérité entremelé, patterns obsolètes intertwined, que les linters ne peuvent pas traiter.
- Les agents sont réticents à supprimer du code et tendent à créer des abstractions excessives ou du code mort, ce qui crée une accumulation de résidus même après des sessions de nettoyage, nécessitant des boucles répétées sans garantie de satisfaction.
- Le problème est que les agents performent bien sur du code généré par d'autres agents ou bien structuré, mais ce qui compte vraiment c'est leur performance sur des codebases réelles mixtes avec code héritage et patterns multiples, où ils se perdent et copient les mauvaises habitudes selon le contexte saisi.
