## Résumé de l'article

Dependabot, l'outil GitHub de gestion automatique des dépendances, impose désormais un délai d'attente de trois jours minimum après la publication d'une nouvelle version avant d'ouvrir une demande de mise à jour. Cette mesure de sécurité est activée par défaut sans configuration requise.

- Le délai n'affecte que les mises à jour de versions ; les correctifs de sécurité restent ouverts immédiatement
- Cette attente réduit le risque de fusionner une version compromise ou défectueuse avant que la communauté n'ait le temps de la détecter
- Les utilisateurs peuvent personnaliser ce délai ou le désactiver via l'option `cooldown` dans `.github/dependabot.yml`
- Le paramètre s'applique à tous les écosystèmes de paquets supportés sur github.com et arrivera dans GitHub Enterprise Server 3.23

## Discussion sur Hacker News (134 commentaires)

**Avis positifs** :
- Les délais de cooldown donnent aux entreprises de cybersécurité le temps d'analyser automatiquement les nouveaux packages avant leur installation massive par les utilisateurs, détectant malveillances et comportements suspects.
- Cela rend plus difficile les attaques éclair ciblant les mainteneurs : même compromise, le compte ne peut plus déployer instantanément du code malveillant à grande échelle.
- Les délais permettent aux mainteneurs de détecter les modifications suspectes (pas de tag git associé, versions rapides successives) et de réagir avant la propagation à l'échelle.
- Comparé à l'absence totale de mesure, augmenter la barre pour les attaquants basiques (scripts d'attaque génériques) offre un bénéfice de sécurité réel, même face à des techniques d'obfuscation.
- C'est une solution pragmatique plus facile à implémenter qu'un système de vérification pré-publication standardisé à travers tous les écosystèmes et répertoires.

**Avis négatifs** :
- Les délais peuvent être contournés par des codes malveillants sophistiqués utilisant des bombes à retardement (activation après le cooldown), rendant la mesure inefficace contre les attaquants de haut niveau.
- Les utilisateurs sont confus : rien ne garantit que les packages ont été audités durant le cooldown, c'est seulement un vœu pieux sans mécanisme de vérification transparent.
- Chaque mise à jour est elle-même une source potentielle de vulnérabilités et de régressions ; forcer les mises à jour fréquentes augmente le risque global au lieu de le réduire.
- Le système crée un faux sentiment de sécurité alors que la vraie solution serait de minimiser les dépendances, d'utiliser des versions fixes et d'auditer réellement les packages plutôt que d'espérer une détection automatique.
- Les délais ne résoudront rien si chaque attaquant ajoute simplement du code obfuscé attendant, reportant le problème sans le résoudre : c'est une course aux armements sans fin.
