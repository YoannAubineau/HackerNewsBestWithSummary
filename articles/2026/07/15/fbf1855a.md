## Résumé de l'article

HTMX est une bibliothèque JavaScript qui permet d'ajouter de l'interactivité aux applications web en minimisant le code JavaScript côté client, tout en conservant le rendu HTML côté serveur. Cet article présente les patterns et bonnes pratiques pour utiliser HTMX avec Go, notamment la gestion des templates, des redirections, des erreurs et des configurations spécifiques.

- Organiser les templates HTML en trois niveaux : un template de base (layout commun), des pages spécifiques, et des fragments réutilisables (partials) qu'on intègre via un rendu templating centralisé
- Créer un type `htmlRenderer` qui charge les templates partagés au démarrage, puis clone et étend cet ensemble pour chaque réponse, permettant de renvoyer soit des pages complètes soit des fragments HTML
- Vérifier le header `HX-Request: true` pour déterminer si la requête provient de HTMX et adapter la réponse en conséquence (partiel ou page complète)
- Gérer les redirections avec le header `HX-Redirect` pour les requêtes HTMX (au lieu de codes 3xx) et revenir à des codes 3xx standard pour les navigateurs sans JavaScript
- Configurer HTMX via des balises meta dans le template de base pour désactiver le cache d'historique, l'héritage d'attributs, les styles d'indicateur, et gérer les codes d'erreur (notamment 422 pour les formulaires avec validation)

## Discussion sur Hacker News (66 commentaires)

**Avis positifs** :
- HTMX permet de revenir à une approche web classique avec rendu serveur et requêtes Ajax, réduisant drastiquement le JavaScript nécessaire et évitant la complexité des frameworks modernes comme React
- L'écosystème Go + HTMX (avec des outils comme Templ, sqlc, SQLite) crée une stack productive et déployable en un seul binaire, apprécié des développeurs et des assistants IA pour sa cohérence et ses cycles d'itération rapides
- HTMX force une philosophie de conception plus native du web et hypermédia, facilitant les tests, le raisonnement des agents IA, et gardant computation et données au même endroit
- Comparé aux frameworks JavaScript complexes, HTMX économise du temps initial et permet de construire rapidement des dashboards et des applications CRUD sans dépendances npm massives
- La combinaison avec SvelteKit ou LiveView offre une meilleure expérience pour les interfaces complexes, tout en conservant une vision plus proche du web que React

**Avis négatifs** :
- Au-delà des applications CRUD basiques, HTMX peine avec les interfaces complexes nécessitant état partagé, composants interconnectés et interactions élaborées (grilles de données, filtres avancés, virtualization)
- Go html/template a une interface maladroite et unintuitive (notamment le clonage), et malgré des alternatives comme Templ, l'écosystème manque de bibliothèques de composants UI prêts à l'emploi comparables à Mantine ou Material-UI
- La complexité du code croît disproportionnément (ratio 2:1) avec HTMX, induisant des hacks bizarres et des cas limites difficiles à résoudre sans être une surréaction aux problèmes de Node
- L'adoption en équipe est difficile : la technologie manque de légitimité auprès des développeurs habitués à React/Vue, générant inertie, méfiance et attribution hâtive des bugs à HTMX plutôt qu'au code
- HTMX fonctionne mieux pour des applications à interactivité faible ; pour les besoins complexes, LiveView/Elixir ou Vue/React offrent une meilleure séparation et une réactivité plus simple et prévisible
