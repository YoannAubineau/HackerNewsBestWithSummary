## Résumé de l'article

Cursor est un environnement de développement assisté par IA comptant plus de 7 millions d'utilisateurs actifs. Une vulnérabilité critique découverte en décembre 2025 permet à un attaquant de placer un fichier git.exe malveillant à la racine d'un projet : Cursor l'exécute automatiquement avec les privilèges de l'utilisateur, sans avertissement ni confirmation, donnant ainsi accès arbitraire au code.

- La vulnérabilité est extrêmement simple : Cursor recherche les binaires Git dans le répertoire du projet et exécute tout git.exe trouvé lors du chargement du projet et de manière répétée pendant son utilisation
- Aucune interaction utilisateur n'est requise (pas de clic, dialogue, avertissement) ; l'exploitation ne dépend d'aucune chaîne complexe ou technique avancée
- Mindgard a signalé le problème à Cursor en décembre 2025 et a relancé plusieurs fois en sept mois ; plus de 197 versions ultérieures ont été livrées sans correction
- Cursor n'a fourni ni communication, ni mise à jour de statut, ni indication de remédiation malgré les multiples tentatives de contact par les chercheurs, y compris via le programme bug bounty HackerOne
- Les utilisateurs d'entreprise doivent appliquer des politiques AppLocker ou Windows App Control ; les utilisateurs individuels doivent ouvrir les référentiels non approuvés uniquement dans une machine virtuelle isolée ou un bac à sable

## Discussion sur Hacker News (140 commentaires)

**Avis positifs** :
- L'absence de réponse de Cursor pendant 6+ mois après confirmation de la vulnérabilité par HackerOne démontre une négligence grave en matière de sécurité et justifie la divulgation publique comme dernier recours
- C'est un vrai problème de conception : Cursor exécute des binaires du répertoire courant sans interaction utilisateur ni sandbox, ce qui viole l'attente raisonnable que l'ouverture d'un IDE ne devrait pas exécuter du code arbitraire
- La vulnérabilité présente un risque réel de chaîne d'approvisionnement : les contributeurs qui clonent des dépôts compromis (branches malveillantes, commits injectés) sont exploités sans le savoir lors de l'ouverture simplement du projet
- Contrairement aux systèmes d'exploitation qui posent ./ en PATH depuis des décennies, Cursor peut facilement utiliser des chemins absolus vers git système au lieu de chercher dans le répertoire courant, comme le font d'autres outils sérieux
- Le problème persiste à travers 197+ versions malgré une exploitation connue depuis décembre 2025, indiquant soit une priorisation délibérée de la vélocité sur la sécurité, soit une indifférence organisationnelle

**Avis négatifs** :
- Pour exploiter cette faille, un binaire malveillant doit déjà être présent sur la machine (clone/téléchargement) : c'est similaire à exécuter une pièce jointe malveillante plutôt qu'une vulnérabilité classique
- Cela découle d'une particularité de Windows (inclusion du répertoire courant dans PATH) datant de décennies, ce qui est un problème systémique, pas uniquement un bug Cursor—npm install, Java build tools et PowerShell partagent le même comportement
- Cursor demande déjà si l'utilisateur fait confiance au projet lors de l'ouverture ; ignorer cet avertissement puis blâmer l'IDE pour exécution de code présent dans le répertoire est mal posé
- Le rapport mélange des problèmes LLM (triage d'une avalanche de faux rapports de sécurité générés par l'IA) avec la négligence de Cursor : les équipes sécurité sont submergées et ne peuvent distinguer les vrais rapports des hallucinations
- Si vous clonez une dépôt compromis et l'ouvrez dans n'importe quel IDE moderne, toute exécution de commandes ou installation (npm, maven, pip) risque la compromission—c'est un problème d'utilisateur téléchargeant du code non vérifié, non un défaut unique de Cursor
