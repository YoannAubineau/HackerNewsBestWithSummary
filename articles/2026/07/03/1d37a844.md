## Résumé de l'article

Selon cette perspective, la revue de code ne cherche pas à trouver des bugs, mais plutôt à identifier le code qui sera difficile à maintenir à long terme. Un reviewer doit évaluer s'il peut comprendre ce que fait le code et comment il fonctionne.

- La revue de code n'est pas un mécanisme fiable pour détecter les bugs
- Le véritable objectif est de repérer le code peu lisible ou peu clair
- Si le reviewer ne comprend pas le code, cela signale un problème de maintenabilité futur
- Les corrections doivent intervenir pendant que l'auteur original connaît bien le code
- L'examen du code seul ne peut garantir l'absence de bugs

## Discussion sur Hacker News (169 commentaires)

**Avis positifs** :
- La maintenabilité du code est effectivement un objectif central : si le code est inintelligible ou mal structuré, il crée des dettes techniques et des problèmes futurs (bifurcations de patterns, inconsistances, etc.)
- Les revues de code servent de filet de sécurité final avant merge, vérifiant que les décisions architecturales convenues sont respectées et que rien n'a été oublié selon le plan établi
- Comprendre le code en profondeur pour évaluer sa maintenabilité est le fondement qui permet ensuite de détecter les bugs, les problèmes de sécurité et les mauvaises pratiques

**Avis négatifs** :
- Les revues de code ont des objectifs multiples et non uniques : transfert de connaissances, détection de bugs réels, vérification de la sécurité, couverture de tests, et partage de responsabilité collective du code
- L'affirmation que 'il est impossible de trouver des bugs en examinant le code' est factuellement fausse : les reviewers trouvent régulièrement des bugs, des failles de sécurité et des cas limites oubliés par l'auteur
- Réduire la revue de code à un seul objectif primaire est une fausse dichotomie qui escamote les bénéfices concrets de la pratique, notamment la prévention des régressions et l'apprentissage collectif
- Avec l'émergence du code généré par IA, le modèle idéalisé de revue asynchrone ne fonctionne plus : les diff géants et le code difficilement compréhensible rendent la revue inefficace et non-scalable
- La pertinence de la maintenabilité comme objectif primaire dépend du contexte organisationnel : petites équipes vs grandes bases de code, bug fixes vs nouveaux développements, absence de processus de design formel
