## Résumé de l'article

Async Rust, le système pour exécuter du code concurrent en Rust de manière agnostique aux exécuteurs, produit actuellement beaucoup de bloat binaire notamment sur les microcontrôleurs. L'auteur analyse comment le compilateur transforme les fonctions async en machines d'état complexes et propose plusieurs optimisations du compilateur pour réduire ce surcoût.

- Le compilateur génère actuellement 360 lignes de MIR pour une simple fonction async imbriquée au lieu de 23 lignes pour l'équivalent non-async, avec des états inutiles (Returned, Panicked) qui causent des paniques à chaque appel après complétion
- Remplacer le panic du state Returned par Poll::Pending réduit la taille binaire de 2-5% sur l'embarqué ; éliminer la machine d'état pour les async blocks sans await sauve 0,2%
- Les futures simples avec un seul await pourraient être intégrées (inlining) dans l'appelant pour éviter une machine d'état supplémentaire, et les états identiques en match expressions pourraient être fusionnés
- LLVM ne peut pas compenser cette inefficacité même avec opt-level=3, notamment parce qu'il ne peut pas éliminer les branches de panic inutiles
- L'auteur a soumis un Project Goal et cherche du financement (~30k€) pour implémenter ces optimisations dans le compilateur Rust

## Discussion sur Hacker News (170 commentaires)

**Avis positifs** :
- Les optimisations manquantes du compilateur pour l'async Rust sont bien réelles et documentées, avec des gains mesurables de 2-5% en taille binaire et 3% de performance sur des codebases embarquées réelles
- L'approche explicite de Rust avec des runtimes pluggables offre une flexibilité supérieure à celle des langages avec runtime par défaut, permettant une meilleure adaptation aux contraintes embarquées versus serveurs web
- L'adoption d'async Rust en systèmes embarqués (ESP32, microcontrôleurs) est révolutionnaire comparée aux alternatives existantes, notamment avec Embassy qui unifie les interfaces entre domaines
- Le problème d'inlining des futures et de gestion d'état machine est fondamental et mérite attention : chaque indirection en async Rust a un coût runtime contrairement au code synchrone
- Les optimisations proposées (hoisting du match, élimination des états inutiles) sont applicables aujourd'hui sans travail compilateur et pourraient bénéficier à des codebases réelles

**Avis négatifs** :
- Le titre est exagérément dramatique et racoleur (clickbait) pour ce qui est essentiellement une discussion d'optimisations mineures, pas un retour au stade MVP
- Les problèmes soulevés concernent principalement des micro-optimisations sur des cas triviaux; les véritables async blocks utiles sont suffisamment complexes pour que le surcoût soit négligeable
- Rust pourrait plutôt nécessiter un runtime executor par défaut dans la stdlib (comme le suggèrent comparaisons avec Go/Python) plutôt que des optimisations locales à la marge
- L'ecosystem dépend massivement de Tokio créant une centralisation problématique, et les solutions sans-io ou executor-agnostic restent marginales et compliquées en pratique
- Le vrai problème reste le function coloring inévitable dans le design actuel; des solutions structurelles (effects algébriques, keyword generics) seraient plus pertinentes que des optimisations du compilateur
