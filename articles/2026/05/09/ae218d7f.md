## Résumé de l'article

Tweet de @jarredsumner :

> 99.8% of bun’s pre-existing test suite passes on Linux x64 glibc in the rust rewrite

## Discussion sur Hacker News (647 commentaires)

**Avis positifs** :
- Démontre que les LLM excèlent pour la traduction de code entre langages, particulièrement avec une suite de tests exhaustive servant de spécification exécutable.
- Rust offre des garanties de sécurité mémoire (borrow checker, destructeurs) que Zig ne fournit pas, adressant directement les fuites mémoire et crashes que Bun a rencontrés.
- Accomplissement technique remarquable : 960 000 lignes portées en 6 jours avec 99,8 % de compatibilité teste, ce qui aurait demandé plusieurs mois à des équipes humaines.
- Le système de type strict et l'application des contraintes au compilateur de Rust conviennent particulièrement bien au code généré par IA, réduisant les erreurs subtiles.
- Les décisions de conception dans un projet réécrit peuvent s'améliorer avec le temps; une base fonctionnelle permet l'itération progressive plutôt que le perfectionnisme initial.

**Avis négatifs** :
- Le code généré par IA (960 000 lignes) ne peut pas être pleinement compris par le développeur principal en 6 jours; risque majeur de dette cognitive et d'absence de propriété intellectuelle réelle.
- Suite de tests insuffisante pour garantir la correction : les tests valident le comportement observable, pas l'architecture, l'efficacité, ou les invariants non testés; les bugs peuvent rester cachés.
- Acquisition par Anthropic et exécution rapide suggèrent une manœuvre marketing pour promouvoir Claude plutôt qu'une décision technique rationnelle; maintien futur du projet en doute.
- Bun a antérieurement critiqué la fusion de ses patchs Zig pour des raisons techniques légitimes; la réaction personnelle du mainteneur (« fatigué des fuites mémoire ») masque possiblement des problèmes de discipline dans l'architecture originale en Zig.
- Maintenabilité à long terme compromise : aucun développeur ne comprend réellement ce million de lignes, créant un risque existentiel pour un composant critique de la stack (runtime JavaScript).
