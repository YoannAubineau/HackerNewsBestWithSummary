## Résumé de l'article

Un développeur web a décidé d'interdire tous les paramètres de requête (query strings) non autorisés sur son site personnel, une mesure destinée à empêcher l'ajout de code de suivi par des tiers aux URL de ses pages.

- Le développeur rejette l'ajout de paramètres de suivi comme les références (ref=) et les UTM parameters (utm_source, etc.) que d'autres sites ajoutent à ses URL
- Il considère que ces pratiques abusent les utilisateurs et que seuls les paramètres qu'il contrôle doivent exister sur son site
- Actuellement son site n'utilise aucun paramètre de requête, mais il prévoit de maintenir une liste blanche des paramètres autorisés s'il en utilise à l'avenir
- Cette restriction a été implémentée dans sa configuration Caddyfile (serveur web)
- Il souligne que chacun est libre de faire ce qu'il souhaite avec son propre site web

## Discussion sur Hacker News (120 commentaires)

**Avis positifs** :
- Rejeter les query strings non autorisés est justifié : c'est une forme de pollution de l'URL qui viole le consentement des utilisateurs, complique le partage de liens, et contourne délibérément les mécanismes de confidentialité existants (headers Referer).
- La pratique d'ajouter des trackers à des URLs tierces crée des problèmes réels : fragilité des URLs, bris de liens imprévisibles, mauvaise expérience utilisateur lors du partage, et confusion lors des tests (exemple de The Browser qui ne testait pas les URLs avec paramètres).
- Utiliser un code d'erreur HTTP pour rejeter les query strings est une riposte créative et défendable : 414 est technically correct selon RFC 9110 (l'ajout de caractères augmente la longueur), c'est humoristique, et ça force une conversation sur l'étiquette du web.
- La robustesse excessive (Postel's law) est devenue contre-productive : elle facilite les bugs de sécurité, les incompatibilités inter-systèmes, et encourage les comportements prédateurs. Des règles strictes encouragent une meilleure conception.

**Avis négatifs** :
- Query strings sont essentiels pour de nombreux usages légitimes : recherches, filtrage, pagination dans les applications monopages où l'utilisateur doit pouvoir partager son état exact. Interdire ALL query strings casse ces fonctionnalités.
- Le site rejeté (qui utilise les query strings pour les polices) n'abusait pas du protocole : c'est un usage standard et conforme. Blâmer la victime plutôt que d'inviter les sites à mieux gérer les URLs inattendues reflète une mauvaise compréhension du problème.
- Beaucoup de sites ajoutent des query strings sans problème : la pratique fonctionne 'assez bien' pour que des entreprises établies (The Browser, YouTube, Amazon) la maintiennent malgré ses défauts, ce qui contredit le catastrophisme de l'argument.
- La solution proposée pénalise les utilisateurs finaux, pas les vrais contrevenants : celui qui reçoit un lien cassé ne contrôle pas les query strings ajoutées. Mieux vaudrait informer les utilisateurs sur comment nettoyer les URLs ou utiliser des outils (ClearURLs) plutôt que de casser leurs navigation.
