## Résumé de l'article

Un incident a affecté les performances de plusieurs services incluant les opérations Git, les demandes d'API, ainsi que la gestion des issues et pull requests sur une plateforme de contrôle de version. L'incident a été résolu.

- L'incident a causé une dégradation des performances pour les opérations Git, les requêtes API, les issues et les pull requests
- Le service est maintenant rétabli et l'incident a été déclaré résolu
- Une analyse détaillée des causes racines sera partagée ultérieurement

## Discussion sur Hacker News (192 commentaires)

**Avis positifs** :
- Les incidents répétés de GitHub confirment un problème structurel réel : croissance exponentielle (14x en commits) couplée à une architecture héritée non adaptée à l'échelle actuelle, particulièrement pour les charges d'écriture distribuées
- L'utilisation massive d'agents IA a créé une charge imprévisible et inédite que GitHub n'a pas su anticiper malgré les annonces de préparation pour 'un milliard de développeurs'
- Les incidents affectent maintenant les fonctionnalités critiques (PRs, opérations Git) de façon répétée, rendant le service unreliable pour les workflows professionnels
- Des alternatives viables existent et fonctionnent bien : Forgejo auto-hébergé, Codeberg, Gitea offrent une expérience plus stable et plus rapide avec une UX plus directe

**Avis négatifs** :
- Attribuer tous les problèmes à Microsoft et à l'IA est une simplification excessive ; GitHub a longtemps fonctionné bien sous Microsoft (8 ans) avant les récents problèmes, et les changements de classification des incidents peuvent expliquer l'apparence soudaine de dégradation en avril 2026
- Les alternatives self-hostées (Forgejo, Gitea) ont elles-mêmes des UX confuses et peu polies, avec des limites pour les petits utilisateurs qui n'ont pas les ressources pour auto-héberger, contrairement à ce que suggèrent les critiques
- GitLab, bien que technique, souffre de problèmes similaires de complexité UI/UX et n'a pas émergé comme concurrent viable, contredisant l'idée qu'une simple alternative règlerait tout
- La croissance des 14x en commits reflète une réelle utilité du service et une adoption massive ; blâmer les agents IA omet que les humains produisent aussi plus de code, et réduire la capacité réduirait la valeur du service
