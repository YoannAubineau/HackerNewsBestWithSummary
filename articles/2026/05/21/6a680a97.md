## Résumé de l'article

asm.js est un sous-ensemble de JavaScript créé par Mozilla en 2013 pour exécuter du code à des vitesses proches du natif sur le web. Firefox 148 désactive ses optimisations par défaut et prévoit de supprimer complètement le code à l'avenir, car WebAssembly l'a remplacé avec succès et offre de meilleures performances.

- asm.js continuera de fonctionner comme du JavaScript classique via le JIT régulier ; aucun site ne sera cassé
- asm.js a permis à des projets comme Unity et Unreal d'exécuter des codebases C/C++ sur le web, ouvrant la voie à WebAssembly quelques années plus tard
- Les sites utilisant asm.js sont encouragés à recompiler en WebAssembly pour obtenir une exécution plus rapide et des binaires plus petits
- La maintenance d'asm.js parallèlement à WebAssembly augmente la charge de travail et la surface d'attaque de la machine virtuelle
- Le compilateur asm.js OdinMonkey, actif depuis 13 ans, sera remplacé par BaldrMonkey et RabaldrMonkey, les compilateurs WebAssembly

## Discussion sur Hacker News (140 commentaires)

**Avis positifs** :
- WebAssembly a mieux évolué qu'asm.js au fil des années avec des améliorations substantielles en performance, accès à SIMD, opérations mémoire en masse, et types GC
- asm.js a rempli son rôle historique de transition en prouvant que compiler du code natif vers le navigateur était possible (Figma, Unreal Engine), avant l'émergence de WASM
- WASM bénéficie d'une standardisation multi-navigateurs et d'une base de code plus robuste, contrairement à asm.js qui était une expérimentation propriétaire à Firefox
- La rétro-compatibilité de asm.js avec JavaScript régulier a permis le déploiement risque-zéro sans créer de dépendance éternelle
- Le succès de WASM valide rétrospectivement la vision originelle d'asm.js : exécuter du code natif rapide dans le navigateur, juste avec une meilleure approche technique

**Avis négatifs** :
- Plusieurs cas d'usage montrent que le SHA-256 custom en asm.js reste plus rapide que les solutions WASM disponibles (2x plus lent en Chrome)
- WASM reste trop isolé de JavaScript : accès API Web restreint, nécessite du code intermédiaire, pas de zero-copy pour les buffers (bien qu'une proposal existe)
- WASM en Chrome a performance problématique comparée à Firefox, suggérant des optimisations JIT toujours inégales entre navigateurs
- La suppression d'asm.js complique la maintenance du code legacy puisque Emscripten a abandonné le support autour de 2020, contrairement à asm.js qui reste exécutable
- L'écosystème WASM manque de maturité : fragmentation des runtimes, gestion mémoire complexe, limitations multithreading via WebWorkers, pas d'accès direct aux APIs web
