## Résumé de l'article

L'auteur décrit son passage de Tailwind CSS à une approche CSS vanilla avec HTML sémantique, après huit ans d'utilisation du framework. Bien que Tailwind lui ait été utile pour débuter, il maîtrise maintenant suffisamment le CSS pour implémenter ses propres systèmes d'organisation et explorer des fonctionnalités avancées.

- **Organisation par composants** : chaque composant reçoit une classe unique avec son propre fichier CSS, évitant les conflits et facilitant la maintenance
- **Systèmes réutilisés de Tailwind** : reset stylesheet, palette de couleurs, échelle de typographie et classes utilitaires copiées ou réinventées pour maintenir une cohérence
- **Responsive design flexible** : utilisation de CSS Grid avec `auto-fit` et `grid-template-areas` plutôt que de nombreuses media queries, réduisant la complexité
- **Build system minimaliste** : utilisation de esbuild pour la production, mais adoption des imports et sélecteurs imbriqués natifs du CSS moderne en développement
- **Motivations philosophiques** : meilleure maîtrise du CSS, limites de Tailwind pour du code complexe, mélanges non maintenables de Tailwind et CSS vanilla, et conviction que valoriser l'expertise CSS est préférable à l'abstraire

## Discussion sur Hacker News (189 commentaires)

**Avis positifs** :
- Écrire du HTML sémantique et structuré d'abord, puis styliser avec CSS améliore les compétences en développement web et l'accessibilité pour tous les utilisateurs
- Tailwind encourage une approche CSS-first qui décourage l'utilisation d'éléments HTML sémantiques et pousse à multiplier les divs pour placer des classes, créant du 'div soup'
- Une structure CSS bien pensée avec separation of concerns (fichiers CSS séparés, BEM, ITCSS) reste maintenable et scalable, même sur de grands projets avec plusieurs développeurs
- CSS moderne offre désormais des outils puissants (@layer, @scope, CSS variables) qui permettent de gérer la cascade efficacement sans avoir besoin d'un framework d'utilitaires
- Les frameworks modernes comme Svelte et Vue permettent de scoper naturellement les styles aux composants, éliminant le besoin principal de Tailwind

**Avis négatifs** :
- CSS purement structuré devient rapidement fragile et difficile à maintenir sur de grandes applications avec plusieurs équipes, la cascade crée des effets de bord imprévisibles
- Tailwind résout des problèmes réels : éviter les conflits de noms, limiter la cognition avec des utilitaires standardisés, et s'adapte très bien à la génération de code par IA et LLM
- Le reproche sur l'accessibilité et le 'div soup' n'est pas spécifique à Tailwind mais vient d'une mauvaise pratique du développeur ; Tailwind ne force pas cette approche
- Lire du code Tailwind prend du temps à s'habituer, mais une fois compris, il est très lisible et permet de maintenir plus facilement les styles de chaque composant en un seul endroit
- Tailwind offre des avantages réels : minification automatique en production, design system avec échelles de couleurs et espacements intégrés, et cohérence d'équipe sans débats de nommage
