I know what you're thinking... and I still can't believe it, but...

This morning, our database flagged a duplicate UUID (v4). I checked, thinking it may have been a double-insert bug or something, but no.

The original UUID was from a record added in 2025 (about a year ago), and today the system inserted a new document with a fresh UUIDv4 and it came up with the exact same one:

b6133fd6-70fe-4fe3-bed6-8ca8fc9386cd

We're using this:
https://www.npmjs.com/package/uuid

I thought this is technically impossible, and it will never happen, and since we're not modifying the UUIDs in any way, I really wonder how that.... is possible!? We're literally only calling:

import { v4 as uuidv4 } from "uuid";

const document_id = uuidv4();

... and then insert into the database, that's it.

## Discussion sur Hacker News (203 commentaires)

**Avis positifs** :
- Une défaillance du générateur de nombres aléatoires ou un mauvais ensemencement (seed) est beaucoup plus probable qu'une vraie collision statistique; plusieurs commentaires pointent vers des bugs connus dans les polyfills JavaScript et les implémentations cryptographiques faibles
- Le passage récent du package uuid d'une allocation mémoire unique réutilisée à plusieurs appels pourrait être la cause directe : les références mémoire sont identiques entre appels asynchrones, produisant des UUID identiques
- Les systèmes virtualisés, les mauvaises initialisations d'entropie et les problèmes historiques de machines virtuelles dans le cloud sont des causes documentées et crédibles de ce type d'anomalie
- UUIDv7 avec composante temporelle ou des alternatives comme ULID/CUID2 seraient plus robustes; les solutions avec horodatage garantissent qu'une collision à 1 an d'intervalle est techniquement impossible
- Il s'agit probablement d'une faille logicielle et non d'un événement statistique : laisser aux utilisateurs générer les UUIDs côté client (navigateur) augmente drastiquement le risque de collision du fait de la faiblesse des sources d'entropie dans les runtimes JavaScript shimés

**Avis négatifs** :
- Les UUIDs générés côté serveur en 2026 ne devraient statistiquement jamais présenter cette collision; un gestionnaire de clés crypto.getRandomValues() moderne sur un serveur disposant d'une entropie suffisante rend cet événement quasiment certain d'être un bug plutôt qu'une réalité probabiliste
- Valider et chercher les collisions dans la base de données échoue le contrat même des UUIDv4, qui sont censés être utilisés directement sans vérification—si vous devez vérifier les collisions, les UUID v4 ne sont pas l'outil approprié
- Le calcul statistique indique que toute collision observée sur 15 000 enregistrements générés correctement demande une enquête bien plus poussée avant d'accuser la probabilité; des explications plus prosaïques existent (traitement en doublon, requêtes rejouées, réutilisation d'objets en mémoire)
- Blâmer UUIDv4 pour une source d'entropie cassée masque le vrai problème : c'est un défaut d'infrastructure ou de configuration, pas un défaut du schéma d'identifiant
