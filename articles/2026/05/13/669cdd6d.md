## Résumé de l'article

Elevator est un traducteur binaire statique capable de convertir des exécutables x86-64 complets en AArch64 sans information de débogage ni code source, en générant plusieurs chemins de traduction pour chaque interprétation possible des octets (données, opcode ou argument d'opcode).

- Contrairement aux systèmes existants, Elevator n'utilise pas d'heuristiques ni de fallbacks à l'exécution, mais considère toutes les interprétations possibles et élimine uniquement les chemins menant à une terminaison anormale
- Les traductions sont construites en composant des « tuiles » de code dérivées automatiquement d'une description du jeu d'instructions source, ce qui rend l'approche déterministe et crée des binaires autonomes sans composant d'exécution dans la base de code de confiance
- Le principal inconvénient est une expansion importante de la taille du code, mais l'avantage clé est que le résultat est le code exact qui s'exécutera, permettant les tests, la validation, la certification et la signature cryptographique avant le déploiement
- Les évaluations sur des binaires réels incluant la suite SPECint 2006 montrent que la traduction binaire statique complète est fiable et pratique, avec des performances égales ou supérieures à l'émulation JIT en mode utilisateur de QEMU

## Discussion sur Hacker News (64 commentaires)

**Avis positifs** :
- L'approche superset CFG est ingénieuse : traduire chaque offset en tant que donnée ET instruction potentielle, puis résoudre à l'exécution par une table de correspondance, contourne élégamment le problème indécidable de la distinction code/donnée
- La certification pour les industries régulées (aviation, médical, finance) est un vrai déblocage : binaires statiques signables et reproduisibles, contrairement aux JIT non déterministes
- Les performances sont compétitives avec QEMU en mode utilisateur (plus rapide que l'émulation TCG) et l'approche offre une garantie complète de correction pour les binaires coopératifs sans code auto-modifiant
- L'expansion 50x du binaire est acceptable si le code chaud reste localisé et reste inaccessible, surtout avec la réorganisation du code au moment de l'édition de lien
- Rosetta et Box64 montrent que ce type de traduction statique/hybride fonctionne bien en pratique, libérant les utilisateurs de besoins de machines secondaires

**Avis négatifs** :
- Les limitations sont fondamentales : pas de code auto-modifiant, pas de JIT, pas d'exception handling, pas de multithreading, support ISA x86 incomplet rendent l'applicabilité très restreinte
- L'expansion 50x du .text est catastrophique pour les performances CPU (cache, prédiction de branchement) : les gains du JIT sont annulés par les défauts de cache massifs, sauf si le code non utilisé reste déchargé
- Rice's theorem s'applique toujours : impossible de résoudre les cas généraux de code adversarial ou généré dynamiquement ; la solution repose sur des hypothèses fortes et non réalistes
- Performance toujours loin derrière Rosetta (optimisé Apple, extensions ARM spéciales) et manque d'une vraie implémentation ouverte ; comparaison avec QEMU favorise l'article car QEMU est une TCG généraliste non optimisée
- Slack sur Asahi reste hors de portée : Electron/Chromium/V8 requièrent JIT et gestion d'exceptions, exposant le fossé entre promesses théoriques et besoins pratiques des applications modernes
