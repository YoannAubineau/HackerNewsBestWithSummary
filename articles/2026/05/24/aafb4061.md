## Résumé de l'article

L'article explore un scénario où une organisation décide délibérément de ne pas relire le code généré par les LLM pour gagner en productivité, en traitant ce code comme du bytecode ou du code compilé. Cette approche ne peut fonctionner que comme décision organisationnelle structurelle, pas au niveau individuel, et requiert une refonte complète des processus de développement selon la loi d'Amdahl.

- Les LLM produisent du code non-déterministe trop rapidement pour qu'une relecture manuelle effective soit possible ; il faut accepter cette limite plutôt que de la nier
- Cette stratégie exige de supprimer les goulots d'étranglement humains (approbations, coordination excessive, bureaucratie) et de donner aux ingénieurs l'autonomie pour prendre des décisions de produit
- La rigueur doit migrer vers les spécifications Markdown standardisées et les tests (distinct de TDD), plutôt que vers la relecture de code
- Les spécifications et tests deviennent les artefacts auditables pour lesquels l'équipe est responsable, tandis que le code généré est un détail d'implémentation
- Cette approche ne fonctionne que si l'organisation fournit un flux de travail quasi-infini et accepte que la refonte soit bon marché avec les agents IA

## Discussion sur Hacker News (183 commentaires)

**Avis positifs** :
- Les spécifications en Markdown et tests formels comme unité de travail principale offrent un moyen de maintenir la rigueur même si le code généré n'est pas relu en détail.
- L'IA peut effectivement accélérer la génération de code fonctionnel pour les tâches courantes et répétitives, libérant les développeurs pour se concentrer sur l'architecture et les décisions métier.
- Le déplacement de la responsabilité vers la définition claire des spécifications plutôt que la rédaction de code reflète une évolution pragmatique du rôle de l'ingénieur vers celui de designer produit.
- Traiter le code généré comme un artefact similaire au bytecode ou au JavaScript transpilé est une analogie valide pour les composants non-critiques simples et bien contenus.

**Avis négatifs** :
- Une spécification suffisamment détaillée pour être formellement vérifiée devient de facto un langage de programmation, rendant cette approche équivalente au statu quo plutôt que révolutionnaire.
- Les LLM génèrent du code non-déterministe avec des défauts subtils que les tests et spécifications seuls ne peuvent pas garantir de détecter, notamment pour les problèmes complexes d'architecture ou de sécurité (fuites de données, migrations).
- Cette approche recycle les promesses non tenues du passé : COBOL, UML, WSDL, outils no-code n'ont jamais supprimé le besoin de programmeurs competents à cause de la complexité inévitable de la spécification elle-même.
- Sans relecture humaine du code, la responsabilité devient floue et les bugs en production sont inévitables ; la concentration sur la vitesse d'itération cache en réalité une accumulation de dette technique impossible à rembourser.
- L'hypothèse que les coûts computationnels et tokens LLM sont 'presque gratuits' masque les vrais coûts d'infrastructure massive et crée une incitation perverse à préférer la quantité de code à sa qualité.
