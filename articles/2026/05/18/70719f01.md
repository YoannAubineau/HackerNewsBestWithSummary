## Résumé de l'article

Archestra, une plateforme open source, a été submergée par des bots IA postant des commentaires et pull requests inutiles, transformant un dépôt de qualité en lieu hostile pour les vrais contributeurs. L'équipe a mis en place un processus d'onboarding obligatoire couplé à une restriction GitHub limitant commentaires et PRs aux contributeurs antérieurs, en exploitant le flag Git `--author` pour créditer les nouveaux contributeurs légitimes sans passer par GitHub.

- Des dizaines de pull requests non testées et des commentaires sans valeur des bots IA ont inondé le dépôt, enterrant les discussions des contributeurs réels et démotivant l'équipe de maintenance.
- GitHub n'offre pas d'interface native pour filtrer les contributeurs ; Archestra a contourné cette limitation en activant "Limit to prior contributors" et en forgeant des commits attribués aux nouveaux utilisateurs via `--author` et leurs adresses noreply.
- Le flux combine un onboarding sur le site d'Archestra (avec CAPTCHA et règles éthiques IA), une GitHub Action qui pousse un commit signé du nouvel utilisateur, et un accès immédiat au dépôt une fois le commit merged.

## Discussion sur Hacker News (171 commentaires)

**Avis positifs** :
- La solution est élégante et efficace : combiner un CAPTCHA avec l'API GitHub et une co-signature git réduit drastiquement le spam (500 bots bloqués en une semaine) sans friction excessive pour les vrais contributeurs.
- L'approche ajoute une friction juste suffisante pour décourager les bots automatisés tout en restant accessible aux humains motivés, contrastant favorablement avec des solutions plus lourdes.
- Le système s'appuie intelligemment sur les mécanismes natifs de Git et GitHub plutôt que d'inventer une nouvelle infrastructure, ce qui le rend simple à maintenir et difficile à contourner.
- Cette initiative met en lumière l'inaction de GitHub/Microsoft face au problème des contributions de faible qualité générées par l'IA, forçant les mainteneurs à bricoler des solutions eux-mêmes.

**Avis négatifs** :
- La solution présente un risque de sécurité : les contributeurs passant le CAPTCHA obtiennent le statut de contributeur et contournent certaines protections GitHub, permettant potentiellement aux bots de s'élever mutuellement en tant que contributeurs.
- Le système n'est probablement que temporaire : les bots apprendront à contourner les CAPTCHAs (ou les exploiteront via des services tiers), ne faisant que repousser le problème plutôt que le résoudre fondamentalement.
- Cette solution constitue un pansement sur un problème systémique : GitHub devrait filtrer le spam à la source plutôt que de forcer chaque projet à inventer ses propres défenses, tandis que Microsoft profite des contributions générées par l'IA via Copilot.
- L'approche risque d'exclure les contributeurs légitimes : les utilisateurs préoccupés par la confidentialité, les personnes sans accès facile aux paiements, ou ceux de pays sanctionnés pourraient être découragés de participer.
