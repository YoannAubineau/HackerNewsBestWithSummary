## Résumé de l'article

Semble est une bibliothèque de recherche de code optimisée pour les agents d'IA, qui indexe et interroge les bases de code en utilisant 98% moins de tokens que les approches grep+read traditionnelles. Elle fonctionne entièrement en CPU, s'intègre comme serveur MCP avec les agents populaires (Claude Code, Cursor, Codex), et retourne des snippets de code pertinents en langage naturel.

- Performances : indexe en ~250 ms, répond aux requêtes en ~1,5 ms sur CPU, 200x plus rapide que les transformers spécialisés en code avec 99% de la qualité de retrieval
- Architecture hybride : combine les embeddings statiques Model2Vec (code-specialized) pour la similarité sémantique et BM25 pour les correspondances lexicales, fusionnés avec Reciprocal Rank Fusion
- Intégration agent : se déploie comme serveur MCP (uv ou pip) ou outil bash, automatiquement utilisé par les agents pour trouver du code via requêtes en langage naturel
- Efficacité tokens : atteint 94% de recall avec seulement 2k tokens contre 100k nécessaires pour grep+read à 85% de recall
- Zéro dépendance externe : aucune clé API, GPU, ou service externe requis ; supporte les chemins locaux et URLs git avec indexation automatique

## Discussion sur Hacker News (49 commentaires)

**Avis positifs** :
- Réduction significative de tokens confirmée dans les benchmarks de retrieval par rapport à grep, même si les gains end-to-end restent à démontrer
- Outil pratique et rapide : indexation en 3 secondes contre 15 minutes pour des outils concurrents, avec résultats plus pertinents
- Potentiel utile pour les petits codebases et comme complément aux approches existantes (LSP, mémoire d'agent) pour améliorer l'efficacité de recherche
- Les modèles Anthropic (Claude) font confiance aux résultats et les utilisent naturellement, contrairement à d'autres outils où les agents rechargent par défiance

**Avis négatifs** :
- Les benchmarks mesurent uniquement la précision de retrieval (NDCG), pas l'impact réel sur les agents : les économies de tokens peuvent être annulées par des tours supplémentaires ou des relectures
- Recherche probabiliste sur les termes : un petit modèle pourrait ne pas trouver des éléments importants, contrairement à grep qui est déterministe
- Comparaison problématique avec grep : les agents font plusieurs appels grep progressifs, pas une grosse requête unique, ce qui rend la comparaison de tokens ambiguë
- Absence de benchmarks end-to-end concrets avec des agents réels (Claude, Copilot) montrant réduction de coûts et temps—les tests utilisateurs montrent des résultats mitigés (gains contextuels contrebalancés par vitesse plus lente)
