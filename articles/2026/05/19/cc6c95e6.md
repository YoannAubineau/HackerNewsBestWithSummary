## Résumé de l'article

Le compte npm atool a été compromis le 19 mai 2026, et l'attaquant a publié 637 versions malveillantes sur 314 paquets en 22 minutes, affectant des dépendances populaires comme echarts-for-react (3,8 M téléchargements/mois) et size-sensor (4,2 M). Le payload est un script Bun obfusqué de 498 KB utilisant la même architecture que le compromis SAP trois semaines plus tôt (Mini Shai-Hulud), conçu pour voler des identifiants AWS, secrets Kubernetes, tokens GitHub/npm, clés SSH et accéder aux coffres de gestionnaires de mots de passe locaux (1Password, Bitwarden, pass, gopass).

- L'attaque utilise deux vecteurs d'exécution : un hook preinstall direct et des commits fantômes orphelins dans le dépôt antvis/G2, contournant les blocages de hooks via la résolution de dépendances github:// de npm
- Les identifiants volés sont exfiltrés via deux canaux parallèles : commits dans des dépôts GitHub créés sous le compte compromis (User-Agent forgé en python-requests/2.31.0) et requêtes HTTPS chiffrées RSA+AES vers t.m-kosche[.]com déguisées en traces OpenTelemetry
- En environnement CI, le payload échange des tokens OIDC GitHub Actions contre des tokens de publication npm, signe les artefacts via Sigstore (Fulcio + Rekor) et injecte de la persistance dans .github/workflows/codeql.yml pour exfiltrer les secrets du dépôt
- Le malware installe une porte dérobée C2 via systemd/LaunchAgent (kitty-monitor) qui interroge l'API GitHub Search pour des commits contenant le mot-clé firedalazer avec des commandes RSA-PSS signées, ne nécessitant aucune infrastructure contrôlée par l'attaquant
- Le payload détecte 20+ plateformes CI (GitHub Actions, GitLab CI, Jenkins, CircleCI, etc.), cible les secrets d'IA (injections de hooks dans Claude Code et VS Code), tente des échappements Docker via le socket hôte et propage l'infection latéralement aux autres projets Node.js locaux

## Discussion sur Hacker News (202 commentaires)

**Avis positifs** :
- npm et les gestionnaires de paquets JavaScript sont particulièrement vulnérables en raison d'une culture d'accumulation de dépendances, de scripts de pré/post-installation non restreints et d'une adoption rapide par des développeurs moins expérimentés
- Des mesures de mitigation concrètes existent et fonctionnent : utiliser min-release-age (délai minimum avant installation), désactiver les scripts de build avec allowBuilds:false, conteneuriser/virtualiser le développement et utiliser des pare-feu d'application (OpenSnitch) pour bloquer les connexions sortantes
- D'autres écosystèmes (PyPI, Cargo, Maven) sont aussi vulnérables aux attaques de chaîne d'approvisionnement, même si npm reçoit plus d'attention médiatique en raison de sa popularité et de ses faiblesses architecturales
- La vraie solution nécessite des changements culturels profonds : réduire la prolifération de dépendances triviales, vendre/distribuer des binaires pré-compilés au lieu de scripts de build, geler les versions de production pendant des périodes plus longues

**Avis négatifs** :
- Aucune mesure de défense en profondeur n'est suffisante : même en utilisant des VMs isolées, des conteneurs avec rootless mode ou en bloquant les connexions réseau, les attaquants trouvent des moyens d'échapper aux bacs à sable (container escape, VM escape) et les credentials restent vulnérables
- npm et Microsoft/GitHub refusent d'implémenter des solutions basiques comme l'interdiction par défaut des scripts de pré/post-installation ou des listes blanches ; la vraie raison semble commerciale (vendre des solutions de sécurité via Defender plutôt que d'adresser le problème à la source)
- Le problème n'est pas techniquement soluble tant que JavaScript reste utilisé systématiquement comme langage backend/serveur par des développeurs frontaux non formés, créant une situation où la facilité d'accès prime sur la sécurité
- Même geler les dépendances ne protège pas réellement : on échange simplement le risque d'une attaque 0-day contre le risque de laisser des vulnérabilités connues non patchées à long terme
