## Résumé de l'article

Un benchmark compare deux approches pour automatiser un panneau d'administration avec une IA : utiliser un agent de vision (qui interprète des captures d'écran) versus un agent appelant directement les endpoints HTTP. Les résultats montrent que l'approche vision consomme environ 500 000 jetons d'entrée en 14-22 minutes, tandis que l'approche API n'en consomme que 12 000 en quelques secondes, illustrant un écart de coût massif.

- L'agent de vision a nécessité une reformulation complexe de la tâche en 14 instructions numérotées pour fonctionner correctement (pagination invisible, absence de signal sur les données partielles).
- L'agent API a complété la tâche en 8 appels identiques avec une variance négligeable (±27 jetons), tandis que la vision a montré une variance extrême (407k-751k jetons sur 3 essais).
- L'architecture elle-même impose le coût : voir pour agir nécessite de capturer chaque état intermédiaire rendu en pixels, quels que soient les progrès des modèles de vision.
- Pour les outils internes contrôlés, générer automatiquement des surfaces API (comme Reflex 0.9 le permet) rend l'approche structurée économiquement supérieure ; la vision reste justifiée uniquement pour les applications tierces non modifiables.

## Discussion sur Hacker News (175 commentaires)

**Avis positifs** :
- L'interface conçue pour les humains est inefficace pour l'IA : les interfaces API spécialisées sont naturellement plus appropriées et efficaces pour les agents, ce qui explique le facteur 45x d'écart de coûts.
- La quantification rigoureuse de l'écart entre vision et API est utile : au-delà du constat évident, le benchmark apporte une donnée précise et révèle des goulots (délais de 17 minutes vs 0,5-2,8s) qui justifient des choix architecturaux.
- Les APIs générés à partir des interfaces existantes offrent une solution pragmatique : transformer les interfaces humaines en APIs programmatiques permet de récupérer efficacité sans refonte complète des applications.
- La vision par ordinateur restera nécessaire pour les applications legacy sans API : certains cas d'usage (ERP fermés, systèmes propriétaires) n'ont pas d'alternative à court terme, justifiant l'existence de cette approche coûteuse.
- De meilleurs outils hybrides (DOM + vision) et petits modèles promettent des gains : combiner accessibilité native, Playwright et modèles optimisés peut réduire considérablement les coûts tout en conservant la flexibilité.

**Avis négatifs** :
- Computer use est un problème résolu depuis longtemps : API et CLI efficaces existent pour des raisons valides ; revenir à la navigation visuelle est une régression coûteuse motivée par des raisons commerciales plutôt que techniques.
- La promesse de l'IA était précisément de ne pas nécessiter de remaniement des systèmes : exiger que tout soit refactorisé pour les agents contredit la valeur première de l'IA d'interagir avec l'existant.
- Aucun incitatif commercial pour exposer les API : les géants du web (Meta, Amazon, réseaux sociaux) bloquent activement les clients tiers et les scrapers ; attendre qu'ils créent des APIs pour les agents est naïf.
- Les cas d'usage réels sont marginaux ou trop risqués : impôts, vérifications d'antécédents, création d'LLC et tâches avec données sensibles ne peuvent pas être confiés à des agents, limitant drastiquement l'utilité pratique.
- L'approche démontre surtout les limites actuelles des modèles : le facteur 45x révèle que la vision n'est pas prête pour la production ; plutôt que de généraliser, cela renforce l'argument pour des interfaces spécialisées.
