## Résumé de l'article

GitHub a confirmé qu'environ 3 800 de ses dépôts internes ont été compromis après qu'un employé ait installé une extension VS Code trojannisée. Le groupe de hackers TeamPCP revendique l'accès à environ 4 000 dépôts privés et demande au moins 50 000 dollars pour les données volées.

- Une extension malveillante a été supprimée de la place de marché VS Code et l'appareil compromis a été isolé par GitHub
- L'exfiltration concerne uniquement les dépôts internes à GitHub ; aucune donnée client externe n'a été affectée selon l'enquête préliminaire
- TeamPCP a précédemment mené des attaques en chaîne d'approvisionnement contre GitHub, PyPI, NPM et Docker, et a revendiqué la campagne « Mini Shai-Hulud » affectant des employés d'OpenAI
- Les extensions VS Code trojannisées ne sont pas rares sur la place de marché ; plusieurs incidents antérieurs ont impliqué des millions d'installations volant des identifiants de développeurs et d'autres données sensibles

## Discussion sur Hacker News (126 commentaires)

**Avis positifs** :
- Les malveillances par extensions malveillantes sont faciles à déployer et difficiles à détecter : petits payloads chiffrés, exfiltration via DNS ou domaines légitimes, absence de re-authentification requise pour git clone.
- Le modèle de sécurité de VSCode est fondamentalement faible : absence d'isolation entre le frontend, backend et extensions ; aucune sandbox réelle ; les workspaces peuvent exécuter du code arbitraire.
- Les mesures de restriction des requêtes sortantes (IP allowlist, limitation des domaines autorisés, audit logs) sont insuffisantes mais utiles pour bloquer une partie des attaques ; le sandboxing des extensions aurait dû être implémenté depuis longtemps.
- La confiance envers les extensions est mal placée : tout développeur peut télécharger une extension sans vérifier son code ; l'écosystème npm/pip/npm partage le même problème systémique de confiance par défaut.
- GitHub et Microsoft pourraient mettre en place des contrôles au niveau de l'organisation (SSO forcé, jetons à expiration courte, restrictions IP) et audit logging, mais ces mesures restent des pansements sur une architecture intrinsèquement non sécurisée.

**Avis négatifs** :
- Il est irréaliste de demander aux développeurs de réécrire toutes les extensions dont ils ont besoin : la réalité économique impose de réutiliser des solutions existantes, et les extensions années d'efforts sont rarement égalées en qualité par des réimplementations maison.
- Sandboxer correctement Node.js et Electron est techniquement très difficile, notamment sur Linux où SUID est obligatoire ; il n'existe pas de solution simple et les plaintes sur ce sujet remontent à 2018 sans progrès.
- Même avec des mesures de sécurité (fine-grained tokens, allowlists IP, 2FA), les attaquants peuvent les contourner rapidement : injection de clients Wireguard/Tailscale, exploitation d'extensions avec raisons légitimes d'accès internet, attaques ciblées sur les vrais comptes à accès large.
- 3800 repos est un nombre plausible pour une grande organisation comme GitHub : accumulation naturelle sur 18+ ans, fork, POCs abandonnés, expériences ; le véritable enjeu n'est pas le nombre mais les secrets critiques potentiellement cachés dedans.
- Payer une rançon n'offre aucune garantie que les données seront supprimées ; les criminels opérant en groupes rationnels maintiendraient leur réputation pour de futures victimes, mais les teenagers anonymes avec ransomware n'obéissent à aucune logique d'entreprise établie.
