## Résumé de l'article

Le langage C et C++ contiennent tellement de cas de comportement indéfini (UB) qu'aucun programmeur ne peut les écrire de manière correcte. Au-delà des erreurs évidentes comme la double libération ou l'accès hors limites, il existe des dizaines de situations subtiles et contre-intuitives où le code peut déclencher un comportement indéfini selon le standard C, notamment l'alignement mémoire incorrect, les conversions de types invalides, ou l'accès à l'adresse zéro.

- Les cas de UB s'étendent bien au-delà des erreurs de mémoire classiques : alignement incorrect des pointeurs, casts de types mal alignés, conversion float-to-int sans vérifications, utilisation de fonctions comme isxdigit() avec des entrées invalides, et division par zéro
- Le UB n'est pas lié aux optimisations du compilateur ; il existe parce que le compilateur peut supposer que le code est valide, même sans optimisations activées
- La norme C23 contient 283 utilisations du terme « indéfini » et de nombreuses autres situations indéfinies par omission
- Les modèles de langage (LLM) surpassent les humains pour détecter les UB dans le code C existant, y compris dans des projets matures comme OpenBSD
- Sans supervision par IA ou une révision experte minutieuse, l'écriture de C/C++ en 2026 représente un risque de sécurité majeur et devrait être considérée comme irresponsable ou contraire aux normes de conformité

## Discussion sur Hacker News (495 commentaires)

**Avis positifs** :
- Le constat est juste : même des experts reconnus en C/C++ (comme ceux mentionnés en exemple) commettent des erreurs UB subtiles, ce qui démontre l'impossibilité pratique de les éviter complètement
- Les outils modernes (UBSan, Valgrind, analyseurs statiques) aident mais ne suffisent pas ; une approche combinée LLM + experts pourrait déterminer plus d'UB qu'aucune méthode seule
- L'UB s'est accumulée depuis 54 ans sans solutions concrètes ; blâmer les programmeurs d'« éviter UB » est inefficace face à cette réalité empirique
- Les compilateurs appliquent une interprétation agressive de l'UB à des fins d'optimisation, exploitant des comportements que le code n'énonce pas explicitement

**Avis négatifs** :
- Plusieurs exemples ne sont pas de véritables UB ou sont plutôt des erreurs de programmation évidentes (appels de NULL, conversion float→int sans gestion, isxdigit avec index non validé)
- C et C++ sont des langages fondamentalement différents depuis une décennie ; les regrouper ensemble (« C/C++ ») est trompeur et diminue la pertinence de la critique envers C++
- L'article est alarmiste sans proposer de solutions concrètes ; utiliser les LLM pour détecter l'UB est peu fiable car ils génèrent aussi de l'UB
- Beaucoup de soi-disant UB n'est jamais un problème en pratique sur les architectures modernes ; l'UB strict sur l'alignement des pointeurs, par exemple, est une relique obsolète
- Écrire du C sans UB est en réalité possible avec discipline et outils (UBSan, flags comme -fwrapv, compiler intrinsics) ; prétendre que c'est « impossible » est exagéré
