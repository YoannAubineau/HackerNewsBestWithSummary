## Résumé de l'article

GitHub Actions, le service d'intégration et de déploiement continu de GitHub, a connu une nouvelle panne. Les pages de statut régionales (Australie, UE, Japon, États-Unis) ne signalent cependant aucun incident ou maintenance documenté.

- Les services GitHub Enterprise Cloud sont affectés dans plusieurs régions
- Aucun incident officiel n'est enregistré sur les pages de statut
- Les utilisateurs peuvent vérifier le statut par région via les domaines dédiés

## Discussion sur Hacker News (181 commentaires)

**Avis positifs** :
- Les pannes répétées de GitHub justifient sérieusement la migration vers des alternatives auto-hébergées (Forgejo, GitLab, Gitea) ou vers des services tiers (Codeberg, SourceHut), qui se sont avérés plus stables à long terme.
- L'augmentation massive du volume (repos, PRs, clones) liée à l'adoption de l'IA et des outils de codex explique partiellement la surcharge, confirmée par les rapports de GitHub eux-mêmes sur l'inflation du trafic.
- L'infrastructure de GitHub souffre d'une transition incomplète vers Azure et conserve une architecture hybride bancale depuis des années, aggravée par des choix architecturaux qui rendent les défaillances d'un composant globales plutôt que localisées.
- Le problème est systémique : les utilisateurs découvrent les pannes avant GitHub, les statuts de dégradation manquent de clarté, et la relation prix/fiabilité est inacceptable pour un service payant par les entreprises.

**Avis négatifs** :
- Les alternatives auto-hébergées exigent une maintenance continue (mises à jour, sauvegarde, surveillance, alertes) que les petites équipes ne peuvent pas se permettre, malgré l'aide des outils d'IA.
- Les concurrents de GitHub (GitLab, Bitbucket) n'affichent pas une instabilité comparable malgré l'exposition similaire au trafic IA, suggérant que les pannes de GitHub relèvent davantage de la mauvaise gestion que de forces externes inévitables.
- Même les solutions tierces comme Depot, Blacksmith.sh ou runs-on.com qui utilisent des runners externes restent dépendantes du plan de contrôle de GitHub et sont donc affectées par ces pannes.
- Attribuer les pannes principalement à l'IA est une simplification : les instabilités ont commencé bien avant (2018-2019 post-acquisition par Microsoft), et les données d'uptime historical ne sont pas fiables pour valider cette théorie.
