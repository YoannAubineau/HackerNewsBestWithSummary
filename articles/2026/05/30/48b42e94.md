## Résumé de l'article

Le Model Context Protocol (MCP) est un protocole lancé fin 2024 pour connecter les modèles de langage à des outils externes (GitHub, Linear, Notion, Slack). Cet article, basé sur des mesures réelles chez Quandri, remet en question son efficacité par rapport aux approches alternatives.

- MCP charge systématiquement les définitions de tous les outils en contexte, même inutilisés : le serveur Linear seul consomme 12 800 tokens (42 outils) pour accéder à une simple issue, soit 65 fois plus que via CLI direct (~200 tokens).
- Les performances sont dégradées : MCP est 3 fois plus lent par appel et 9,4 fois plus lent au premier appel (initialisation comprise) comparé à une API REST directe, en raison de la couche de processus supplémentaire.
- Les CLI existants combinés avec des « Skills » (documentation intégrée au contexte) représentent une alternative plus légère : zéro coût en contexte, exécution directe, débogage dans le terminal.
- MCP reste pertinent pour les bases de données (exécution de requêtes sécurisée) et les services sans CLI, mais est sur-dimensionné pour la plupart des flux de travail développeurs.
- Chez Quandri, remplacer les serveurs MCP par des Skills a libéré ~21 000 tokens de contexte et éliminé les défaillances d'initialisation quotidiennes.

## Discussion sur Hacker News (218 commentaires)

**Avis positifs** :
- MCP reste vital pour les entreprises : il permet de centraliser l'accès aux APIs internes, de gérer les authentifications via OAuth de manière standardisée, et d'offrir un contrôle fine des permissions que les agents ne peuvent pas contourner (essentiel pour la conformité et la sécurité)
- MCP résout un vrai problème de distribution pour les services sans CLI existante (Slack, Notion, Linear) et permet aux entreprises non-tech de partager des outils sans burden de maintenance décentralisée
- Les critiques sur la consommation de contexte sont obsolètes : Claude Code a implémenté le chargement différé des schémas MCP, réduisant la consommation de tokens de 85%+, résolvant largement le principal reproche
- MCP pousse l'adoption massive d'interfaces machine-readable : des centaines d'entreprises construisent des MCP quand elles n'avaient jamais eu d'API publique, démocratisant l'accessibilité des outils via agents
- MCP est plus qu'un simple protocole de tools : il inclut des prompts distribués par serveur, des ressources, et des mécanismes d'authentification standardisés que les CLIs ne proposent pas

**Avis négatifs** :
- MCP ajoute une couche d'indirection inutile : pour les cas où une CLI ou une API REST existe déjà, MCP crée de la complexité supplémentaire sans bénéfice significatif par rapport à simplement pointer l'agent vers une documentation claire
- L'expérience utilisateur de la gestion MCP est très défaillante : découverte opaque, pas de gestion facile des permissions individuelles, manque de tooling pour savoir l'état des serveurs MCP, ce qui frustre même les utilisateurs avancés
- Le contexte reste consommé pour les MCPs mal conçus avec trop de tools ou de descriptions gonflées, et les réponses JSON des MCPs sont plus verboses que les sorties shell qu'on peut filtrer avec grep/head
- MCP encourage un faux sentiment d'urgence : beaucoup de companies construisent des MCPs simplement parce que c'est la mode, sans réel besoin, suivant un effet de bandwagon plutôt que des avantages techniques
- Pour les use-cases complexes ou avec état significatif (comme AWS), MCP montre ses limites et les developers se tournent vers des CLIs ou du Code Mode, suggérant que MCP n'est qu'une solution transitoire pour un problème d'inertie des modèles actuels
