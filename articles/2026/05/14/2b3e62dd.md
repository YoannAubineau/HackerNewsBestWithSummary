## Résumé de l'article

CVE-2026-42945 est un débordement de buffer critique dans le module ngx_http_rewrite_module de NGINX, introduit en 2008, permettant l'exécution de code à distance non authentifiée. Cette vulnérabilité a été découverte automatiquement par le système d'analyse de sécurité depthfirst, accompagnée de trois autres failles de corruption mémoire.

- Le bug résulte d'une incohérence dans le processus deux passes du moteur de script NGINX : le calcul de taille ignore le flag is_args, tandis que la copie de données l'active, provoquant un débordement lors de l'échappement URI
- L'exploitation utilise des techniques de heap feng shui pour corrompre les pointeurs de nettoyage dans une structure ngx_pool_t adjacente et invoquer system()
- La preuve de concept comprend des scripts de configuration (setup.sh, docker-compose.yml) et un exploit Python testés sur Ubuntu 24.04.3 LTS
- La vulnérabilité affecte les serveurs NGINX utilisant les directives rewrite et set

## Discussion sur Hacker News (58 commentaires)

**Avis positifs** :
- La vulnérabilité affecte un motif très courant en production : les directives rewrite avec captures non nommées suivies de set, dangereuse depuis 2008
- L'exploit démontre une chaîne d'attaque sophistiquée (heap feng shui + corruption de pointeurs) transformant un débordement en RCE non authentifiée
- La découverte autonome par outils d'IA de sécurité montre le potentiel de ces technologies pour identifier des failles anciennes dormantes
- Les correctifs sont disponibles rapidement (F5 pour 1.31.0/1.30.1, OpenResty pour 1.27/1.29) et une mitigation simple existe (captures nommées)

**Avis négatifs** :
- Le PoC publié nécessite ASLR désactivé, condition rare sur les distributions Linux modernes par défaut
- L'exploitation exige des conditions préalables précises (rewrite + ? + set avec capture), combinaison peu commune en pratique selon de nombreux retours
- Bien que contourner ASLR soit théoriquement possible, aucune preuve publique n'existe encore, les affirmations restent spéculatives
- Les versions mineure constante de Nginx (1.x) ne reflètent pas l'effort de développement réel, comparaison trompeuse avec React versioning
- La plupart des configurations Nginx modernes utilisent proxy_pass vers des backends plutôt que rewrite complexes, limitant l'exposition réelle
