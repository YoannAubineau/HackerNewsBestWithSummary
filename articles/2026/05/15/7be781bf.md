## Résumé de l'article

Bun, un runtime JavaScript, a entrepris une réécriture en Rust mais la base de code contient des erreurs de sécurité mémoire. Le code utilise des pointeurs invalides (dangling references) dans des blocs `unsafe`, ce qui viole les garanties du Rust et crée un comportement indéfini détectable par les outils de vérification comme Miri.

- La base de code échoue les vérifications Miri standards, qui détectent les violations de sécurité mémoire
- Un exemple concret montre un pointeur persistant après la suppression de l'objet source, créant une référence flottante
- Le problème expose une pratique dangereuse : construire des références brutes sans maintenir la validité de la mémoire pointée
- Les commentaires soulignent que l'IA génère mal du code Rust et recommandent d'embaucher des développeurs Rust expérimentés

## Discussion sur Hacker News (202 commentaires)

**Avis positifs** :
- Une approche progressive de port-et-correction est valide : faire d'abord une traduction mécanique 1:1 en Rust (même avec du code unsafe), puis corriger les problèmes incrementalement, est une stratégie reconnue (c2rust fonctionne ainsi).
- Le projet en est encore à une étape expérimentale : ce n'est pas une version stable/publiée, juste un travail en cours sur main ; les outils comme Miri peuvent maintenant identifier ces problèmes pour les corriger avant toute release.
- L'effort bénéficie à long terme : même un port imparfait offre accès au système de types de Rust et aux outils de détection, permettant d'améliorer progressivement la sécurité mémoire comparée à Zig.
- Les tests existants validant la parité fonctionnelle constituent un point de départ solide pour rouler la migration et itérer.

**Avis négatifs** :
- Fusionner un million de lignes de code généré par IA directement sur main sans revue humaine approfondie est irresponsable envers les 90k+ utilisateurs qui font confiance à Bun.
- L'absence de communication claire : pas de blog post expliquant la stratégie progressive, créant une asymétrie entre l'annonce flashy et la correction ultérieure—exploitation classique du marketing.
- Merging du code non-audité avec undefined behavior connu démontre un manque de rigueur élémentaire (ne pas même exécuter Miri avant merge est inexcusable).
- Le processus soulève des doutes sur la maintenabilité : avec un slop-code de cette taille, seuls les outils IA peuvent itérer dessus, créant une dépendance dangereuse à l'automatisation plutôt qu'à la compréhension humaine.
- L'asymétrie de communication (annonce flashy/correction discrète) et le traitement d'une version de travail comme preuve que le projet est mûr trahissent la confiance établie avec la communauté.
