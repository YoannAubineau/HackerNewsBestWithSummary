## Résumé de l'article

Turso, une réécriture de SQLite en cours de développement, ferme son programme de récompense de 1 000 dollars pour les bugs causant une corruption de données. Le programme, actif depuis près d'un an, a attiré des centaines de faux rapports générés par des systèmes d'IA et des scripts automatisés, submergeant les mainteneurs et rendant impossible le maintien d'une politique d'open contribution.

- Le programme initial a récompensé 5 contributeurs de qualité qui ont trouvé des bugs réels, notamment Pavan Nambi qui a découvert plus de dix bugs dans SQLite lui-même via ses méthodes de test formelles
- Les faux rapports se sont multipliés exponentiellement, avec des soumissions délibérément absurdes (injection de données corrompues dans la base, modification du code source pour créer des bugs artificiels, démonstration triviale de fonctionnalités normales comme des "vulnérabilités")
- Un système de vérification a été mis en place pour bloquer les soumissions suspectes, mais les bots ont commencé à rouvrir les issues et à demander une inspection manuelle
- L'asymétrie des coûts est critique : générer une fausse soumission prend une minute, analyser et répondre prend des heures aux mainteneurs
- Turso reste engagée envers l'open source mais reconnaît que les incitations financières ne fonctionnent plus dans un système ouvert confronté à l'automatisation à grande échelle

## Discussion sur Hacker News (195 commentaires)

**Avis positifs** :
- AI-generated submissions massively pollute bug bounty programs with low-effort slop, making human review prohibitively time-consuming and justifying the closure decision
- The core problem is unsolvable through technical gatekeeping alone—AI can easily create new accounts (Sybil attacks), evade bans, and argue endlessly with reviewers, turning moderation into whack-a-mole
- Reading and understanding code is now the real bottleneck, not writing it; AI amplifies this by flooding maintainers with massive PRs that require disproportionate review effort
- Bug bounty programs inadvertently incentivize bad actors to automate mass submissions; removing the monetary incentive historically attracts higher-quality, genuinely altruistic security reports

**Avis négatifs** :
- Closing the program throws out legitimate bug hunters along with the slop; monetary friction (e.g., $5–$20 submission fee) could filter spam while allowing genuine researchers to participate
- AI-assisted work isn't inherently low-quality—skilled engineers using AI tools do ship features 2–5× faster with solid code; the real issue is incentivizing high-volume, low-effort submissions
- Some proposals like subscription fees, proof-of-work systems, or reputation-based tiers could address the spam problem without full closure, though implementation costs remain a barrier
- Human-written slop and bad PRs existed before AI; the underlying issue is insufficient vetting processes, not AI itself—better automated screening or review workflows could coexist with the program
