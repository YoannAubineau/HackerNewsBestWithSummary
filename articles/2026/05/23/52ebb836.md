## Résumé de l'article

yt-dlp, un outil de téléchargement vidéo, restreint désormais son support du runtime JavaScript Bun en raison de problèmes de compatibilité et de sécurité. Bun est un runtime JavaScript moderne conçu pour remplacer Node.js.

- Seules les versions de Bun 1.2.11 à 1.3.14 seront supportées, contre 1.0.31 minimum auparavant
- La limite inférieure est relevée pour des raisons de sécurité : les versions antérieures à 1.2.0 ignorent le fichier de verrouillage lors de la construction du paquet ejs, un risque face aux attaques de la chaîne d'approvisionnement npm
- La limite supérieure est fixée à 1.3.14 car c'est la dernière version basée sur le code Zig original ; Bun a été réécrit en Rust via Claude après cette version
- Le support de Bun est déclaré déprécié : yt-dlp pourra abandonner complètement ce support si sa maintenance devient trop coûteuse
- Des informations détaillées sont disponibles sur le wiki EJS du projet

## Discussion sur Hacker News (351 commentaires)

**Avis positifs** :
- Les runtimes alternatifs (Node, Deno) restent supportés, offrant des options stables et éprouvées pour les utilisateurs de yt-dlp.
- Une réécriture de 1 million de lignes sans examen humain exhaustif représente un risque technique réel, justifiant une approche prudente plutôt que d'adopter une dépendance instable.
- La décision de maintainers de ne pas supporter une infrastructure critique qu'ils ne comprennent pas est une bonne pratique d'ingénierie, protégeant les utilisateurs des futurs problèmes de maintenabilité.
- Attendre que le code vibe-généré soit éprouvé en production avant de s'engager est une stratégie raisonnable et peut être revisitée ultérieurement si Bun démontre sa fiabilité.

**Avis négatifs** :
- Aucun bug n'a été observé depuis la réécriture ; juger une dépendance sur des spéculations plutôt que sur des données concrètes n'est pas rigoureux techniquement.
- La suite de tests exhaustive (dizaines de milliers de tests) a validé la réécriture Rust ; celle-ci ressemble davantage à une traduction 1:1 qu'à du véritable code généré aléatoirement.
- La décision semble davantage idéologique (rejet de l'IA en général) que fondée sur des problèmes d'ingénierie observables, particulièrement quand Bun s'est montré fiable pendant deux ans.
- Les développeurs Bun, payés par Anthropic et ayant accès aux outils internes les plus avancés, ont probablement revu le code plus rigoureusement que le suggère la rhétorique du « vibe coding ».
- Deno a aussi reçu du code généré par IA ; rejeter sélectivement Bun applique un double standard, d'autant que le Bun support n'était que secondaire à Node/Deno dans yt-dlp.
