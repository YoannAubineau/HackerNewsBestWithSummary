## Résumé de l'article

Le journal de développement de Zig pour 2026 documente les progrès majeurs sur plusieurs fronts : le nouveau linker ELF, qui permet maintenant des reconstructions incrémentales rapides avec support des bibliothèques externes ; une refonte du système de build séparant le processus de configuration du processus d'exécution ; et des améliorations à la résolution de types du compilateur.

- Le linker ELF nouvellement introduit en Zig 0.16.0 a progressé jusqu'à pouvoir compiler le compilateur Zig lui-même avec LLVM et LLD, et offre des reconstructions incrémentales en millisecondes sur x86_64 Linux, bien que la génération de debug DWARF reste à implémenter.
- Le système de build scinde désormais la compilation de build.zig (le « configurer ») de l'exécution du graphe de build (le « maker »), réduisant le temps d'exécution de « zig build --help » de 150 ms à 14 ms (une amélioration de 90 %).
- La refonte de la résolution des types permet à Zig d'ignorer les analyses inutiles des types n'utilisés qu'en tant qu'espace de noms et améliore significativement les messages d'erreur de boucles de dépendances.
- Les packages dépendants sont maintenant stockés localement dans un répertoire zig-pkg à côté de build.zig, avec un cache global compressé; un nouveau flag --fork permet de surcharger temporairement les dépendances pour développement.
- Zig migre vers l'utilisation des API natives Windows (ntdll) plutôt que kernel32, éliminant les allocations de heap inutiles et les points de défaillance dans des opérations comme la lecture d'entropie et les accès fichiers.

## Discussion sur Hacker News (94 commentaires)

**Avis positifs** :
- Les améliorations du linker ELF et la compilation incrémentale réalisent une promesse historique de Zig : permettre une itération rapide (style Python/JS) avec la performance de C, transformant Zig en véritable remplaçant de C.
- Ce travail est le fruit d'efforts de long terme (depuis 2020) et démontre un engagement stratégique cohérent, indépendamment des drames externes ; les développeurs apprécient cette communication transparente.
- Zig se distingue par sa philosophie axée sur le plaisir du développement et la liberté, contrairement à Rust qui impose des contraintes strictes ; sa communauté est perçue comme bienveillante et dépourvue d'évangélisme agressif.
- Le langage offre un excellent système de compilation et de ciblage (comptime, allocateurs), ce qui en fait une cible de transpilation supérieure à Rust ou C pour créer des langages de haut niveau.
- Ces améliorations ouvrent des perspectives pour des applications pratiques (DAW, UI native) qu'Andrew Kelly envisageait dès le départ, particulièrement une fois des frameworks UI natifs créés.

**Avis négatifs** :
- Fondamentalement, Zig reste limité aux niches où C est viable ; les réclamations qu'il permettra l'itération rapide comme Python/JS ignorent que la gestion mémoire et des allocations forcera toujours une complexité que les vrais langages dynamiques évitent.
- L'absence de version 1.0 pose un problème d'adoption commerciale et d'engagement à long terme ; le langage change encore trop pour que les entreprises prennent le risque d'investir massivement.
- La suppression de `@cImport` au profit d'une configuration de build system est une régression UX pour le portage progressif de code C existant, forçant les développeurs à rester sur d'anciennes versions ou réécrire massivement.
- Le philosophe de contrôle total des allocations rend Zig fondamentalement incompatible avec une véritable itération rapide ; pourquoi accepter des compromis entre liberté et sécurité quand des langages existants (Go, Rust) offrent déjà des choix clairs ?
- La vitesse de compilation n'est pas déterminée par le langage seul mais par l'implémentation du compilateur ; les résultats observés sur Nim ou autres langages montrent que Zig ne sera jamais radicalement plus rapide que des alternatives bien optimisées.
