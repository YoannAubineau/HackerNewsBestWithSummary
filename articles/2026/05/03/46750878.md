## Résumé de l'article

Acai.sh est un outil open-source conçu pour améliorer la collaboration entre humains et agents IA lors du développement de logiciels, en se concentrant sur le suivi de la couverture des exigences plutôt que sur la génération de code automatique. L'auteur partage son expérience de surspécification ("specsmaxxing") et propose une approche structurée utilisant le format YAML pour définir les critères d'acceptation numérotés (ACID) et suivre leur implémentation.

- Acai.sh fournit un modèle `feature.yaml` pour écrire des spécifications cohérentes, une CLI pour intégrer les spécifications dans le CI/CD, et un tableau de bord web pour examiner les exigences plutôt que les pull requests fichier par fichier
- Le workflow proposé suit quatre étapes : spécifier la fonctionnalité, générer le code via IA, examiner les exigences sur le tableau de bord, puis itérer en modifiant la spécification elle-même
- L'outil s'oppose aux approches concurrentes (SpecKit, OpenSpec, Kiro, Traycer.ai) en mettant l'accent sur le suivi de l'alignement des spécifications plutôt que sur l'automatisation complète de la génération ou la documentation non structurée
- Les spécifications décrivent le comportement désiré (non actuel), excluent les détails cosmétiques, et nécessitent une numérotation stable pour que les modifications ultérieures demandent une réalignement du code
- L'approche s'inscrit dans une vision plus large : d'abord maximiser la couverture de test ("testmaxxing"), puis permettre aux agents IA de réagir automatiquement aux tests échoués et aux alertes sans intervention humaine

## Discussion sur Hacker News (251 commentaires)

**Avis positifs** :
- Les spécifications écrites et numérotées offrent une traçabilité cruciale et permettent aux équipes de référencer facilement les exigences dans le code, résolvant le problème de la perte de contexte avec le code généré par IA
- Structurer les specs en YAML/format machine-lisible améliore la compression d'information (1 spec dense = 10x+ de code) et permet aux non-développeurs et partenaires commerciaux de comprendre et modifier les exigences
- Cette approche formalise ce qui existait déjà implicitement (specs dans les conversations ou la tête des gens) et crée un processus d'ingénierie plutôt que du 'vibe coding', essentiellement en formalisant une pratique historique des analystes logiciels
- Combiner specs structurées avec tests exécutables (Gherkin, BDD) crée un cycle de vérification automatisé où les agents peuvent itérer vers le respect des critères, réduisant les glissements et augmentant la maintenabilité

**Avis négatifs** :
- YAML est un mauvais choix pour cela : format verbose et sujet aux erreurs d'indentation, particulièrement problématique pour les LLM qui peinaient déjà avec la syntaxe ; markdown ou code directement seraient préférables et tout aussi parsables
- Le code lui-même est déjà la vraie spécification ; créer une couche supplémentaire est redondant et maintient deux sources de vérité qui divergeront. Les spécifications 'maxxées' génèrent souvent une maintenance insoutenable et finissent obsolètes
- L'approche suppose une spécification complète en amont, ignorant 30 ans de leçons agiles : aucune spec ne survit au contact avec la réalité, les détails critiques émergent pendant l'implémentation et les besoins métier changent une fois qu'on teste
- Pour les projets brownfield réalistes (legacy code, équipes distribuées, systèmes complexes), cette approche ne scale pas. Le coût réel (tokens, temps) reste très élevé et est masqué par l'enthousiasme ; peu de preuves empiriques sur le ROI par rapport au développement humain direct
- Le terme 'specsmaxxing' et la tonalité générale du post recylent des tendances hype cycliques (editor wars, note-taking apps, Agile) où chaque génération réinvente les mêmes solutions avec une nouvelle emballage sans résoudre les problèmes structurels des LLM
